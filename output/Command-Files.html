23.1.3命令文件 ¶

用于GDB的命令文件是一个文本文件，由一行行的GDB命令组成。也可以包含注释（以#开头的行）。命令文件中的空行不执行任何操作；它不像在终端中那样表示重复上一个命令。

你可以使用 source 命令来请求执行命令文件。请注意，source 命令也用于评估不是命令文件的脚本。可以使用 script-extension 设置来配置确切的行为。 请参阅 扩展GDB。

source [-s] [-v] 文件名 ¶

执行命令文件 文件名。

命令文件中的行通常按顺序执行，除非执行顺序被下面描述的某个 流控制命令 改变。命令执行时不会被打印出来。任何命令中的错误都会终止命令文件的执行，并将控制权返回给控制台。

GDB首先在当前目录中搜索 文件名。如果未在该目录中找到该文件，并且 文件名未指定目录，那么GDB也会在源搜索路径（由 ‘directory’ 命令指定）中查找该文件； 但不会搜索 $cdir，因为编译目录与脚本无关。

如果指定了 -s，那么即使 文件名 指定了目录，GDB也会在搜索路径中搜索 文件名。 搜索是通过将 文件名 附加到搜索路径的每个元素来完成的。例如，如果 文件名 是 mylib/myscript 并且搜索路径包含 /home/user，那么GDB将查找脚本 /home/user/mylib/myscript。 如果 文件名 是绝对路径，也会进行搜索。例如，如果 文件名 是 /tmp/myscript 并且搜索路径包含 /home/user，那么GDB将查找脚本 /home/user/tmp/myscript。 对于类似DOS的系统，如果 文件名 包含驱动器规范，在拼接之前会将其去掉。例如，如果 文件名 是 d:myscript 并且搜索路径包含 c:/tmp，那么GDB将查找脚本 c:/tmp/myscript。

如果给定了 -v（用于详细模式），那么GDB会在执行每个命令时显示它。该选项必须在 文件名 之前给出，并在其他任何地方都被解释为文件名的一部分。

在交互式使用时会要求确认的命令，在命令文件中使用时会直接执行而不询问。许多通常会打印消息说明其操作的GDB命令，从命令文件中调用时会省略这些消息。

GDB也接受来自标准输入的命令输入。在这种模式下，正常输出到标准输出，错误输出到标准错误。从标准输入提供的命令文件中的错误不会终止命令文件的执行 —— 执行会继续处理下一个命令。 

gdb < cmds > log 2>&1

（上述语法会根据所使用的 shell 而有所不同。）这个例子将执行文件 cmds 中的命令。所有输出和错误都将被定向到 log。

由于存储在命令文件中的命令往往比交互式输入的命令更通用，它们经常需要处理复杂的情况，例如变量和符号的不同或意外值、被调试程序构建方式的变化等。GDB提供了一组流控制命令来处理这些复杂性。使用这些命令，你可以编写复杂的脚本，例如遍历数据结构、有条件地执行命令等。

if ¶
else

这个命令允许在你的脚本中包含有条件执行的命令。if 命令接受一个参数，即要求值的表达式。后面跟着一系列只有在表达式为真（其值非零）时才会执行的命令。 然后可以选择有一个 else 行，后面跟着一系列只有在表达式为假时才会执行 的命令。列表的结尾由包含 end

while ¶

这个命令允许编写循环。它的语法与 if 类似：该命令接受一个参数，即要求值的表达式，后面必须跟着要执行的命令，每行一个，以 end 结束。这些命令被称为循环的 主体。只要表达式求值为真，while 主体中的命令就会被重复执行。

loop_break ¶

这个命令退出包含它的 while 循环。在那个 while 的 end 行之后，脚本的执行继续。

loop_continue ¶

这个命令跳过包含它的 while 循环主体中其余命令的执行。执行分支到 while 循环的开头，在那里它会求值控制表达式。

end ¶

终止作为 if、else 或 while 流控制命令主体的命令块。
