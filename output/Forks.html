4.11调试分叉 ¶

在大多数系统上，GDB没有对使用 fork 函数创建额外进程的程序进行调试的特殊支持。当一个程序进行分叉时，GDB将继续调试父进程，而子进程将不受阻碍地运行。如果你在子进程随后执行的任何代码中设置了断点，子进程将收到一个 SIGTRAP 信号（除非它捕获了该信号），这将导致它终止。

然而，如果您想调试子进程，有一个不太麻烦的解决方法。在子进程在分叉后执行的代码中添加对 sleep 的调用。仅在设置了某个环境变量或某个文件存在时进行睡眠可能会很有用，这样当您不想在子进程上运行GDB时就不需要延迟。当子进程正在睡眠时，使用 ps 程序获取它的进程ID。然后告诉GDB（如果您也在调试父进程，则是新调用的GDB）附加到子进程（请参阅 调试一个已经运行的进程）。从那时起，您就可以像调试任何其他附加的进程一样调试子进程。

在某些系统上，GDB提供了对使用 fork 或 vfork函数创建额外进程的程序进行调试地支持。在GNU/Linux平台上，内核版本2.5.46及更高版本支持此功能。

分叉调试命令在原生模式下以及在 target remote 模式或 target extended-remote 模式下连接到 gdbserver 时均受支持。

默认情况下，当一个程序进行分叉时，GDB将继续调试父进程，而子进程将不受阻碍地运行。

如果您想跟踪子进程而不是父进程，请使用命令 set follow-fork-mode。

set follow-fork-mode 模式 ¶

设置调试器对程序调用 fork 或 vfork 的响应。对 fork 或 vfork 的调用会创建一个新进程。模式 参数可以是：

parent

分叉后调试原始进程。子进程不受阻碍地运行。这是默认设置。

child

分叉后调试新进程。父进程不受阻碍地运行。

show follow-fork-mode ¶

显示调试器对 fork 或 vfork 调用的当前响应。

在Linux上，如果您想调试父进程和子进程，请使用命令 set detach-on-fork。

¶

告诉gdb在分叉后是分离其中一个进程，还是保留对它们两者的调试器控制。

on

子进程（或父进程，取决于 follow-fork-mode 的值）将被分离并允许独立运行。这是默认设置。

off

两个进程都将由GDB控制。一个进程（子进程或父进程，取决于 follow-fork-mode 的值）像往常一样被调试，而另一个进程被暂停。

show detach-on-fork ¶

显示detach-on-fork模式是开启还是关闭。

如果您选择将“detach-on-fork”模式设置为关闭，那么GDB将保留对所有分叉进程（包括嵌套分叉）的控制。您可以使用 info inferiors 命令列出由GDB控制的分叉进程，并使用 inferior 命令在不同的分叉之间切换（请参阅 调试多个下级连接和程序）。

要停止调试其中一个分叉进程，您可以使用 detach inferiors 命令从它分离（允许它独立运行），或者使用 kill inferiors 命令杀死它。请参阅 调试多个下级连接和程序。

如果您要求调试子进程并且在 vfork 之后紧接着是 exec，GDB会执行新目标直到新目标中的第一个断点。如果您在原始程序中的 main 上设置了断点，该断点也会设置在子进程的 main 上。

在某些系统上，当子进程由 vfork 产生时，在 exec 调用完成之前，您无法调试子进程或父进程。

如果在 exec 调用执行后向GDB发出 run 命令，新目标将重新启动。要重新启动父进程，请使用 file命令并将父可执行文件名作为其参数。默认情况下，在 exec 调用执行后，GDB会丢弃上一个可执行映像的符号。您可以使用 set follow-exec-mode 命令更改此行为。

set follow-exec-mode 模式 ¶

设置调试器对程序调用 exec 的响应。exec 调用会替换进程的程序映像。

follow-exec-mode 可以是：

new

GDB创建一个新的下级并将进程重新绑定到这个新的下级。在 exec 调用之前进程正在运行的程序可以在之后通过重新启动原始下级来重新启动。

例如： 

(gdb) info inferiors
(gdb) info inferior  
  Id   Description   Executable
* 1    <null>        prog1
(gdb) run
process 12020 is executing new program: prog2
Program exited normally.
(gdb) info inferiors
  Id   Description   Executable
  1    <null>        prog1
* 2    <null>        prog2
same

GDB使进程绑定到同一个下级。新的可执行映像替换下级中加载的上一个可执行文件。在 exec 调用之后使用例如 run 命令重新启动下级，会重新启动进程在 exec 调用之后正在运行的可执行文件。这是默认模式。

例如： 

(gdb) info inferiors
  Id   Description   Executable
* 1    <null>        prog1
(gdb) run
process 12020 is executing new program: prog2
Program exited normally.
(gdb) info inferiors
  Id   Description   Executable
* 1    <null>        prog2

follow-exec-mode 在原生模式和 target extended-remote 模式下均受支持。

您可以使用 catch 命令使GDB在每次进行 fork、vfork 或 exec 调用时停止。请参阅 设置捕获点。
