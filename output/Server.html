20.3 使用 gdbserver 程序 ¶

gdbserver 是一个用于类Unix系统的控制程序，它允许你通过 target remote 或 target extended-remote 将你的程序与远程 GDB 连接起来 —— 但无需链接通常的调试存根。

gdbserver 并不是调试存根的完全替代品，因为它本质上需要与 GDB 本身相同的操作系统设施。实际上，一个能够运行 gdbserver 以连接到远程 GDB 的系统也可以在本地运行 GDB！不过，gdbserver 有时还是很有用的，因为它是一个比 GDB 本身小得多的程序。它也比整个 GDB 更容易移植，所以你可以通过使用 gdbserver 在新系统上更快地开始。最后，如果您为实时系统开发代码，您可能会发现实时操作中涉及的权衡使得在另一个系统上尽可能多地进行开发工作更加方便，例如通过交叉编译。您可以使用 gdbserver 进行类似的调试选择。

GDB 和 gdbserver 通过串行线或TCP连接进行通信，使用标准的 GDB 远程串行协议。

警告： gdbserver 没有任何内置安全性。不要在连接到任何公共网络的情况下运行 gdbserver；与 gdbserver 的 GDB 连接提供了对目标系统的访问权限，其权限与运行 gdbserver 的用户相同。

20.3.1 运行 gdbserver ¶

在目标系统上运行 gdbserver。你需要一份你想要调试的程序副本，包括它所需的任何库。gdbserver 不需要你的程序的符号表，所以如果需要节省空间，你可以去除程序的符号表。主机系统上的 GDB 将处理所有符号。

要使用服务器，你必须告诉它如何与 GDB 通信；你的程序的名称；以及你的程序的参数。通常的语法是： 

目标> gdbserver 通信方式 程序 [ 参数 ... ]

通信方式 可以是设备名称（用于使用串行线），或者是TCP主机名和端口号，或者是 - 或 stdio 以使用 gdbserver 的标准输入/输出。 例如，要使用参数 ‘foo.txt’ 调试Emacs，并通过串行端口 /dev/com1 与 GDB 通信： 

目标> gdbserver /dev/com1 emacs foo.txt

gdbserver 被动地等待主机 GDB 与它通信。

要使用TCP连接而不是串行线： 

目标> gdbserver host:2345 emacs foo.txt

与前一个例子的唯一区别是第一个参数，它指定你正在通过TCP与主机 GDB 通信。‘host:2345’ 参数意味着 gdbserver 期望从机器 ‘host’ 到本地TCP端口2345建立TCP连接。（目前，‘host’ 部分被忽略。）只要端口号不与目标系统上已使用的任何TCP端口冲突（例如，23 保留给 telnet），你可以选择任何你想要的端口号。16 你必须在主机 GDB 的 target remote 命令中使用相同的端口号。

当通过ssh启动 gdbserver 时，stdio 连接很有用： 

(gdb) target remote | ssh -T hostname gdbserver - hello

提供给ssh的 ‘-T’ 选项是因为我们不需要远程伪终端，也不需要转义字符处理。当提供一个命令时，ssh默认会这样做，提供这个标志是为了使其明确。如果你愿意，你可以省略它。

使用stdio连接启动的程序将 /dev/null 作为 stdin，并且 stdout、stderr 通过连接到 gdbserver 的管道发送回gdb进行显示。 stdout 和 stderr 使用同一个管道。

20.3.1.1 附加到正在运行的程序 ¶

在某些目标上，gdbserver 还可以附加到正在运行的程序。这是通过 --attach 参数完成的。语法是： 

目标> gdbserver --attach 通信方式 进程ID

进程ID 是当前正在运行的进程的进程ID。不需要将 gdbserver 指向正在运行进程的二进制文件。

在 target extended-remote 模式下，你也可以使用 GDB 的attach命令进行附加（见 在远程连接类型中附加）。

如果你的目标系统有 pidof 实用程序，你可以按进程名而不是进程ID来调试进程： 

目标> gdbserver --attach 通信方式 `pidof 程序`

如果运行着多个 程序 副本，或者 程序 有多个线程，大多数版本的 pidof 支持 -s 选项，只返回第一个进程ID。

20.3 .1.2 gdbserver的TCP端口分配生命周期 ¶

本节仅适用于运行 gdbserver 以监听TCP端口的情况。

在 target remote 模式下，当所有被调试的进程都终止后，gdbserver 通常会终止。另一方面，对于 target extended-remote，即使没有进程了，gdbserver 仍会保持运行。GDB 通常在退出时终止派生的被调试进程，这通常也会在 target remote 模式下终止 gdbserver。因此，当连接意外中断，并且 GDB 无法要求 gdbserver 杀死其被调试进程时，即使在 target remote 模式下，gdbserver 也会保持运行。

当 gdbserver 保持运行时，GDB 稍后可以再次连接到它。这种重新连接对于诸如 断开连接的跟踪 等功能很有用。为了完整起见，一次最多只能连接一个 GDB。
