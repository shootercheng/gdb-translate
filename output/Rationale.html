F.5 原理 ¶

上述一些设计决策可能存在争议。

栈溢出/下溢怎么办？

GDB应该能够查询目标以发现其栈大小。有了这些信息，GDB可以在翻译时确定给定表达式是否会导致栈溢出。但本规范并非关于GDB应该进行何种错误检查。

为什么要用LONGEST来做所有事情？

速度并不重要，但代理代码大小很重要；使用LONGEST会引入一堆支持代码来进行除法等操作。所以这是一个严重的问题。

首先，请注意，对于不同的操作数大小，你不需要不同的字节码。你可以在不知道目标上栈元素实际大小的情况下生成代码。如果目标只支持32位整数，而你不发送任何64位字节码，一切仍然可以正常工作。这里的观察结果是，MIPS和Alpha只有固定大小的寄存器，即使大多数指令只对全尺寸字进行操作，你仍然可以获得C的语义。你只需要确保在正确的时间对所有内容进行正确的符号扩展。所以不需要字节码的32位和64位变体。只需使用你支持的最大大小来实现所有内容即可。

GDB当然应该检查目标支持哪些大小，以便用户能更早而不是更晚得到错误。但这些信息对于正确性来说并非必要。

为什么没有>或<=运算符？

我希望保持解释器小巧，而且我们不需要它们。我们可以将less_操作码与log_not组合，并交换操作数的顺序，从而得到所有四个不对称比较运算符。例如，(x <= y) 等同于 ! (x > y)，也就是 ! (y < x)。

为什么有log_not？
为什么有ext？
为什么有zero_ext？

这些都可以很容易地从其他指令合成，但我预计它们会经常被使用，而且它们很简单，所以我把它们包含进来以保持字节码字符串简短。

log_not 等同于 const8 0 equal；它在一半的关系运算符中使用。

ext n 等同于 const8 s - n lsh const8 s - n rsh_signed -- 其中 s 是栈元素的大小；当值应该是有符号时，它紧跟在 refm 和 reg 字节码之后。见下一个项目符号列表。

zero_ext n 等同于 constm mask log_and；每当我们推送寄存器的值时都会使用它，因为我们不能假设寄存器的高位不是垃圾值。

为什么没有带符号扩展的ref运算符变体？

因为那样会使ref运算符的数量翻倍，而且无论如何我们访问位域时都需要ext字节码。

为什么没有带常量地址的ref运算符变体？

因为那样又会使ref运算符的数量翻倍，而const32 address ref32只长一个字节。

为什么refn运算符必须支持未对齐读取？

只要可执行文件的调试信息告诉GDB，GDB就会生成在未对齐地址读取多字节值的字节码。此外，GDB在生成字节码时并不知道指针的值，所以它无法确定特定的读取是否会对齐。

特别是，结构体位域可能有几个字节长，但不遵循对齐规则；紧凑结构体的成员也不一定对齐。

一般来说，在正确的C代码中有许多未对齐引用的情况，要么是程序员明确要求，要么是编译器自行决定。因此，让GDB代理字节码在所有情况下都能正确工作，比让GDB在每种情况下都猜测编译器是否做了常规处理要简单。
