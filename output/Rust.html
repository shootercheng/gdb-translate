15.4.8 Rust ¶

GDB支持 Rust编程语言。类型和值的打印以及表达式解析相当完整。然而，有一些需要注意的特殊情况和漏洞。

  • 行规范（见 位置规范）从不相对于当前的包。相反，它们的行为就好像存在一个全局的包命名空间，有点类似于 extern crate 的行为方式。

    也就是说，如果 GDB 在包 ‘A’ 的模块 ‘B’ 中的一个函数的断点处停止，那么 break B::f 将尝试在名为 ‘B’ 的包中名为 ‘f’ 的函数处设置一个断点。

    由于这种方法，行规范也不能使用 ‘self::’ 或 ‘super::’ 来引用项。

  • 因为 GDB 在表达式中实现了Rust的名称查找语义，它有时会在名称前加上当前的包。例如，如果 GDB 在包 ‘K’ 的一个断点处停止，那么 print ::x::y 将尝试查找符号 ‘K::x::y’。

    然而，由于在调试时能够引用其他包很有用，GDB 提供了 extern 扩展来规避此问题。要使用该扩展，只需在路径表达式前加上 extern 来引用否则不可用的 “全局” 作用域。

    在上述示例中，如果您想引用包 ‘x’ 中的符号 ‘y’，您将使用 print extern x::y。

  • Rust表达式求值器不支持 “类似语句” 的表达式，如 if 或 match，也不支持lambda表达式。
  • 元组表达式未实现。
  • Rust表达式求值器目前未实现 Drop 特性。求值器可能创建的对象永远不会被销毁。
  • GDB 未实现泛型的类型推断。为了调用泛型函数或以其他方式引用泛型项，则必须手动指定类型参数！
  • GDB 目前对Rust使用C++ 去混淆器。在大多数情况下，这不会导致任何问题。然而，在表达式上下文中，完成泛型函数名会给出语法上无效的结果。这是因为Rust要求在函数名及其泛型参数之间使用 ‘::’ 运算符。例如，GDB 可能提供一个补全，如 crate::f<u32>，而解析器需要 crate::f::<u32>！
  • 截至撰写本文时，Rust编译器（版本1.8）在其生成的调试信息中有一些漏洞。这些漏洞阻止了 GDB 实现某些功能：
    • 不能通过特性进行方法调用。
    • 未实现运算符重载。
    • 在单态化函数中调试时，不能使用泛型类型名称。
    • 类型 Self 不可用。
    • use 语句不可用，因此某些名称在包中可能不可用。
