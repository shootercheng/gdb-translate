5.1.2 设置观察点 ¶

你可以使用观察点在表达式的值发生变化时停止执行，而无需预测这种情况可能发生的特定位置。（这有时被称为数据断点。）表达式可以像单个变量的值一样简单，也可以像由运算符组合的多个变量一样复杂。示例包括：

  • 对单个变量值的引用。
  • 转换为适当数据类型的地址。例如，‘*(int *)0x12345678’ 将观察指定地址处的4字节区域（假设 int 占用4字节）。
  • 任意复杂的表达式，如 ‘a*b + c/d’。该表达式可以使用程序原生语言中有效的任何运算符（见 使用不同语言的GDB）。

即使表达式尚未求值，你也可以为其设置观察点。例如，在 ‘global_ptr’ 初始化之前，你可以为 ‘*global_ptr’ 设置观察点。当你的程序设置 ‘global_ptr’ 且表达式产生有效值时，GDB将停止。如果表达式通过除修改变量之外的其他方式变得有效（例如，如果 ‘*global_ptr’ 指向的内存由于 malloc 调用而变得可读），GDB可能直到表达式下次变化时才会停止。

根据你的系统，观察点可能通过软件或硬件实现。GDB通过单步执行你的程序并每次测试变量的值来进行软件观察点设置，这比正常执行慢数百倍。（但这可能仍然值得，以捕获你不知道程序的哪个部分是罪魁祸首的错误。）

在某些系统上，例如大多数基于PowerPC或x86的目标，GDB包括对硬件观察点的支持，这不会减慢程序的运行速度。

watch [-l|-location] expr [thread thread-id] [mask maskvalue] [task task-id] ¶

为表达式设置观察点。当程序写入表达式 expr 并使其值发生变化时，GDB将中断。此命令最简单（也是最常用）的用法是观察单个变量的值： 

(gdb) watch foo

如果命令包含 [thread thread-id] 参数，GDB仅在由 thread-id 标识的线程修改变量 expr 的值时中断。如果任何其他线程修改变量 expr 的值，GDB不会中断。请注意，以这种方式限制为单个线程的观察点仅适用于硬件观察点。

同样，如果给出 task 参数，那么观察点将特定于指定的Ada任务（见 Ada任务扩展）。

通常，观察点尊重表达式中变量的作用域（见下文）。-location 参数告诉GDB改为观察表达式引用的内存。在这种情况下，GDB将计算表达式 expr，获取结果的地址，并观察该地址处的内存。结果的类型用于确定观察内存的大小。如果表达式的结果没有地址，GDB将打印错误。

[mask maskvalue] 参数允许创建掩码观察点，如果当前架构支持此功能（例如，PowerPC嵌入式架构，见 PowerPC嵌入式）。掩码观察点除了要观察的地址外还指定一个掩码。掩码指定在将下级访问的地址与观察点地址进行匹配时应忽略地址的某些位（掩码中重置的位）。因此，掩码观察点同时观察许多地址 —— 那些未掩码位与观察点地址中的未掩码位相同的地址。mask 参数意味着 -location。 示例： 

(gdb) watch foo mask 0xffff00ff
(gdb) watch *0xdeadbeef mask 0xffffff00
rwatch [-l|-location] expr [thread thread-id] [mask maskvalue] ¶

设置一个观察点，当程序读取表达式 expr 的值时将中断。

awatch [-l|-location] expr [thread thread-id] [mask maskvalue] ¶

设置一个观察点，当程序读取或写入表达式 expr 时将中断。

info watchpoints [list…] ¶

此命令使用与 info break 相同的格式打印观察点列表（见 设置断点）。

如果你要观察数值输入地址的变化，需要对其进行解引用，因为地址本身只是一个永远不会改变的常量。GDB拒绝创建观察永远不变值的观察点： 

(gdb) watch 0x600850
无法观察常量值0x600850。
(gdb) watch *(int *) 0x600850
观察点1：*(int *) 6293584

GDB会尽可能设置硬件观察点。硬件观察点执行速度非常快，调试器会在值发生变化的精确指令处报告值的变化。如果GDB无法设置硬件观察点，它会设置一个软件观察点，软件观察点执行速度较慢，会在变化发生后的下一条 语句 而不是指令处报告值的变化。

你可以使用 set can-use-hw-watchpoints 0 命令强制GDB仅使用软件观察点。将此变量设置为零时，GDB将永远不会尝试使用硬件观察点，即使底层系统支持它们。（请注意，在将 can-use-hw-watchpoints 设置为零之前设置的硬件辅助观察点仍将使用硬件机制来观察表达式的值。）

set can-use-hw-watchpoints ¶

设置是否使用硬件观察点。

show can-use-hw-watchpoints ¶

显示当前使用硬件观察点的模式。

对于远程目标，你可以限制GDB将使用的硬件观察点数量，见 设置远程硬件断点限制。

当你发出 watch 命令时，GDB会报告 

硬件观察点 num: expr

如果它能够设置硬件观察点。

目前，awatch 和 rwatch 命令只能设置硬件观察点，因为在不检查每条正在执行的指令的情况下，无法检测到不改变观察表达式值的数据访问，而GDB目前不会这样做。如果GDB发现无法使用 awatch 或 rwatch 命令设置硬件断点，它会打印如下消息： 

表达式无法用读/访问观察点实现。

有时，GDB无法设置硬件观察点是因为观察表达式的数据类型比目标机器上的硬件观察点所能处理的更宽。例如，某些系统只能观察最多4字节宽的区域；在这样的系统上，你不能为产生双精度浮点数（通常为8字节宽）的表达式设置硬件观察点。作为一种解决方法，可能可以将大区域分解为一系列较小的区域，并使用单独的观察景点进行观察。

如果你设置了太多硬件观察点，当你恢复程序执行时，GDB可能无法插入所有观察点。由于在程序即将恢复之前，活动观察点的确切数量是未知的，GDB在你设置观察点时可能无法警告你，并且只有在程序恢复时才会打印警告： 

硬件观察点 num: 无法插入观察点

如果发生这种情况，请删除或禁用一些观察点。

观察引用许多变量的复杂表达式也可能耗尽硬件辅助观察点可用的资源。这是因为GDB需要使用单独分配的资源来观察表达式中的每个变量。

如果你使用 print 或 call 交互式调用函数，你设置的任何观察点在GDB到达另一种断点或调用完成之前都将无效。

当局部（自动）变量或涉及此类变量的表达式超出作用域时，即当执行离开定义这些变量的块时，GDB会自动删除观察它们的观察点。特别是，当被调试的程序终止时，所有 局部变量都会超出作用域，因此只有观察全局变量的观察点会保留设置。如果你重新运行程序，你需要再次设置所有此类观察点。一种方法是在 main 函数的入口处设置一个代码断点，当它中断时，设置所有观察点。

在多线程程序中，观察点将检测每个线程对观察表达式的更改。

警告： 在多线程程序中，软件观察点的作用有限。如果GDB创建了一个软件观察点，它只能观察单个线程中表达式的值。如果你确信表达式只能由于当前线程的活动而改变（并且如果你也确信没有其他线程可以成为当前线程），那么你可以像往常一样使用软件观察点。但是，当非当前线程的活动改变表达式时，GDB可能不会注意到。（相比之下，硬件观察点会观察所有线程中的表达式。）

见 设置远程硬件观察点限制。
