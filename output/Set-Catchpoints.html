5.1.３ 设置捕获点 ¶

你可以使用捕获点使调试器在某些类型的程序事件发生时停止，例如C++异常或共享库的加载情况。使用catch命令来设置捕获点。

catch event ¶

当event发生时停止。event可以是以下任何一种：

throw [regexp] ¶
rethrow [regexp]
catch [regexp]

C++异常的抛出、重新抛出或捕获。

如果给出了regexp，那么只有类型与正则表达式匹配的异常才会被捕获。

在某些系统上，便利变量$_exception在与异常相关的捕获点处可用。它保存着正在抛出的异常。

目前GDB中C++异常处理存在一些限制：

  • 对这些命令的支持因系统而异。目前，仅支持使用‘gnu-v3’ C++ ABI的系统（见 配置当前ABI）。
  • 正则表达式功能和$_exception便利变量依赖于libstdc++中某些SDT探针的存在。如果这些探针不存在，那么这些功能就无法使用。这些探针最早在GCC 4.8版本中可用，但你的GCC中是否有这些探针还取决于其构建方式。
  • $_exception便利变量仅在设置与异常相关的捕获点的指令处有效。
  • 当命中与异常相关的捕获点时，GDB会在实现C++运行时异常支持的系统库中的某个位置停止，通常是libstdc++。你可以使用up（见 选择帧）来进入你的代码。
  • 如果你交互式地调用一个函数，当函数执行完毕时，GDB通常会将控制权返回给你。然而，如果调用引发了异常，该调用可能会绕过将控制权返回给你的机制，导致你的程序要么中止，要么继续运行直到它命中一个断点、捕获一个GDB正在监听的信号或退出。即使你为异常设置了捕获点，在交互式调用中异常捕获点也是禁用的。有关使用set unwind-on-terminating-exception控制此行为的信息，请参见 调用程序函数。
  • 你不能交互式地引发异常。
  • 你不能交互式地安装异常处理程序。
exception [name] ¶

引发的Ada异常。如果在命令末尾指定了异常名称（例如 catch exception Program_Error），调试器将仅在引发此特定异常时停止。否则，当引发任何Ada异常时，调试器将停止执行。

当在用户定义的异常上插入异常捕获点，且该异常名称与语言定义的异常之一相同时，必须使用完全限定名称作为异常名称。否则，GDB将假定它应在预定义异常上停止，而不是用户定义的异常。例如，假设在包Pck中定义了一个名为Constraint_Error的异常，那么用于捕获此类异常的命令是 catch exception Pck.Constraint_Error。

便利变量$_ada_exception保存正在抛出的异常的地址。在为这样的捕获点设置条件时，这可能会很有用。

exception unhandled ¶

引发但未被程序处理的异常。便利变量$_ada_exception的设置与 catch exception相同。

handlers [name] ¶

正在处理的Ada异常。如果在命令末尾指定了异常名称 （例如 catch handlers Program_Error），调试器将仅在处理此特定异常时停止。 否则，当处理任何Ada异常时调试器将停止执行。

当在用户定义的 异常上插入处理程序捕获点，且该异常名称与语言定义的异常之一相同时，必须使用完全限定名称作为异常名称。否则，GDB将假定它 应在预定义异常上停止，而不是用户定义的异常。例如，假设一个名为 Constraint_Error的异常在包Pck中定义，那么用于捕获此类异常处理的命令是 catch handlers Pck.Constraint_Error。

便利变量$_ada_exception的设置与 catch exception相同。

assert ¶

Ada断言失败。请注意，此捕获点不会设置便利变量 $_ada_exception。

exec ¶

对 exec 的调用。

syscall ¶
syscall [name | number |\span class="r">group:groupname | \span class="r">g:groupname] …

对系统调用（也称为syscall）的调用或从系统调用返回。系统调用是应用程序向操作系统（OS）或OS系统服务之一请求服务 的一种机制。GDB可以捕获被调试程序发出的一些或所有系统调用，并显示每个系统调用的相关信息。如果未指定任何参数，则将捕获对所有系统调用的调用和从所有系统调用的返回。

name可以是对底层OS有效的任何系统调用名称。具体哪些系统调用有效取决于OS。在GNU和Unix系统上，你可以在 /usr/include/asm/unistd.h 中找到有效系统调用名称的完整列表。

通常GDB预先知道每个OS哪些系统调用是有效的，因此你可以使用GDB命令行完成功能（见 命令完成）来列出可用选项。

你也可以按数字指定系统调用。系统调用的编号是传递给OS的系统调用调度器以标识所请求服务的值。当你通过名称指定系统调用时，GDB使用其系统调用数据库将名称转换为相应的数字代码，但如果GDB的数据库没有你系统上的完整系统调用列表（例如，因为GDB落后于OS升级），直接使用编号可能会很有用。

你可以使用group:语法（g:是等效的简写）一次性指定要捕获的一组相关系统调用。例如，在某些平台上，GDB允许你通过将参数 group:network 传递给 catch syscall 来捕获所有与网络相关的系统调用。请注意，并非所有系统调用组在每个系统上都可用。你可以使用命令完成功能（见 命令完成）来列出你环境中可用的系统调用组。

以下示例说明了如果不提供参数，此命令的工作方式：

(gdb) catch syscall  
Catchpoint 1 (syscall)  
(gdb) r  
Starting program: /tmp/catch-syscall  
  
Catchpoint 1 (call to syscall 'close'), \  
	   0xffffe424 in __kernel_vsyscall ()  
(gdb) c  
Continuing.  
  
Catchpoint 1 (returned from syscall 'close'), \  
	0xffffe424 in __kernel_vsyscall ()  
(gdb)

这是通过名称捕获系统调用的示例：

(gdb) r

Catchpoint 1 (call to syscall 'chroot'), \

(gdb) c
