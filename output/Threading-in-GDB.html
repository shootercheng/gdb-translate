23.3.2.2 GDB中的线程 ¶

GDB不是线程安全的。如果你的Python程序使用多个线程，你必须小心，只在GDB线程中调用特定于GDB的函数。GDB提供了一些函数来帮助处理这个问题。

函数：gdb.blocked_signals () ¶

如前所述（见 基础Python），某些信号必须传递到GDB主线程。blocked_signals函数返回一个上下文管理器，它将在进入时阻塞这些信号。这可以在启动新线程时使用，以确保在那里阻塞信号，如下所示： 

with gdb.blocked_signals():
   start_new_thread()
类：gdb.Thread ¶

这是Python的 threading.Thread 类的子类。它重写了 start 方法以调用 blocked_signals，这使得它成为一个易于使用的直接替代品，用于创建在GDB中能正常工作的线程。

函数：gdb.interrupt () ¶

这会使GDB的行为就好像用户在终端输入了一个Control-C字符一样。也就是说，如果底层程序正在运行，它会被中断；如果一个GDB命令正在执行，则会停止；如果一个Python命令正在运行，将会引发 KeyboardInterrupt。

与GDB中的大多数Python API不同，interrupt是线程安全的。

函数：gdb.post_event (event) ¶

将一个无参数的可调用对象event放入GDB的内部事件队列中。这个可调用对象将在稍后的某个时间点，在GDB的事件处理过程中被调用。使用 post_event 发布的事件将按照它们发布的顺序运行；但是，无法知道它们相对于GDB内部的其他事件何时会被处理。

与GDB中的大多数Python API不同，post_event是线程安全的。例如： 

(gdb) python
>import threading
>
>class Writer():
> def __init__(self, message):
>        self.message = message;
> def __call__(self):
>        gdb.write(self.message)
>
>class MyThread1 (threading.Thread):
> def run (self):
>        gdb.post_event(Writer("Hello "))
>
>class MyThread2 (threading.Thread):
> def run (self):
>        gdb.post_event(Writer("World\n"))
>
>MyThread1().start()
>MyThread2().start()
>end
(gdb) Hello World
