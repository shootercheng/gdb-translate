4.2启动你的程序 ¶

run ¶
r

使用run命令在GDB下启动你的程序。 你必须首先使用GDB的参数指定程序名（见进入和退出GDB），或者使用file或exec-file命令（见指定文件的命令）。

如果你在一个支持进程的执行环境中运行你的程序，run会创建一个子进程并使该进程运行你的程序。在一些没有进程的环境中，run会跳转到你的程序的开头。其他目标，如“remote”，总是在运行。如果你得到这样的错误消息： 

“remote”目标不支持“run”。
试试“help target”或“continue”。

那么使用continue来运行你的程序。你可能首先需要load（见load）。

程序的执行会受到它从上级接收的某些信息的影响。GDB提供了指定这些信息的方法，你必须在启动程序之前进行指定。（你可以在启动程序后更改它，但这样的更改只会在你下次启动程序时影响你的程序。）这些信息可以分为四类：

参数。

将给你的程序的参数指定为run命令的参数。如果你的目标上有一个可用的shell，那么会使用该shell来传递参数，这样你就可以在描述参数时使用正常的约定（如通配符扩展或变量替换）。 在Unix系统中，你可以使用SHELL环境变量来控制使用哪个shell。如果你没有定义SHELL，GDB会使用默认的shell（/bin/sh）。你可以使用set startup-with-shell命令禁用任何shell的使用（详情见下文）。

环境。

你的程序通常从GDB继承其环境，但你可以使用GDB命令set environment和unset environment来更改影响你的程序的环境部分。见你的程序的环境。

工作目录。

你可以使用命令set cwd设置你的程序的工作目录。如果你没有使用这个命令设置任何工作目录，如果是本地调试，你的程序将继承GDB的工作目录；如果是远程调试，将继承远程服务器的工作目录。见你的程序工作目录。

标准输入和输出。

你的程序通常使用与GDB相同的设备作为标准输入和标准输出。你可以在run命令行中重定向输入和输出，或者使用tty命令为你的程序设置不同的设备。 见你的程序的输入和输出。

警告：虽然输入和输出重定向有效，但你不能使用管道将你正在调试的程序的数据传递给另一个程序；如果你尝试这样做，GDB可能最终调试错误的程序。

当你发出run命令时，你的程序会立即开始执行。关于如何安排你的程序停止，请见停止和继续。一旦你的程序停止，你可以使用print或call命令调用你的程序中的函数。见检查数据。

如果自GDB上次读取符号文件以来，你的符号文件的修改时间发生了变化，GDB会丢弃其符号表并再次读取它。当它这样做时，GDB会尝试保留你当前的断点。

start ¶

主程序的名称可能因语言而异。对于C或C++语言，主程序名称始终是main，但其他语言，如Ada，不需要为其主程序指定特定名称。调试器提供了一种方便的方法来启动程序的执行并在主程序的开头停止，这取决于所使用的语言。

“start”命令相当于在主程序的开头设置一个临时断点，然后调用“run”命令。

一些程序包含一个详细说明阶段，在调用主程序之前会执行一些启动代码。这取决于编写你的程序所使用的语言。例如，在C++中，静态和全局对象的构造函数在调用main之前会被执行。因此，调试器可能会在到达主程序之前停止。然而，临时断点将保留以停止执行。

将给你的程序的参数指定为“start”命令的参数。这些参数将原封不动地传递给底层的“run”命令。请注意，如果在后续调用“start”或“run时没有提供参数，将重用相同的参数。

有时需要在详细说明阶段调试程序。在这些情况下，使用start命令会使你的程序的执行停止得太晚，因为程序已经完成了详细说明阶段。在这种情况下，要么在运行程序之前在你的详细说明代码中插入断点，要么使用starti命令。

starti ¶

“starti”命令相当于在程序执行的第一条指令处设置一个临时断点，然后调用“run”命令。对于包含详细说明阶段的程序，“starti”命令将在详细说明阶段的开始处停止执行。

set exec-wrapper wrapper ¶
show exec-wrapper
unset exec-wrapper

当设置了“ exec-wrapper”时，指定的包装器用于启动程序进行调试。GDB使用形式为exec wrapper program的shell命令启动你的程序。会为program及其参数添加引号，但不会为wrapper添加引号，因此如果适合你的shell，你应该添加引号。包装器运行直到它执行你的程序，然后GDB接管控制。

你可以使用任何最终使用其参数调用execve的程序作为包装器。几个标准的Unix实用程序可以做到这一点，例如env和nohup。任何以exec "$@"结尾的Unix shell脚本也可以工作。

例如，你可以使用env将一个环境变量传递给被调试的程序，而无需在你的shell环境中设置该变量： 

(gdb) set exec-wrapper env 'LD_PRELOAD=libtest.so'
(gdb) run

在大多数目标上进行本地调试时，此命令可用，但不包括DJGPP、Cygwin和MS Windows。

set startup-with-shell
set startup-with-shell on
set startup-with-shell off
show startup-with-shell

在Unix系统上，默认情况下，如果你的目标上有一个可用的shell，GDB会使用它来启动你的程序。“run”命令的参数会传递给shell，shell会进行变量替换、展开通配符字符并执行I/O重定向。在某些情况下，禁用这种对shell的使用可能会很有用，例如，当调试shell本身或诊断启动失败时，如： 

(gdb) run
启动程序：./a.out
在启动期间程序以信号SIGSEGV终止，段错误。

这表明指定的shell或使用“ exec-wrapper”指定的包装器崩溃了，而不是你的程序。最常见的情况是，这是由你的shell的非交互式模式初始化文件中的某些奇怪内容引起的，例如C-shell的.cshrc、Z shell的$.zshenv，或者BASH的BASH_ENV环境变量中指定的文件。
