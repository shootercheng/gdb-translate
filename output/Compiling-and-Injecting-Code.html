17.7 在GDB中编译和注入代码 ¶

GDB支持按需编译以及将代码注入到在GDB下运行的程序中。要启用此功能，必须安装使用libcc1.so构建的GCC 5.0或更高版本。 此功能通过以下命令实现。

compile code 源代码 ¶
compile code -raw -- 源代码

使用在GDB中找到的当前语言的编译器编译源代码（见使用GDB与不同语言）。如果使用GDB中指定的当前语言不支持编译和注入，或者编译器不支持此功能，将打印错误消息。如果源代码编译并链接成功，GDB将加载生成的目标代码，并在当前选定的进程上下文中执行它。需要注意的是，编译后的代码会立即执行。执行后，编译后的代码将从GDB中删除，并且您定义的任何新类型或变量都将被删除。

该命令允许您以两种方式指定源代码。 最简单的方法是向命令提供一行代码。例如： 

compile code printf ("hello world\n");

如果您在命令行上指定选项以及源代码，它们可能会冲突，可以使用“--”分隔符将选项与实际源代码分开。例如： 

compile code -r -- printf ("hello world\n");

或者，您可以将源代码作为多行文本输入。要进入此模式，请在命令后不跟任何文本调用“compile code”命令。这将启动多行编辑器，并允许您根据需要输入任意行数的源代码。完成输入后，在其自己的行上输入“end”以退出编辑器。 

compile code
>printf ("hello\n");
>printf ("world\n");
>end

指定“-raw”可防止GDB将提供的源代码包装在可调用范围内。在这种情况下，您必须通过定义名为_gdb_expr_的函数来指定代码的入口点。例如，当源代码需要“#include”行，否则可能会与进程符号冲突时，可能需要使用“-raw”选项。

compile file 文件名 ¶
compile file -raw 文件名

与“compile code”类似，但从文件名中获取源代码。 

compile file /home/user/example.c

文件名参数支持转义和平引用，见作为命令参数的文件名。

compile print [[选项] --] 表达式
compile print [[选项] --] /f 表达式

使用在GDB中找到的当前语言的编译器编译并执行表达式（见使用GDB与不同语言）。默认情况下，表达式的值将以适合其数据类型得格式打印；您可以通过指定“/f”选择不同的格式，其中f是指定格式的字母；见输出格式。“compile print”命令接受与“print”命令相同的选项；见打印选项。

compile print [[选项] --] ¶
compile print [[选项] --] /f

或者，您可以将表达式（生成它的源代码）作为多行文本输入。要进入此模式，请在命令后不跟任何文本调用“compile print”命令。这将启动多行编辑器。

可以使用以下命令检查编译和注入代码的过程：

set debug compile ¶

打开或关闭显示GDB编译和注入代码的过程。默认是关闭。

show debug compile

显示显示GDB编译和注入代码过程的当前状态。

set debug compile-cplus-types ¶

打开或关闭显示C++类型转换调试信息。默认是关闭。

show debug compile-cplus-types

显示显示C++类型转换调试信息的当前状态。

17.7.1 编译命令的编译选项 ¶

GDB需要为要注入的代码指定正确的编译选项，部分是为了使其ABI与进程兼容，部分是为了使注入的代码与GDB的注入过程兼容。

使用的选项，按优先级递增：

目标架构和操作系统选项（gdbarch）

这些选项取决于目标处理器类型和目标操作系统，通常它们至少指定32位（-m32）或64位（-m64）编译选项。

记录在目标中的编译选项

GCC（自4.7版起）根据GCC选项“-grecord-gcc-switches”将用于编译的选项存储到DWARF调试信息的“DW_AT_producer”部分。否则GCC不会生成DWARF。此功能仅与默认情况下“-g”生成DWARF的平台相关，否则可以尝试使用“-gdwarf-4”强制生成DWARF。

由“set compile-args”设置的编译选项

您可以使用以下命令覆盖编译选项：

set compile-args ¶

设置用于使用“compile”命令编译和注入代码的编译选项。这些选项会覆盖目标架构和/或在进程编译期间存储的选项中任何冲突的选项。

show compile-args

显示编译选项覆盖的当前状态，但这不会显示编译期间实际使用的所有选项，为此请使用set debug compile。

17.7.2 使用编译命令时的注意事项 ¶

使用“compile”命令时需要注意一些事项。由于每种语言的注意事项不同，下面的表格按语言突出显示了特定问题。

C代码示例和注意事项

当GDB中的语言设置为“C”时，编译器将尝试使用“C”编译器编译源代码。提供给“compile”命令的源代码对变量和类型的访问权限与它作为当前正在GDB中调试的程序的一部分时通常的访问权限大致相同。

下面是一个示例程序，它是后续示例的基础。这个程序已经编译并加载到GDB中，就像任何其他正常的调试会话一样。 

void function1 (void)
{
   int i = 42;
   printf ("function 1\n");
}

void function2 (void)
{
   int j = 12; 
   function1 ();
}

int main(void)
{
   int k = 6;
   int *p;
   function2 ();
   return 0;
}

为了本节示例的目的，上面的程序已经编译、加载到GDB中，停在函数“main”处，并且GDB正在等待用户输入。

要在GDB中访问任何程序的变量和类型，该程序必须使用调试信息进行编译和打包。“compile”命令也不例外。没有调试信息，您仍然可以使用“compile”命令，但您可以访问的变量和类型将非常有限。

因此，考虑到这一点，上面的示例已经启用调试信息进行编译。“compile”命令将可以访问所有变量和类型（除了那些可能已被优化掉的）。目前，由于GDB已将程序停在“main”函数中，“compile”命令将可以访问变量“k”。您可以调用“compile”命令并输入一些源代码来设置“k”的值。您也可以读取它，或者对该变量进行您在“C”中通常会进行的任何操作。请注意，在“compile”命令中对进程变量的更改是持久的。在以下示例中： 

compile code k = 3;

变量“k”现在为3。它将保留该值，直到示例程序中的其他内容更改它，或者另一个“compile”命令更改它。

正常的作用域和访问规则适用于由“compile”命令编译和注入的源代码。在示例中，变量“j”和“k”尚不可访问，因为程序当前停在“main”函数中，这些变量不在该函数的作用域内。因此，以下命令： 

compile code j = 3;

将导致编译错误消息产生。

一旦程序继续执行，这些变量将进入作用域并变得可访问；然后您通过“compile”命令指定的代码将能够访问它们。

您可以使用“compile”命令在源代码中创建变量和类型。作为“ compile”命令一部分创建的变量和类型在程序运行期间对程序的其余部分不可见。以下示例有效： 

compile code int ff = 5; printf ("ff is %d\n", ff);

但是，如果在该命令完成后您在GDB中输入以下内容： 

compile code printf ("ff is %d\n'', ff);

将引发编译器错误，因为变量“ff”不再存在。由“compile”命令生成并注入的目标代码在执行结束时会被删除。将提交给“compile”命令的代码创建的变量值赋给程序变量时要谨慎。以下示例有效： 

compile code int ff = 5; k = ff;

变量“ff”的值被赋给“k”。变量“k”不需要“ff”的存在来维持它被赋予的值。然而，在赋值时指针需要特别小心。如果用“compile”命令编译的源代码更改了示例程序中指针的地址，可能指向在“compile”命令中创建的变量，那么当命令退出时，该指针将指向无效位置。以下示例可能会给您调试的程序带来问题： 

compile code int ff = 5; p = &ff;

在这个示例中，当“compile”命令执行提供给它的源代码时，“p”将指向“ff”。然而由于（示例）程序中的变量会保留其赋值，当命令退出时，变量“p”将指向无效位置。应该遵循一个一般规则，即您应该要么将任何赋值的指针赋为“NULL”，要么在命令退出前将指针恢复到有效位置。

对于在“compile”命令中定义的任何结构体、联合体和typedef，也必须同样小心。在“compile”命令中定义的类型在下一个“compile”命令中将不再可用。因此，如果您将一个变量转换为在“compile”命令中定义的类型，必须小心确保任何未来解析该类型的需求都能实现。 

(gdb) compile code static struct a { int a; } v = { 42 }; argv = &v;
(gdb) compile code printf ("%d\n", ((struct a *) argv)->a);
gdb命令行:1:36: 错误: 解引用指向不完整类型‘struct a’的指针
编译失败。
(gdb) compile code struct a { int a; }; printf ("%d\n", ((struct a *) argv)->a);
42

被编译器优化掉的变量对于提交给“compile”命令的代码是不可访问的。访问这些变量将生成编译器错误，GDB将把这些错误打印到控制台。

17.7.3 编译命令的编译器搜索 ¶

GDB需要找到用于调试的进程的GCC，对于与运行GDB的架构不同的远程目标，这可能不明显。会在GDB主机上的环境变量“PATH”中搜索与目标架构和操作系统匹配的GCC二进制文件。此搜索可以通过下面的“set compile-gcc GDB命令”覆盖。“PATH”取自执行GDB的 shell，而不是由“set environment GDB命令”设置的值。见程序的环境。

具体来说，根据正在调试的目标进程，会在“PATH”中搜索与正则表达式“arch(-[^-]*)?-os-gcc”匹配的二进制文件。“arch”是处理器名称，支持多架构，例如，“i386”和“x86_64”目标都查找模式“(x86_64|i.86)”，“s390”和“s390x”目标都查找模式“s390x?”。“os”目前仅支持模式“linux(-gnu)?”。

在Posix主机上，GDB还需要找到编译器的共享库“libcc1.so"。它在默认共享库搜索路径中搜索（可通过常用环境变量“LD_LIBRARY_PATH”覆盖），与“PATH”或“set compile-gcc”设置无关。与此相反，“libcc1plugin.so"是根据找到的编译器的安装来找到的，可能由“set compile-gcc”命令指定。

set compile-gcc ¶

设置用于使用“compile”命令编译和注入代码的编译命令。如果未设置此选项（它被设置为空字符串），将进行上述搜索，这是默认设置。

show compile-gcc

显示当前编译命令GCC驱动文件名。如果已设置，它是主命令“gcc”，通常例如在“x - 86_64-linux - gnu - gcc”名称下找到。
