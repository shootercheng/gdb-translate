21.1.4.1 对没有调试符号的DLL的支持 ¶

在Windows上，你的程序所依赖的一些DLL通常不包含符号调试信息（例如，kernel32.dll）。当GDB在一个DLL中没有识别到任何调试符号时，它会依赖于DLL导出表中包含的最少符号信息。本节描述如何使用这些符号，GDB内部将其称为“最小符号”。

请注意，在被调试程序开始执行之前，不会加载任何DLL。解决这个问题最简单的方法就是启动程序 —— 可以通过设置断点或者让程序运行一次直到完成。

21.1.4.**2 DLL名称前缀 ¶

为了符合Microsoft调试工具使用的命名约定，DLL导出符号会带有一个基于DLL名称的前缀，例如 KERNEL32!CreateFileA。纯名称也会被放入符号表中，所以 CreateFileA 通常就足够了。在某些情况下，程序中可能会存在名称冲突（特别是如果可执行文件本身包含完整的调试符号），这时在引用DLL的内容时就需要使用完全限定名称。在名称周围使用单引号以避免感叹号（“!”）被解释为语言运算符。

请注意，DLL的内部名称可能全是大写，即使DLL的文件名是小写，反之亦然。由于GDB中的符号是区分大小写的，这可能会导致一些混淆。如果有疑问，可以尝试使用 info functions 和 info variables 命令，甚至 maint print msymbols（请参阅 检查符号表）。这里有一个例子： 

(gdb) info function CreateFileA
所有与正则表达式 "CreateFileA" 匹配的函数：

非调试符号：
0x77e885**4  CreateFileA
0x77e885**4  KERNEL32!CreateFileA

(gdb) info function !
所有与正则表达式 "!" 匹配的函数：

非调试符号：
0x6100114c  cygwin1!__assert
0x61004034  cygwin1!_dll_crt0@0
0x610042**0  cygwin1!dll_crt0(per_process *)
[等等...]

21.1.4.**第三部分 使用最小符号 ¶

从DLL导出表中提取的符号不包含太多类型信息。GDB所能做的只是根据包含该符号的链接器段来猜测一个符号是指函数还是变量。还要注意，除非程序正在运行，否则DLL中包含的内存的实际内容是不可用的。这意味着在没有运行程序的情况下，你无法检查变量的内容或反汇编DLL中的函数。

变量通常被视为指针并自动解引用。因此，通常需要在变量名前加上取地址运算符（“&”），并在命令中提供显式的类型信息。这里有一个问题类型的例子： 

(gdb) print 'cygwin1!__argv'
'cygwin1!__argv' 具有未知类型；将其转换为其声明的类型

(gdb) x 'cygwin1!__argv'
'cygwin1!__argv' 具有未知类型；将其转换为其声明的类型

以及两种可能的解决方案： 

(gdb) print ((char **)'cygwin1!__argv')[0]
$2 = 0x22fd98 "/cygdrive/c/mydirectory/myprogram"

(gdb) x/2x &'cygwin1!__argv'
0x610c0aa8 <cygwin1!__argv>:    0x10021608      0x00000000
(gdb) x/x 0x10021608
0x10021608:     0x0022fd98
(gdb) x/s 0x0022fd98
0x22fd98:        "/cygdrive/c/mydirectory/myprogram"

即使在程序开始执行之前，也可以在DLL内设置断点。然而，在这种情况下，GDB无法检查函数的初始指令以跳过函数的帧设置代码。你可以通过使用“*&”在原始内存地址设置断点来解决这个问题： 

(gdb) break *&'python22!PyOS_Readline'
断点1位于0x1e04eff0

这些扩展的作者并不完全相信在像 kernel32.dll 这样的共享DLL中设置断点是完全安全的安全。
