15.4.10.11 Ada模式的已知特性 ¶

除了前面列出的遗漏之处（见 Ada的遗漏），我们还知道GDB中Ada模式存在一些问题和限制，其中一些将在调试器和GNU Ada编译器的未来计划版本中得到修复。

  • 编译器选择不在存储中作为对象实现的静态常量对调试器不可见。
  • 函数参数列表中的命名参数关联被忽略（参数列表被视为按位置排列）。
  • 许多有用的库包目前对调试器不可见。
  • 定点算术、转换、输入和输出使用浮点算术进行，可能会给出仅近似主机机器上结果的结果。
  • GNAT编译器从不为Ada语言定义的任何标准符号生成前缀 Standard。GDB知道这一点：当你使用它时，它会从名称中去除前缀，并且永远不会在局部符号中查找你如此限定的名称，也不会与其他包或子程序中的符号进行匹配。如果你在程序中的任何地方定义了除参数和局部变量之外的实体，其简单名称与 Standard 中的名称匹配，GNAT在此处缺乏限定可能会导致混淆。当这种情况发生时，可以通过显式地用包 Standard 限定有问题的名称来解决混淆。

较旧版本的编译器有时会生成错误的调试信息，导致调试器错误地打印受影响实体的值。在某些情况下，调试器能够自动解决问题。在其他情况下，调试器能够解决问题，但必须专门启用解决方法。

set ada trust-PAD-over-XVS on

配置GDB在计算Ada实体的值时严格遵循GNAT编码，特别是当涉及 PAD 和 PAD___XVS 类型时（有关GNAT编译器使用的编码的完整描述，请参阅GCC源中的 ada/exp_dbug.ads

set ada trust-PAD-over-XVS off

这与GNAT编译器使用的编码有关。如果GDB有时为某些实体打印错误的值，将 ada trust-PAD-over-XVS 更改为 off 会激活一种解决方法，可能会修复问题。将 ada trust-PAD-over-XVS 设置为 off 始终是安全的，但这会带来轻微的性能损失，因此建议除非必要，否则将此设置保留为 on。

在内部，调试器还依赖于编译器遵循一些称为“GNAT编码”的约定，所有这些都记录在GCC源中的 gcc/ada/exp_dbug.ads 中。此编码描述了应为某些类型生成调试信息的方式。 特别是，此约定使用了 描述性类型，这些是纯粹为帮助调试器而生成的人工类型。

这些编码是在当时使用的调试信息格式不够强大，无法描述Ada中可用的一些更复杂类型的时候定义的。由于DWARF允许我们表达几乎所有Ada特性，长期目标是慢慢地用它们的纯DWARF等效物替换这些描述性类型。为了促进这种过渡，有一个新的维护选项可用于强制调试器忽略那些描述性类型。它允许用户快速评估没有它们时GDB的工作情况。

maintenance ada set ignore-descriptive-types [on|off] ¶

控制调试器是否应忽略描述性类型。默认值是不忽略描述性类型（off）。

maintenance ada show ignore-descriptive-types ¶

显示GDB是否忽略描述性类型。
