18.1 指定文件的命令 ¶

你可能需要指定可执行文件和核心转储文件名。通常的做法是在启动时，使用GDB启动命令的参数（见进入和退出GDB）。

偶尔，在GDB会话期间需要切换到不同的文件。或者你运行GDB时忘记指定想要使用的文件。又或者你正在通过gdbserver调试远程目标（见使用gdbserver程序）。在这些情况下，GDB中用于指定新文件的命令就很有用了。

file filename ¶

使用filename作为要调试的程序。读取它的符号和纯内存内容。它也是你使用run命令时执行的程序。如果你没有指定目录，并且在GDB工作目录中没有找到该文件，GDB会使用环境变量PATH作为搜索目录列表，就像 shell 在寻找要运行的程序时那样。你可以使用path命令更改此变量的值，这对GDB和你的程序都有效。

filename参数支持转义字符和引号，见作为命令参数的文件名。

你可以使用file命令将未链接的目标.o文件加载到GDB中。你将无法 “运行” 目标文件，但可以反汇编函数并检查变量。此外，如果底层的BFD功能支持，你可以使用gdb -write通过此技术修补目标文件。请注意，在这种情况下，GDB既不能解释也不能修改重定位，因此分支和一些初始化变量可能会指向错误的位置。但这个功能偶尔仍然很方便。

file

不带参数的file命令会使GDB丢弃它所拥有的关于可执行文件和符号表的任何信息。

exec-file [ filename ] ¶

指定要运行的程序（但不是符号表）在filename中。如有必要，GDB会搜索环境变量PATH来定位你的程序。省略filename意味着丢弃关于可执行文件的信息。

filename参数支持转义字符和引号，见作为命令参数的文件名。

symbol-file [ filename [ -o offset ]] ¶

从文件filename中读取符号表信息。必要时会搜索PATH。使用file命令可以从同一个文件中获取符号表和要运行的程序。

如果指定了可选的offset，它会被添加到符号文件中每个节的起始地址。这在程序在运行时被重定位的情况下很有用，例如启用了kASLR的Linux内核。

不带参数的symbol-file命令会清除GDB中关于你程序符号表的信息。

symbol-file命令会使GDB忘记一些断点和自动显示表达式的内容。这是因为它们可能包含指向记录符号和数据类型的内部数据的指针，而这些是GDB内部正在丢弃的旧符号表数据的一部分。

执行一次symbol-file命令后，再次按下RET它不会重复执行。

filename参数支持转义字符和引号，见作为命令参数的文件名。

当GDB针对特定环境进行配置时，它会理解为该环境生成的标准格式的调试信息；你可以使用GNU编译器，或其他遵循本地约定的编译器。 通常使用GNU编译器能获得最佳结果；例如，使用GCC你可以为优化后的代码生成调试信息。

对于大多数类型的目标文件，除了使用COFF的旧SVR3系统外，symbol-file命令通常不会立即读取完整的符号表。相反，它会快速扫描符号表以查找存在哪些源文件和符号。细节会在稍后根据需要逐个源文件读取。

这种两阶段读取策略的目的是使GDB启动更快。在大多数情况下，除了在读取特定源文件的符号表细节时偶尔会有停顿外，它是不可见的。（如果需要，set verbose命令可以将这些停顿转换为消息。见可选的警告和消息。）

我们尚未为COFF实现两阶段策略。当符号表以COFF格式存储时，symbol-file会立即读取完整的符号表数据。请注意，“COFF中的stabs” 仍然采用两阶段策略，因为调试信息实际上是stabs格式。

symbol-file [ -readnow ] filename ¶
file [ -readnow ] filename

如果你想确保GDB拥有完整的符号表，可以使用任何加载符号表信息命令的 “-readnow” 选项来覆盖GDB读取符号表的两阶段策略。

symbol-file [ -readnever ] filename ¶
file [ -readnever ] filename

你可以使用 “-readnever” 选项指示GDB永远不读取filename中包含的符号信息。见--readnever。

core-file [filename] ¶
core

指定用作 “内存内容” 的核心转储文件的位置。传统上，核心文件只包含生成它们的进程的地址空间的某些部分；GDB可以访问可执行文件本身以获取其他部分。

不带参数的core-file命令指定不使用核心文件。

请注意，当你的程序在GDB下实际运行时，核心文件会被忽略。因此，如果你的程序一直在运行，而你希望调试核心文件，你必须杀死程序正在运行的子进程。为此，使用kill命令（见杀死子进程）。

add-symbol-file filename [ -readnow | -readnever ] [ -o offset ] [ textaddress ] [ -s section address … ] ¶

add-symbol-file命令从文件filename中读取额外的符号表信息。当filename已通过其他方式动态加载到正在运行得程序中时，你会使用此命令。textaddress参数给出文件的文本段已加载到得内存地址。你可以使用任意数量的 “-s section address” 对来指定其他节的基地址。如果省略一个节，GDB将使用在filename中找到的默认地址。任何address或textaddress都可以作为表达式给出。

如果指定了可选的offset，它会被添加到每个节的起始地址，但那些已明确指定地址的节除外。

文件filename的符号表会被添加到最初使用symbol-file命令读取的符号表中。你可以多次使用add-symbol-file命令；这样读取的新符号数据会与旧数据一起保留。

filename参数支持转义字符和引号，见作为命令参数的文件名。

可以使用命令remove-symbol-file撤销更改。

虽然filename通常是共享库文件、可执行文件或其他已完全重定位以加载到进程中的目标文件，但只要满足以下条件，你也可以从可重定位的.o文件加载符号信息：

  • 文件的符号信息仅引用该文件中定义的链接器符号，而不引用其他目标文件定义的符号，
  • 文件的符号信息引用的每个节实际上都已按文件中出现的方式加载到下级进程中，
  • 你可以确定每个节加载到的地址，并将这些地址提供给add-symbol-file命令。

一些嵌入式操作系统，如Sun Chorus和VxWorks，可以将可重定位文件加载到正在运行的程序中；这样的系统通常很容易满足上述要求。然而，重要的是要认识到，许多原生系统使用复杂的链接过程（例如.linkonce节分解和C++构造函数表汇编），这使得满足这些要求变得困难。一般来说，不能假设使用add-symbol-file读取可重定位目标文件的符号信息会与以正常方式将可重定位目标文件链接到程序中具有相同的效果。

使用add-symbol-file后按下RET它不会重复执行。

remove-symbol-file filename ¶
remove-symbol-file -a address

删除通过add-symbol-file命令添加的符号文件。要删除的文件可以通过其filename或内存中该符号文件边界内的address来标识。例如： 

(gdb) add-symbol-file /home/user/gdb/mylib.so 0x7ffff7ff9480
add symbol table from file "/home/user/gdb/mylib.so" at
   .text_addr = 0x7ffff7ff9480
(y or n) y
Reading symbols from /home/user/gdb/mylib.so...
(gdb) remove-symbol-file -a 0x7ffff7ff9480
Remove symbol table from file "/home/user/gdb/mylib.so"? (y or n) y
(gdb)

address可以是任何求值为地址的表达式。

使用remove-symbol-file后按下RET它不会重复执行。

filename参数支持转义字符和引号，见作为命令参数的文件名。
