E.10通知数据包 ¶

GDB远程串行协议包括通知，即不需要确认的数据包。GDB和存根程序都可以发送通知（尽管目前定义的唯一通知是由存根程序发送的）。通知携带信息，而不会产生确认的往返延迟，因此对于偶尔丢包不是问题的低影响通信很有用。

通知数据包的形式为‘% 数据 # 校验和’，其中数据是通知的内容，校验和是数据的校验和，其计算和格式化方式与普通GDB数据包相同。通知的数据中从不包含‘$’、‘%’或‘#’字符。收到通知后，接收方不会发送‘+’或‘-’来确认通知的接收或报告其损坏情况。

每个通知的数据都以一个名称开头，该名称不包含冒号字符，后面跟着一个冒号字符。

接收方应默默地忽略损坏的通知和他们不理解的通知。接收方在收到格式正确的通知时，无论是否理解，都应重新启动超时周期。

发送方应仅在本协议描述指定允许时才发送此处描述的通知。将来，我们可能会扩展协议，以允许在新的上下文中使用现有的通知；此规则有助于旧的发送方避免使新的接收方感到困惑。

（旧版本的GDB会忽略接收到的字节，直到看到普通数据包开始的‘$’字节，因此新的存根程序可以发送通知而不用担心会使旧的客户端感到困惑。目前没有定义GDB要发送的通知，但我们假设大多数旧的存根程序也会忽略它们。）

每个通知由三部分组成：

‘名称:事件’

通知数据包由发起交换的一方（目前只有存根程序这样做）发送，其中事件携带有关通知的特定信息，名称指定通知的名称。

‘确认’

另一方（通常是GDB）发送的确认，以确认交换并请求事件。

异步通知机制的目的是向GDB报告远程存根程序中发生的有趣事情。

远程存根程序可以随时发送通知名称:事件，但GDB会在适当的时候确认该通知。在GDB确认之前，通知事件处于待处理状态。一次只能有一个通知处于待处理状态；如果在GDB确认之前发生了其他事件，存根程序必须将它们排队，以便稍后响应来自GDB的确认数据包进行同步传输。由于通知机制不可靠，如果存根程序认为GDB可能没有收到通知，它可以重新发送通知。

具体来说，当GDB没有从存根程序读取其他输入时，或者当GDB期望读取正常的同步响应或对其发送的数据包的‘+’/‘-’确认时，可能会出现通知。通知数据包与存根程序的任何其他通信都不同，因此不存在歧义。

收到通知后，GDB应通过向存根程序发送一个确认数据包作为常规的同步请求来确认它。这种确认不需要立即发生，因为GDB可以先向存根程序发送其他无关的数据包，存根程序应正常处理这些数据包。

收到确认数据包后，如果存根程序有其他排队的事件要报告给GDB，它应通过发送一个正常的事件来响应。然后GDB应发送另一个确认数据包以请求进一步的响应；同样，它也可以发送其他无关的数据包，存根程序应正常处理这些数据包。

如果存根程序收到一个确认数据包且没有其他事件要报告，存根程序应返回一个‘OK’响应。此时，GDB已完成处理一个通知，存根程序也已完成发送任何排队的事件。在收到最终的‘OK’之前，GDB不会接受任何新的通知。如果发生进一步的通知事件，存根程序应发送一个新的通知，GDB应接受该通知，并重复该过程。

异步通知的过程可以通过以下示例来说明： 

<- %Stop:T0505:98e7ffbf;04:4ce6ffbf;08:b1b6e54c;thread:p7526.7526;core:0;
...
-> vStopped
<- T0505:68f37db7;04:40f37db7;08:63850408;thread:p7526.7528;core:0;
-> vStopped
<- T0505:68e3fdb6;04:40e3fdb6;08:63850408;thread:p7526.7529;core:0;
-> vStopped
<- OK

定义了以下通知：

通知确认事件描述
停止vStopped回复。回复具有停止回复的形式，如在 停止回复数据包 中所述。有关GDB如何确认这些通知的信息，请参阅 远程协议对非停止模式的支持。报告非停止模式下的异步停止事件。
