18.4特殊节中的调试信息 ¶

一些系统提供预构建的可执行文件和库，它们有一个特殊的‘.gnu_debugdata’节。这个特性被称为MiniDebugInfo。这个节包含一个LZMA压缩的对象，用于为回溯提供额外的符号。

这个节的目的是提供额外的最小调试信息，用于简单的回溯。它不是用于替代完整的单独调试信息（见单独文件中的调试信息）。下面的例子展示了预期的用法；然而，GDB目前对这个节中可能包含的调试信息类型没有限制。

GDB支持这个扩展。如果这个节存在，那么在找不到其他调试信息源且GDB配置了LZMA支持的情况下，就会使用它。

可以使用objcopy和其他标准实用工具轻松创建这个节： 

# 从主二进制文件中提取动态符号，不需要在正常符号表中也有这些。
nm -D binary --format=posix --defined-only \
  | awk '{ print $1 }' | sort > dynsyms

# 从调试信息中提取所有文本（即函数）符号。
# （注意，实际上我们也接受“D”符号，这是为了像PowerPC64这样使用函数描述符的平台。）
nm binary --format=posix --defined-only \
  | awk '{ if ($2 == "T" || $2 == "t" || $2 == "D") print $1 }' \
  | sort > funcsyms

# 保留所有不在动态符号表中的函数符号。
comm -13 dynsyms funcsyms > keep_symbols

# 将完整的调试信息分离到调试二进制文件中。
objcopy --only-keep-debug binary debug

# 复制完整的调试信息，只保留一组最小的符号并删除一些不必要的节。
objcopy -S --remove-section.gdb_index --remove-section.comment \
  --keep-symbols=keep_symbols debug mini_debuginfo

# 从原始二进制文件中删除完整的调试信息。
strip --strip-all -R.comment binary

# 将压缩数据注入到原始二进制文件的.gnu_debugdata节中。
xz mini_debuginfo
objcopy --add-section.gnu_debugdata=mini_debuginfo.xz binary
