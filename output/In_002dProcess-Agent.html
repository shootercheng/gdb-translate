31 进程内代理 ¶

传统的调试模型从概念上讲速度较慢，但运行良好，因为大多数错误可以在调试模式执行中重现。然而，随着多核或众核处理器成为主流，多线程程序越来越流行，会有越来越多的错误只在正常模式执行时才会出现，例如线程竞争，因为调试器对程序计时的干扰可能会掩盖这些错误。另一方面，在某些应用中，调试器长时间中断程序执行以便开发人员了解其行为的任何有用信息是不可行的。如果程序的正确性取决于其实时行为，调试器引入的延迟可能会导致程序失败，即使代码本身是正确的。能够在不中断程序的情况下观察其行为是很有用的。

因此，传统的调试模型过于侵入性，无法重现某些错误。为了减少对程序的干扰，我们可以减少调试器执行的操作数量。进程内代理是一个共享库，与被调试程序在同一进程中运行，并且能够自行执行一些调试操作。结果，只有在必要时调试器才会参与，从而可以相应地提高调试性能。请注意，对程序的干扰可以减少，但不能完全消除，因为进程内代理仍然会停止或减慢程序的速度。

进程内代理在执行调试操作时可以解释并执行代理表达式（见 GDB代理表达式机制）。代理表达式可用于不同目的，例如在跟踪点收集数据以及在断点处进行条件评估。

你可以使用以下命令控制是否使用进程内代理辅助调试：

set agent on ¶

使进程内代理代表调试器执行一些操作。用户请求的哪些操作将由进程内代理完成取决于其功能。例如，如果您请求在进程内代理中评估断点条件，并且进程内代理也具有此功能，那么断点条件将在进程内代理中进行评估。

set agent off ¶

禁用进程内代理执行调试操作。所有操作将由GDB执行。

show agent ¶

显示进程内代理执行调试操作的当前设置。
