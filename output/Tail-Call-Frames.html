11.2 尾调用帧 ¶

函数B可以在其最后一条语句中调用函数C。在未优化的编译中，对C的调用紧接着在B代码末尾的返回指令。优化编译器可能会将函数B中的调用和返回替换为直接跳转到函数C。这种跳转指令的使用被称为尾调用。

在函数C执行期间，函数调用栈帧中不会有任何迹象表明它是从B进行尾调用的。如果函数A经常调用函数B，而函数B又尾调用函数C，那么GDB会将A视为C的调用者。然而，在某些情况下，GDB可以确定C是从B进行尾调用的，并会为此创建一个虚拟调用帧，其返回地址的设置就好像B正常调用C一样。

目前只有DWARF 2调试格式支持此功能，并且编译器必须生成“DW_TAG_call_site”标签。使用GCC时，你需要在编译期间指定-O -g以获取此信息。

info frame命令（见关于一个帧的信息）会通过“tail call frame”文本指示尾调用帧类型，如下例GDB输出所示： 

(gdb) x/i $pc - 2
   0x40066b <b(int, double)+11>: jmp 0x400640 <c(int, double)>
(gdb) info frame
栈层1，帧位于0x7fffffffda30：
 rip = 0x40066d在b (amd64-entry-value.cc:59)；保存的rip为0x4004c5
 尾调用帧，0x7fffffffda30处帧的调用者
 源语言c++。
 未知地址的参数列表。
 未知地址的局部变量，前一帧的sp为0x7fffffffda30

检测所有可能的代码路径执行可能会发现它们不明确。没有存储执行历史（可能永远不会为此目的使用反向执行程序），并且最后已知的调用者可能通过多个不同的跳转序列到达已知的被调用者。在这种情况下，GDB仍然会尝试显示至少所有明确的顶级尾调用者和所有明确的底层尾被调用者（如果有的话）。

set debug entry-values ¶

设置为on时，启用在函数入口和尾调用时打印帧参数值的分析消息。它将显示它考虑的所有可能有效的尾调用代码路径。它还将打印它们与最终明确的（可能是部分甚至为空的）代码路径结果的交集。

show debug entry-values ¶

显示在函数入口和尾调用时打印帧参数值的分析消息的当前状态。

例如，尾调用的分析消息可以显示为什么未识别函数c的虚拟尾调用帧（由于变量x的间接引用）： 

static void __attribute__((noinline, noclone)) c (void);
void (*x) (void) = c;
static void __attribute__((noinline, noclone)) a (void) { x++; }
static void __attribute__((noinline, noclone)) c (void) { a (); }
int main (void) { x (); return 0; }

另一种可能性是不明确的虚拟尾调用帧解析： 

int i;
static void __attribute__((noinline, noclone)) f (void) { i++; }
static void __attribute__((noinline, noclone)) e (void) { f (); }
static void __attribute__((noinline, noclone)) d (void) { f (); }
static void __attribute__((noinline, noclone)) c (void) { d (); }
static void __attribute__((noinline, noclone)) b (void)
{ if (i) c (); else e (); }
static void __attribute__((noinline, noclone)) a (void) { b (); }
int main (void) { a (); return 0; }
