5.2继续和单步执行 ¶

继续意味着恢复程序执行，直到程序正常完成。相比之下，单步执行意味着只再执行程序的一个“步骤”，其中“步骤”可能意味着一行源代码，或者一条机器指令（取决于你使用的具体命令）。无论是继续执行还是单步执行，由于断点或信号，你的程序可能会更早停止。（如果它因信号而停止，你可能想使用handle，或者使用‘signal 0’来恢复执行（见 信号），或者你可以单步进入信号的处理程序（见 单步执行和信号处理程序）。）

continue [忽略计数] ¶
c [忽略计数]
fg [忽略计数]

在程序上次停止的地址恢复程序执行；在该地址设置的任何断点都将被绕过。可选参数忽略计数允许你指定在此位置进一步忽略断点的次数；其效果类似于ignore（见 断点条件）。

参数忽略计数仅在程序因断点而停止时才有意义。在其他时候，continue的参数将被忽略。

同义词c和fg（表示前台，因为被调试的程序被视为前台程序）纯粹是为了方便而提供的，并且与continue具有完全相同的行为。

要在不同的位置恢复执行，你可以使用return（见 从函数返回）回到调用函数；或者使用jump（见 在不同地址继续）跳转到程序中的任意位置。

使用单步执行的一种典型技术是在你认为存在问题的函数或程序部分的开头设置一个断点（见 断点、观察点和捕获点），运行你的程序直到它在该断点处停止，然后单步执行可疑区域，检查感兴趣的变量，直到你看到问题出现。

step ¶

继续运行你的程序，直到控制权到达不同的源代码行，然后停止并将控制权返回给GDB。此命令缩写为s。

警告：如果你在控制位于没有调试信息的函数内时使用step命令，执行将继续，直到控制权到达有调试信息的函数。同样，它不会单步进入没有调试信息的函数。要单步执行没有调试信息的函数，请使用下面描述的stepi命令。

step命令仅在源代码行的第一条指令处停止。这可以防止在switch语句、for循环等中可能发生的多次停止。如果在该行内调用了有调试信息的函数，step将继续停止。换句话说，step会单步进入该行内调用的任何函数。

此外，step命令仅在函数有行号信息时才进入该函数。否则，它的行为类似于next命令。这避免了在MIPS机器上使用cc -gl时出现的问题。以前，如果关于例程有任何调试信息，step会进入子例程。

step 计数

像在step中一样继续运行，但执行计数次。如果在计数步之前到达断点，或者发生了与单步执行无关的信号，单步执行将立即停止。

next [计数] ¶

继续到当前（最内层）堆栈帧中的下一行源代码。这与step类似，但代码行内出现的函数调用将不停顿地执行。当控制权到达你发出next命令时正在执行的原始堆栈级别的不同代码行时，执行停止。此命令缩写为n。

参数计数是一个重复计数，与step中的相同。

next命令仅在源代码行的第一条指令处停止。这可以防止在switch语句、 for循环等中可能发生的多次停止。

set step-mode ¶
set step-mode on

set step-mode on命令使step命令在函数的第一条指令处停止，该函数不包含调试行信息，而不是跳过它。

在你可能有兴趣检查没有符号信息的函数的机器指令，并且不想让GDB自动跳过该函数的情况下，这很有用。

set step-mode off

使step命令跳过任何不包含调试信息的函数。这是默认设置。

show step-mode

显示GDB是否会在没有源代码行调试信息的函数中停止或跳过。

finish ¶

继续运行，直到选定堆栈帧中的函数返回后立即停止。打印返回值（如果有）。此命令可以缩写为fin。

将此与return命令（见 从函数返回）进行对比。

set print finish [on|off] ¶
show print finish

默认情况下，finish命令会显示函数返回的值。可以使用set print finish off禁用此功能。禁用后，该值仍会输入到值历史记录中（见 值历史记录），但不会显示。

until ¶
u

继续运行，直到到达当前堆栈帧中当前行之后的一行源代码。此命令用于避免在循环中多次单步执行。它类似于next命令，不同之处在于当until遇到跳转时，它会自动继续执行直到程序计数器大于跳转的地址。

这意味着当你单步执行完一个循环到达末尾时，until会使你的程序继续执行直到它退出循环。相比之下，在循环末尾使用next命令只会单步回到循环的开头，这迫使你单步执行下一次迭代。

如果程序试图退出当前堆栈帧，until总是会停止你的程序。

如果机器代码的顺序与源代码行的顺序不匹配，until可能会产生一些违反直觉的结果。例如，在以下调试会话的摘录中，f（frame）命令显示执行在第206行停止；然而当我们使用until时，我们到达了第195行： 

(gdb) f
#0  main (argc=4, argv=0xf7fffae8) at m4.c:206
206                 expand_input();
(gdb) until  
195             for ( ; argc > 0; NEXTARG) {

发生这种情况是因为，为了提高执行效率，编译器在循环末尾而不是开头生成了循环结束测试的代码——尽管在C for循环中测试是写在循环体之前的。当until命令推进到这个表达式时，它似乎单步回到了循环的开头；然而，它实际上并没有回到更早的语句——就实际的机器代码而言并非如此。

不带参数的until通过单条指令单步执行，因此比带参数的until慢。

until 位置规范
u 位置规范

继续运行你的程序，直到它到达由解析位置规范得到的代码位置，或者当前堆栈帧返回。位置规范是 位置规范 中描述的任何一种形式。此命令形式使用临时断点，因此比不带参数的until更快。仅当指定的位置在当前帧中时，才会实际到达该位置。这意味着until可用于跳过递归函数调用。例如，在下面的代码中，如果当前位置是第96行，发出until 99将在同一个阶乘调用中执行程序直到第99行，即在内部调用返回之后。 

94	int factorial (int value)
95	{
96	    if (value > 1) {
97            value *= factorial (value - 1);
98	    }
99	    return (value);
100     }
advance 位置规范 ¶

继续运行你的程序直到它到达由解析位置规范得到的代码位置，或者当前堆栈帧返回。位置规范是 位置规范 中描述的任何一种形式。此命令类似于until，但advance不会跳过递归函数调用，并且目标代码位置不必与当前位置在同一帧中。

stepi ¶
stepi 参数
si

执行一条机器指令，然后停止并返回调试器。

在按机器指令单步执行时，通常做‘display/i $pc’很有用。这会使GDB在你的程序每次停止时自动显示下一条要执行的指令。见 自动显示。

参数是一个重复计数，与step中的相同。

nexti ¶
nexti 参数
ni

执行一条机器指令，但如果它是一个函数调用，则继续执行直到函数返回。

参数是一个重复计数，与next中的相同。

默认情况下，如果可用，GDB会使用目标辅助的范围单步执行。换句话说，每当你使用单步执行命令（例如，step，next）时，GDB会告诉目标单步执行相应的指令地址范围，而不是发出多个单步。这加快了行单步执行的速度，特别是对于远程目标而言。理想情况下，你应该没有理由想要关闭范围单步执行。然而，调试信息中的错误、远程存根（对于远程目标）中的错误，甚至GDB中的错误，都可能导致在启用目标辅助范围单步执行时行单步执行行为不正确。如有必要，你可以使用以下命令关闭范围单步执行：

set range-stepping ¶
show range-stepping

控制是否启用范围单步执行。

如果为on，并且目标支持它，GDB会告诉目标自己单步执行一个地址范围，而不是发出多个单步。如果为off，GDB总是发出单步，即使目标支持范围单步执行。默认值为on。
