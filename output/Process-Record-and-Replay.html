7记录下级进程的执行并重放它 ¶

在某些平台上，GDB提供了一个特殊的进程记录和重放目标，它可以记录进程执行的日志，并在以后使用正向和反向执行命令进行重放。

当使用这个目标时，如果执行日志包含下一条指令的记录，GDB将在重放模式下进行调试。在重放模式下，下级进程不会真正执行代码指令。相反，代码执行期间通常发生的所有事件都从执行日志中获取。虽然在重放模式下代码不会真正执行，但寄存器（包括程序计数器寄存器）的值和下级进程的内存仍然会像正常情况一样改变。它们的内容从执行日志中获取。

如果下一条指令的记录不在执行日志中，GDB将在记录模式下进行调试。在这种模式下，下级进程正常执行，GDB记录执行日志以供将来重放。

进程记录和重放目标支持反向执行（见反向运行程序），即使下级进程运行的平台不支持。然而，在这种情况下，反向执行受到执行日志中记录的指令范围的限制。换句话说，在不直接支持反向执行的平台上，只能在重放模式下进行反向执行。

在反向调试时，只要执行日志包含上一条指令的记录，GDB将在重放模式下工作；否则，如果平台支持反向执行，它将在记录模式下工作，或者如果不支持则停止。

目前，ARM、Aarch64、龙芯架构、Moxie、PowerPC、PowerPC64、S/390、RISC-V和运行GNU/Linux的x86（i386/amd64）支持进程记录和重放。进程记录和重放可用于本地调试，也可用于通过gdbserver进行远程调试。

记录下级进程时，GDB可能会在单步执行命令和显示执行历史的命令期间打印辅助信息。

对于支持进程记录和重放的架构环境，GDB提供以下命令：

record 方法 ¶

此命令启动进程记录和重放目标，可以将记录方法指定为参数。如果没有参数，该命令使用full记录方法。可用的记录方法如下：

full

使用GDB的软件记录和重放实现进行完整记录/重放记录。此方法允许重放和反向执行。

btrace 格式

英特尔处理器支持的硬件指令记录，此方法不记录数据。此外，数据收集在环形缓冲区中，因此当缓冲区满时，旧数据将被覆盖。它允许有限的反向执行。在反向执行期间，变量和寄存器不可用。在远程调试中，断开连接时记录继续。重新连接后，可以检查记录的数据。可以使用record stop停止记录。

记录格式可以指定为参数。如果没有参数，命令会选择记录格式。可用的记录格式如下：

bts ¶

使用分支跟踪存储（BTS）记录格式。在这种格式中，处理器在btrace环形缓冲区中为每个执行的分支存储一个从/到记录．

pt ¶

使用英特尔处理器跟踪记录格式。在这种格式中，处理器以压缩形式存储执行跟踪，然后由GDB解码。

跟踪可以以非常低的开销进行记录。与BTS相比，压缩跟踪格式还允许小跟踪缓冲区已经包含大量指令。

另一方面．解码记录的执行跟踪比解码BTS跟踪更昂贵。这主要是由于要处理的指令数量增加。你应该小心增加缓冲区大小。

并非所有记录格式在所有处理器上都可用。

进程记录和重放目标只能调试已经在运行的进程。因此，你需要首先使用run或start命令启动进程，然后使用record 方法命令开始记录。

当启动进程记录和重放目标时，位移单步执行（见位移单步执行）将自动禁用。这是因为进程记录和重放目标不支持位移单步执行。

如果下级进程处于非停止模式（见非停止模式）或异步执行模式（见后台执行），并非所有记录方法都可用。full记录方法不支持这两种模式。

record stop ¶

停止进程记录和重放目标。当进程记录和重放目标停止时，整个执行日志将被删除，下级进程将被终止，或者保持其最终状态。

当你在记录模式下停止进程记录和重放目标时（在执行日志的末尾），下级进程将在接下来要记录的指令处停止。换句话说，如果您记录一段时间然后停止记录，下级进程将保持与未进行记录时相同状态。

另一方面，如果进程记录和重放目标在重放模式下停止（即，不在执行日志的末尾，而是在更早的某个点），下级进程将在该更早的状态变为“活动”，然后可以从该状态继续对进程进行通常的“活动”调试。

当下级进程退出，或者GDB与其分离时，进程记录和重放目标将自动停止自身。

record goto ¶

转到执行日志中的特定位置。有几种方法可以指定要转到的位置：

record goto begin
record goto start

转到执行日志的开头。

record goto end

转到执行日志的末尾。

record goto n

转到执行日志中的指令编号n。

record save 文件名 ¶

将执行日志保存到文件文件名。默认文件名是gdb_record.进程ID，其中进程ID是下级进程的进程ID。

此命令可能并非对所有记录方法都可用。

record restore 文件名 ¶

从文件文件名恢复执行日志。文件必须使用record save创建。

set record full insn-number-max 限制 ¶
set record full insn-number-max unlimited

设置full记录方法要记录的指令限制。默认值是200000。

如果限制是一个正数，那么当记录的指令数量大于限制时，GDB将开始从日志中删除指令。对于每一条新记录的指令，GDB将删除最早记录的指令，以将记录的指令数量保持在限制范围内。（由于删除记录的指令会丢失信息，GDB允许您通过下面描述的stop-at-limit选项来控制达到限制时的行为。）

如果限制是unlimited或零，GDB将永远不会从执行日志中删除记录的指令。记录的指令数量仅受可用内存的限制。

show record full insn-number-max ¶

显示使用full记录方法要记录的指令限制。

set record full stop-at-limit

控制full记录方法在记录的指令数量达到限制时的行为。如果为ON（默认），当第一次达到限制时，GDB将停止并询问您是否要停止下级进程或继续运行它并记录执行日志。如果您决定继续记录，每一条新记录的指令将导致最旧的指令被删除。

如果此选项为OFF，GDB将自动删除最旧的记录，为每一条新记录腾出空间，而不会询问。

show record full stop-at-limit

显示stop-at-limit的当前设置。

set record full memory-query

控制当GDB无法为full记录方法记录指令引起的内存变化时的行为。如果为ON，在这种情况下，GDB将询问是否停止下级进程。

如果此选项为OFF（默认），GDB将自动忽略此类指令对内存的影响。稍后，当GDB重放此执行日志时，它将把这条指令的日志标记为不可访问，并且不会影响重放结果。

show record full memory-query

显示memory-query的当前设置。

btrace记录目标不跟踪数据。为了方便起见，在重放时，GDB直接从正在运行的程序中读取只读内存，假设只读区域的地址不变。例如，这使得在重放时可以反汇编代码，但不能打印变量。

在某些情况下，能够检查变量可能会很有用。你可以使用以下命令：

set record btrace replay-memory-access

控制重放期间访问内存时btrace记录方法的行为。如果为read-only（默认），GDB将只允许访问只读内存。 如果为read-write，GDB将允许访问只读和读写内存。请注意，访问的内存对应于正在运行的目标，不一定对应于当前的重放位置。

set record btrace cpu 标识符

设置在解码跟踪时用于启用处理器勘误修复的处理器。

处理器勘误是由处理器设计或制造引起的处理器操作中的缺陷。它们可能导致跟踪与规范不匹配。反过来，这可能会导致跟踪解码失败。GDB可以检测错误的跟踪数据包并进行纠正，从而避免解码失败。这些纠正措施称为勘误修复，并根据记录跟踪的处理器启用。

默认情况下，GDB会尝试自动检测处理器，并为其应用必要的修复。但是，如果GDB尚未支持该处理器，您可能需要指定它。此命令允许您这样做，也允许禁用修复。

参数标识符标识CPU，格式为：供应商:处理器标识符。此外，还有两个特殊标识符，none和auto（默认）。

目前支持以下供应商标识符和相应的处理器标识符：

intel系列/型号[/步进]

在GNU/Linux系统上，可以从/proc/cpuinfo获取处理器系列、型号和步进。

如果标识符为auto，为记录跟踪的处理器启用勘误修复。如果标识符为none，则禁用勘误修复。

例如，在新系统上使用旧版本的GDB时，解码可能会失败，因为GDB不支持新处理器。指定一个GDB支持的旧处理器通常就足够了。

(gdb) info record
Active record target: record-btrace
Recording format: Intel Processor Trace.
Buffer size: 16kB.
Failed to configure the Intel Processor Trace decoder: unknown cpu.
(gdb) set record btrace cpu intel:6/158
(gdb)
show record btrace replay-memory-access ¶

显示replay-memory-access的当前设置。

show record btrace cpu

显示用于启用跟踪解码勘误修复的处理器。

set record btrace bts buffer-size 大小 ¶
set record btrace bts buffer-size unlimited

设置以BTS格式进行分支跟踪时请求的环形缓冲区大小。默认值为64KB。

如果大小是一个正数，那么GDB将尝试为每个使用btrace记录方法和BTS格式的新线程分配至少大小字节的缓冲区。实际获得的缓冲区大小可能与请求的大小不同。使用info record命令查看每个使用btrace记录方法和BTS格式的线程的实际缓冲区大小。

如果限制是unlimited或零，GDB将尝试分配一个4MB的缓冲区。

更大的缓冲区意味着更长的跟踪。另一方面，GDB在处理分支跟踪数据之前也需要更长的时间。

show record btrace bts buffer-size 大小

显示以BTS格式进行分支跟踪时请求的环形缓冲区大小的当前设置。

set record btrace pt buffer-size 大小 ¶
set record btrace pt buffer-size unlimited

设置以英特尔处理器跟踪格式进行分支跟踪时请求的环形缓冲区大小。默认值为16KB。

如果大小是一个正数，那么GDB将尝试为每个使用btrace记录方法和英特尔处理器跟踪格式的新线程分配至少大小字节的缓冲区。实际获得的缓冲区大小可能与请求的大小不同。使用info record命令查看每个线程的实际缓冲区大小。

如果限制是unlimited或零，GDB将尝试分配一个4MB的缓冲区。

更大的缓冲区意味着更长的跟踪。另一方面，GDB在处理分支跟踪数据之前也需要更长的时间。

show record btrace pt buffer-size 大小

显示以英特尔处理器跟踪格式进行分支跟踪时请求的环形缓冲区大小的当前设置。

set record btrace pt event-tracing

启用或禁用以英特尔处理器跟踪格式进行分支跟踪时的事件跟踪。启用后，事件将在执行期间作为辅助信息记录，并在单步执行命令和显示执行历史的命令期间打印。更改此设置对正在进行的记录没有影响。默认值为关闭。

show record btrace pt event-tracing

显示英特尔处理器跟踪事件跟踪的当前设置。

info record ¶

根据记录方法显示有关记录的各种统计信息：

full

对于full记录方法，它显示进程记录的状态及其内存中的执行日志缓冲区，包括：

  • 是否处于记录模式或重放模式．
  • 记录的最低指令编号（从当前执行日志开始记录指令时起计数）。
  • 记录的最高指令编号。
  • 即将重放的当前指令（如果处于重放模式）。
  • 执行日志中包含的指令数量。
  • 执行日志中可能包含的最大指令数量。
btrace

对于btrace记录方法，它显示：

  • 记录格式。
  • 已记录的指令数量。
  • 由记录的指令形成的顺序控制流块的数量。
  • 是否处于记录模式或重放模式。

对于bts记录格式，它还显示：

  • 性能环形缓冲区的大小。

对于pt记录格式，它还显示：

  • 性能环形缓冲区的大小。
record delete ¶

当记录目标在重放模式下运行时（“在过去”），删除后续执行日志，并从当前地址开始记录新的执行日志。这意味着你将放弃之前记录的“未来”，并开始记录一个新的“未来”。
