20.5.2 你必须为存根程序做的事情 ¶

GDB附带的调试存根程序是为特定的芯片架构设置的，但它们没有关于你的调试目标机器其他部分的信息。

首先，你需要告诉存根程序如何与串口通信。

int getDebugChar() ¶

编写这个子例程以从串口读取单个字符。对于你的目标系统，它可能与 getchar 相同；如果需要，使用不同的名称以便你区分两者。

void putDebugChar(int) ¶

编写这个子例程以向串口写入单个字符。对于你的目标系统，它可能与 putchar 相同；如果需要，使用不同的名称以便你区分两者。

如果你希望 GDB 能够在程序运行时停止它，你需要使用中断驱动的串口驱动程序，并安排在接收到 ^C（即 ‘\003’，控制 - C字符）时停止。这是 GDB 用于告诉远程系统停止的字符。

要使调试目标向 GDB 返回正确的状态，可能需要对标准存根程序进行更改；一种快速而粗糙的方法是直接执行断点指令（“粗糙”的部分是 GDB 报告的是 SIGTRAP 而不是 SIGINT）。

你还需要提供的其他例程包括：

void exceptionHandler (int exception_number, void *exception_address) ¶

编写此函数以将 exception_address 安装到异常处理表中。你需要这样做，因为存根程序无法知道目标系统上的异常处理表是什么样的（例如，处理器的表可能在 ROM 中，其中包含指向 RAM 中一个表的条目）。 exception_number 指定应更改的异常；其含义取决于架构（例如，不同的数字可能表示除零、未对齐访问等）。当此异常发生时，控制应直接转移到 exception_address，并且处理器状态（堆栈、寄存器等）应与处理器异常发生时相同。因此，如果要使用跳转指令到达 exception_address，它应该是一个简单的跳转，而不是跳转到子例程。

对于386，exception_address 应作为中断门安装，以便在处理程序运行时屏蔽中断。该门应处于特权级别0（最高特权级别）。SPARC 和68k存根程序能够自行屏蔽中断，无需 exceptionHandler 的帮助。

void flush_i_cache() ¶

仅在 SPARC 和 SPARCLITE 上，编写此子例程以刷新目标机器上的指令缓存（如果有的话）。如果没有指令缓存，此子例程可能是一个空操作。

在具有指令缓存的目标机器上，GDB 需要此函数以确保程序状态稳定。

你还必须确保这个库例程可用：

void *memset(void *, int, int) ¶

这是标准库函数 memset，用于将一块内存区域设置为已知值。如果你有 libc.a 的免费版本之一，可以在那里找到 memset；否则，你必须从硬件制造商处获取它，或者自己编写。

如果你不使用GNU C编译器，可能还需要其他标准库子例程；这因存根程序而异，但一般来说，存根程序可能会使用 GCC 作为内联代码生成的任何常见库子例程。
