附录J .gdb_index 部分格式 ¶

本节记录由 save gdb-index 创建的索引部分（见 索引文件加速GDB）。索引部分是特定于DWARF的；本描述中假定读者对DWARF有所了解。

映射索引文件格式设计为在任何架构上都可直接 mmap。在大多数情况下，一个数据项使用一个小端序32位整数值表示，称为 offset_type。大端序机器在使用这些值之前必须进行字节交换。本规则的例外情况会另行说明。数据的布局方式始终遵循对齐原则。

一个映射索引由几个区域按顺序排列组成。

  1. 文件头。这是一系列值，除非另有说明，否则为 offset_type 类型：
    1. 版本号，当前为9。版本1、2和3已过时。版本4使用的哈希函数与版本5和6不同。版本6包括内联函数的符号，而版本4和5不包括。版本7在符号表的CU索引中添加了属性。版本8指定来自DWARF类型单元（‘DW_TAG_type_unit’）的符号引用类型单元的符号表，而不是使用该类型的编译单元（‘DW_TAG_comp_unit’）。版本9在索引中添加了主函数 的名称和语言信息。

      GDB 只有在指定 set use-deprecated-index-sections on 时才会读取版本4、5或6的索引。GDB对可能损坏的版本7索引有一个变通方法，因此目前它没有被标记为过时。

    2. CU列表在文件开头的偏移量。
    3. 类型CU列表在文件开头的偏移量。请注意，这个区域可能为空，在这种情况下，这个偏移量将等于下一个偏移量。
    4. 地址区域在文件开头的偏移量。
    5. 符号表在文件开头的偏移量。
    6. 快捷表在文件开头的偏移量。
    7. 常量池在文件开头的偏移量。
  2. CU列表。这是一系列64位小端序值对，按CU偏移量排序。每对中的第一个元素是 .debug_info 节中一个CU的偏移量。每对中的第二个元素是该CU的长度。在映射中的其他地方对CU的引用使用CU索引，它只是此表中的基于0的索引。请注意，如果有类型CU，那么从概念上讲，CU和类型CU为了CU索引的目的形成一个单一列表。
  3. 类型CU列表。这是一系列64位小端序三元组。在一个三元组中，第一个值是CU偏移量，第二个值是CU中的类型偏移量，第三个值是类型签名。类型CU列表不排序。
  4. 地址区域。地址区域由一系列地址条目组成。每个地址条目有三个元素：
    1. 低地址。这是一个64位小端序值。
    2. 高地址。这是一个64位小端序值。与 DW_AT_high_pc 一样，该值比末尾多一个字节。
    3. CU索引。这是一个 offset_type 值。
  5. 符号表。这是一个开放地址哈希表。哈希表的大小始终是2的幂。

    哈希表中的每个槽由一对 offset_type 值组成。第一个值是符号名称在常量池中的偏移量。第二个值是CU向量在常量池中的偏移量。

    如果两个值都是0，那么哈希表中的这个槽为空。这是可以的，因为虽然0是一个有效的常量池索引，但它不能同时是字符串和CU向量的有效索引。

    表条目的哈希值通过对符号名称应用迭代哈希函数来计算。从初始值 r = 0 开始，字符串中的每个（无符号）字符 ‘c’ 根据索引版本使用公式合并到哈希中：

    版本4

    公式为 r = r * 67 + c - 113。

    版本5到7

    公式为 r = r * 67 + tolower (c) - 113。

    终止符 ‘\0’ 不包含在哈希中。

    哈希表中使用的步长通过 ((hash * 17) & (size - 1)) | 1 计算，其中 ‘hash’ 是哈希值，‘size’ 是哈希表的大小。步长用于在处理哈希冲突时找到下一个候选槽。

    哈希表中C++ 符号的名称进行了规范化。我们目前没有对规范化算法的简单描述；如果您打算创建新的索引部分，您必须阅读代码。

  6. 快捷表 这是一个具有以下字段的数据结构：
    主函数的语言

    一个 offset_type 值，指示主函数的语言作为 DW_LANG_ 常量。如果没有主函数信息，此值将为零。

    主函数的名称

    一个 offset_type 值，指示主函数名称在常量池中的偏移量。如果主函数语言的值为零，则必须忽略此值。

  7. 常量池。这只是一堆字节。它的组织方式使得对齐正确：CU向量首先存储，然后是字符串。

    常量池中的CU向量是一系列 offset_type 值。第一个值是向量中CU索引的数量。每个后续值是CU列表中一个CU的索引和符号属性。哈希表中的这个元素用于指示哪些CU定义了符号以及符号是如何使用的。 有关每个CU索引+属性条目的格式，请参见下文。

    常量池中的字符串以零结尾。

在 .gdb_index 版本7中，属性被添加到CU索引值中。如果一个符号在一个CU中有多个用途，那么每个用途都有一个CU索引+属性值。

每个CU索引+属性条目的格式如下（位0 = 最低有效位）：

位0 - 23

这是CU在CU列表中的索引。

位24 - 27

这些位保留供将来使用，必须为零。

位28 - 30

CU中符号的类型。

0

此值保留，不应使用。 通过保留零，完整的 offset_type 值与索引的先前版本向后兼容。

1

符号是一个类型。

2

符号是一个变量或枚举值。

3

符号是一个函数。

4

任何其他类型的符号。

5、6、7

这些值保留。

位31

如果值是全局的，则此位为零，如果是静态的，则为一。

确定一个符号是全局还是静态是复杂的。权威参考是 GDB 源代码中的 dwarf2read.c 文件。

这段伪代码描述了索引中符号类型和全局/静态属性的计算。 

is_external = get_attribute (die, DW_AT_external);
language = get_attribute (cu_die, DW_AT_language);
switch (die->tag)
  {
  case DW_TAG_typedef:
  case DW_TAG_base_type:
  case DW_TAG_subrange_type:
    kind = TYPE;
    is_static = 1;
    break;
  case DW_TAG_enumerator:
    kind = VARIABLE;
    is_static = language != CPLUS;
    break;
  case DW_TAG_subprogram:
    kind = FUNCTION;
    is_static = ! (is_external || language == ADA);
    break;
  case DW_TAG_constant:
    kind = VARIABLE;
    is_static = ! is_external;
    break;
  case DW_TAG_variable:
    kind = VARIABLE;
    is_static = ! is_external;
    break;
    case DW_TAG_namespace:
    kind = TYPE;
    is_static = 0;
    break;
  case DW_TAG_class_type:
  case DW_TAG_interface_type:
  case DW_TAG_structure_type:
  case DW_TAG_union_type:
  case DW_TAG_enumeration_type:
    kind = TYPE;
    is_static = language != CPLUS;
    break;
  default:
    assert (0);
  }
