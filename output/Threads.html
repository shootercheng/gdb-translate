4.10 使用多线程调试程序 ¶

在一些操作系统中，如GNU/Linux和Solaris，一个程序可能有多个执行线程。不同操作系统中线程的精确语义有所不同，但一般来说，单个程序的线程类似于多个进程 —— 不同的是它们共享一个地址空间（也就是说，它们都可以检查和修改相同的变量）。另一方面，每个线程都有自己的寄存器和执行栈，可能还有私有内存。

GDB提供了以下用于调试多线程程序的功能：

  • 新线程的自动通知
  • ‘thread thread-id’，用于在不同线程间切换的命令
  • ‘info threads’，用于查询现有线程的命令
  • ‘thread apply [thread-id-list | all] args’， 用于将一个命令应用于一组线程的命令
  • 特定于线程的断点
  • ‘set print thread-events’，用于控制线程启动和退出时消息的打印。
  • ‘set libthread-db-search-path path’，如果默认选择的 libthread_db 与程序不兼容，则允许用户指定使用哪个 libthread_db。

GDB的线程调试功能允许你在程序运行时观察所有线程 —— 但是每当GDB获得控制权时，总有一个特定的线程是调试的重点。这个线程被称为当前线程。调试命令从当前线程的角度显示程序信息。

每当GDB在你的程序中检测到一个新线程时，它会以 ‘[New systag]’ 的形式显示目标系统对该线程的标识，其中systag是一个线程标识符，其形式因特定系统而异。例如，在GNU/Linux上，你可能会看到 

[New Thread 0x41e02940 (LWP 25582)]

当GDB注意到一个新线程时。相比之下，在其他系统上，systag可能只是类似 ‘process 368’ 的形式，没有进一步的限定词。

出于调试目的，GDB为一个下级的每个线程关联一个自己的线程编号 —— 始终是一个整数。这个编号在一个下级的所有线程之间是唯一的，但在不同下级的线程之间不是唯一的。

你可以使用限定的 inferior-num.thread-num 语法（也称为限定线程ID）来引用一个下级中的给定线程，其中inferior-num是下级编号，thread-num是给定下级的线程编号。例如，线程 2.3 指的是下级2中的线程3。如果你省略 inferior-num（例如，thread 3），那么GDB会推断你指的是当前下级中的一个线程。

在你创建第二个下级之前，GDB不会显示线程ID的 inferior-num 部分，尽管你始终可以使用完整的 inferior-num.thread-num 形式来引用下级1（初始下级）中的线程。

一些命令接受以空格分隔的线程ID列表作为参数。列表元素可以是：

  1. ‘info threads’ 显示中的第一个字段所示形式的线程ID，可以带或不带下级限定符。例如，‘2.1’ 或 ‘1’。
  2. 线程编号范围，同样可以带或不带下级限定符，如 inf.thr1-thr2 或 thr1-thr2。例如，‘1.2-4’ 或 ‘2-4’。
  3. 使用星号通配符指定的一个下级的所有线程，可以带或不带下级限定符，如 inf.*（例如，‘1.*’）或 *。前者指的是给定下级的所有线程，后者不带下级限定符指的是当前下级的所有线程。

例如，如果当前下级是1，而下级7有一个ID为7.1的线程，那么线程列表 ‘1 2-3 4.5 6.7-9 7.*’ 包括下级1中的线程1到3、下级4中的线程5、下级6中的线程7到9以及下级7中的所有线程。也就是说，以展开的限定形式，与 ‘1.1 1.2 1.3 4.5 6.7 6.8 6.9 7.1’ 相同。

除了每个下级的编号外，每个线程还被分配一个唯一的全局编号，也称为全局线程ID，是一个整数。与线程ID中的线程编号部分不同，没有两个线程具有相同的全局ID，即使你正在调试多个下级。

从GDB的角度来看，一个进程总是至少有一个线程。换句话说，即使程序不是多线程的，GDB也会为程序的 “主线程” 分配一个线程编号。

调试器便利变量 ‘$_thread’ 和 ‘$_gthread’ 分别包含当前线程的每个下级线程编号和全局线程编号。你可能会发现这在编写断点条件表达式、命令脚本等方面很有用。便利变量 ‘$_inferior_thread_count’ 包含当前下级中活动线程的数量。 有关便利变量的一般信息，请参阅 便利变量。

在非停止模式下运行时（请参阅 非停止模式），新线程可以随时创建，现有线程可以随时退出，‘$_inferior_thread_count’ 每次求值时可能会返回不同的值。

如果GDB检测到程序是多线程的，它会在关于在断点处停止的通常消息中增加命中断点的线程的ID和名称。 

线程2 "客户端" 在client.c:68处命中断点1，send_message ()

同样，当程序收到信号时： 

线程1 "主" 收到信号SIGINT，中断。
info threads [-gid] [-stopped] [-running] [thread-id-list] ¶

显示一个或多个线程的信息。无参数时显示所有线程的信息。你可以使用线程ID列表语法（请参阅 线程ID列表）指定要显示的线程列表。

GDB按以下顺序为每个线程显示：

  1. GDB分配的每个下级线程编号
  2. 如果指定了 ‘-gid’ 选项，则为GDB分配的全局线程编号
  3. 目标系统的线程标识符（systag）
  4. 线程的名称（如果已知）。线程可以由用户命名（请参阅下面的 thread name），或者在某些情况下由程序本身命名。
  5. 该线程的当前栈帧摘要

GDB线程编号左边的星号 ‘*’ 表示当前线程。

例如， 

(gdb) info threads
  Id   Target Id             Frame
* 1    process 35 thread 13  main (argc=1, argv=0x7ffffff8)
  2    process 35 thread 23  0x34e5 in sigpause ()
  3    process 35 thread 27  0x34e5 in sigpause ()
    at threadtest.c:68

如果你正在调试多个下级，GDB将使用限定的 inferior-num.thread-num 格式显示线程ID。否则，只显示 thread-num。

如果你指定 ‘-stopped’ 选项，GDB会过滤命令输出，只打印已停止的线程。类似地，如果指定 ‘-running’ 选项，GDB会过滤输出，只打印正在运行的线程。如果线程数量很多，这些选项有助于减少输出列表。如果你同时指定这两个选项，GDB会打印已停止和正在运行的线程。

如果你指定 ‘-gid’ 选项，GDB会显示一列，指示每个线程的全局线程ID： 

(gdb) info threads
  Id   GId  Target Id             Frame
  1.1  1    process 35 thread 13  main (argc=1, argv=0x7ffffff8)
  1.2  3    process 35 thread 23  0x34e5 in sigpause ()
  1.3  4    process 35 thread 27  0x34e5 in sigpause ()
* 2.1  2    process 65 thread 1   main (argc=1, argv=0x7ffffff8)

在Solaris上，你可以使用特定于Solaris的命令显示有关用户线程的更多信息：

maint info sol-threads ¶

显示Solaris用户线程的信息。

thread thread-id
