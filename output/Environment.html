4.4你的程序的环境 ¶

环境由一组环境变量及其值组成。环境变量通常记录诸如你的用户名（user name）、主目录（home directory）、终端类型（terminal type）以及运行程序的搜索路径（search path）等信息。通常你通过shell设置环境变量，并且它们会被你运行的所有其他程序继承。在调试时，尝试使用修改后的环境运行你的程序而无需再次启动GDB可能会很有用。

path directory ¶

将directory添加到将传递给你的程序的PATH环境变量（可执行文件的搜索路径）的开头。GDB使用的PATH的值不会改变。你可以指定几个目录名，用空格或系统特定的分隔符字符分隔（在Unix上是‘:’，在MS-DOS和MS-Windows上是‘;’）。如果directory已经在路径中，它会被移到前面，这样就会更快地被搜索到。

你可以使用字符串‘$cwd’来指代GDB搜索路径时的当前工作目录。如果你使用‘.’代替，它指代你执行path命令的目录。在将directory添加到搜索路径之前，GDB会用当前路径替换directory参数中的‘.’。

show paths ¶

显示可执行文件的搜索路径列表（PATH环境变量）。

show environment [varname] ¶

打印程序启动时要传递给它的环境变量varname的值。如果你不提供varname，则打印要传递给你的程序的所有环境变量的名称和值。你可以将environment缩写为env。

set environment varname [=value]

将环境变量varname设置为value。该值会为你的程序（以及GDB用来启动它的shell）改变，而不是为GDB本身改变。value可以是任何字符串；环境变量的值只是字符串，任何解释都由你的程序本身提供。value参数是可选的；如果省略，变量将被设置为空值。

例如，这个命令： 

set env USER = foo

告诉被调试的程序，在随后运行时，它的用户名为‘foo’。（这里‘=’周围的空格是为了清晰起见；实际上它们不是必需的。）

请注意，在Unix系统上，GDB通过一个shell运行你的程序，该shell也会继承用set environment设置的环境。如有必要，你可以通过使用‘env’程序作为包装器来避免这种情况，而不是使用set environment。有关这样做的示例，请参阅 设置执行包装器。

用户设置的环境变量也会被传输到gdbserver，以便在启动远程子进程时使用。请参阅 QEnvironmentHexEncoded。

unset environment varname

从要传递给你的程序的环境中删除变量varname。这与‘set env varname =’不同；unset environment会从环境中删除变量，而不是为其赋空值。

用户取消设置的环境变量在启动远程子进程时也会在gdbserver上被取消设置。请参阅 QEnvironmentUnset。

警告：在Unix系统上，如果存在，GDB会使用你的SHELL环境变量指定的shell运行你的程序（如果不存在则使用/bin/sh）。如果你的SHELL变量指定了一个在非交互式启动时运行初始化文件的shell—例如C-shell的.cshrc文件、Z shell的$.zshenv文件，或者bash的BASH_ENV环境变量中指定的文件—你在该文件中设置的任何变量都会影响你的程序。你可能希望将环境变量的设置移动到仅在你登录时运行的文件中，例如.login或.profile。
