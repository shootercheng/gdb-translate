18.3单独文件中的调试信息 ¶

GDB允许你将程序的调试信息放在一个与可执行文件本身分开的文件中，并且GDB能够自动找到并加载该调试信息。由于调试信息可能非常大，有时甚至比可执行代码本身还大，因此一些系统将其可执行文件的调试信息分布在单独的文件中，用户可以仅在需要调试问题时才安装这些文件。

GDB支持两种指定单独调试信息文件的方式：

  • 可执行文件包含一个调试链接，它指定了单独调试信息文件的名称。单独调试文件的名称通常是可执行文件名.debug，其中可执行文件名是相应可执行文件的名称，不包含前导目录（例如，对于/usr/bin/ls，其调试文件名为ls.debug）。此外，调试链接为调试文件指定了一个32位的循环冗余校验（CRC）校验和，GDB使用它来验证可执行文件和调试文件是否来自同一构建。
  • 可执行文件包含一个构建ID，这是一个唯一的位串，在相应的调试信息文件中也存在。（仅在某些操作系统上，当对二进制文件使用ELF或PE文件格式以及GNU Binutils时，才支持此功能。）有关此功能的更多详细信息，请参阅The GNU Linker中命令行选项中对--build-id命令行选项的描述。构建ID并未明确指定调试信息文件的名称，但可以根据构建ID计算得出，如下所示。

根据调试信息文件的指定方式不同，GDB使用两种不同的方法来查找调试文件：

  • 对于“调试链接”方法，GDB首先在可执行文件所在的目录中查找指定的文件，然后在该目录的一个名为.debug的子目录中查找，最后在每个全局调试目录下，在一个名称与可执行文件绝对文件名的前导目录相同的子目录中查找。（在MS-Windows/MS-DOS上，可执行文件前导目录的驱动器字母会转换为一个单字母的子目录，例如，d:/usr/bin/会转换为/d/usr/bin/，因为Windows文件系统不允许文件名中包含冒号。）
  • 对于“构建ID”方法，GDB在每个全局调试目录的.build-id子目录中查找一个名为nn/nnnnnnnn.debug的文件，其中nn是构建ID位串的前2个十六进制字符，nnnnnnnn是位串的其余部分。（实际的构建ID字符串为32个或更多十六进制字符，而不是10个。）GDB可以使用构建ID自动查询debuginfod服务器，以便下载本地找不到的单独调试文件。有关更多信息，请参阅使用Debuginfod下载调试资源。

例如，假设你要求GDB调试/usr/bin/ls，它有一个调试链接指定文件ls.debug，并且其构建ID的十六进制值为abcdef1234。如果全局调试目录列表包括/usr/lib/debug，那么GDB将按以下指定顺序查找以下调试信息文件：

  • /usr/lib/debug/.build-id/ab/cdef1234.debug
  • /usr/bin/ls.debug
  • /usr/bin/.debug/ls.debug
  • /usr/lib/debug/usr/bin/ls.debug。

如果仍然找不到调试文件并且启用了debuginfod（请参阅使用Debuginfod下载调试资源），GDB将尝试从debuginfod服务器下载该文件。

全局调试信息目录默认为由GDB配置选项--with-separate-debug-dir设置的值，并由通过GDB配置选项--additional-debug-dirs提供的以冒号分隔的目录列表进行扩充。在GDB运行期间，你也可以设置全局调试信息目录，并查看GDB当前正在使用的目录列表。

set debug-file-directory 目录 ¶

将GDB搜索单独调试信息文件的目录设置为目录。可以通过路径分隔符连接多个路径组件来设置。

show debug-file-directory ¶

显示GDB搜索单独调试信息文件的目录。

调试链接是可执行文件中一个名为.gnu_debuglink的特殊节。该节必须包含：

  • 一个文件名，去除任何前导目录组件，后面跟着一个零字节，
  • 零到三个字节的填充，以达到该节内下一个四字节边界所需，
  • 一个四字节的CRC校验和，以与可执行文件本身相同的字节序存储。该校验和是通过下面给出的函数对调试信息文件的完整内容进行计算得出的，将零作为crc参数传递。

只要可执行文件格式能够包含一个名为.gnu_debuglink且内容如上所述的节，任何可执行文件格式都可以携带调试链接。

构建ID是可执行文件（以及GDB可能考虑的其他ELF二进制文件）中的一个特殊节。这个节通常名为.note.gnu.build-id，但该名称不是必需的。它包含已构建文件的唯一标识 —— 在同一构建树的多个构建中，该ID保持不变。默认算法SHA1会生成160位（40个十六进制字符）的构建ID字符串内容。在带有符号的原始构建二进制文件、其剥离变体以及单独的调试信息文件中，都存在具有相同值的相同节。

调试信息文件本身应该是一个普通的可执行文件，包含一整套链接器符号、节和调试信息。调试信息文件的节应该与原始文件具有相同的名称、地址和大小，但它们不需要包含任何数据 —— 这很像普通可执行文件中的.bss节。

GNU二进制实用工具（Binutils）包包括“objcopy”实用工具，它可以使用以下命令生成单独的可执行文件/调试信息文件对： 

objcopy --only-keep-debug foo foo.debug
strip -g foo

这些命令从可执行文件foo中移除调试信息，并将其放置在文件foo.debug中。你可以使用第一种、第二种方法或两者来链接这两个文件：

  • 调试链接方法还需要以下额外命令，以便在foo中留下一个调试链接： 
    objcopy --add-gnu-debuglink=foo.debug foo

    Ulrich Drepper的elfutils包从0.53版本开始，包含一个版本的strip命令，使得命令strip foo -f foo.debug具有与上述两个objcopy命令和ln -s命令相同的功能。

  • 使用ld --build-id或GCC对应的gcc -Wl,--build-id将构建ID嵌入到主可执行文件中。自2.18版本以来，GNU二进制实用工具（Binutils）包中提供了构建ID支持以及调试文件分离的兼容性修复。

.gnu_debuglink中使用的CRC是IEEE 802.3中定义的CRC-32，使用的多项式为： 

 x32 + x26 + x23 + x22 + x16 + x12 + x11
 + x10 + x8 + x7 + x5 + x4 + x2 + x + 1

该函数按字节计算，首先取每个字节的最低有效位。使用初始模式0xffffffff，以确保前导零会影响CRC，并且最终结果取反以确保尾随零也会影响CRC。

注意：这与处理远程串行协议 qCRC数据包（请参阅qCRC数据包）时使用的CRC多项式相同。然而，在远程串行协议的情况下，CRC是按最高有效位优先计算的，并且结果不取反，因此尾随零对CRC值没有影响。

为了完整描述，下面展示了生成.gnu_debuglink中使用的CRC的函数代码。对最初提供的crc参数取反意味着，对该函数的初始调用传入零时，将开始使用0xffffffff计算CRC。 

unsigned long gnu_debuglink_crc32 (unsigned long crc, unsigned char *buf, size_t len)
{
  static const unsigned long crc32_table[256] =
    {
      0x00000000, 0x77073096, 0xee0e612c, ...（此处省略部分内容）... 0x2d02ef8d
    };
  unsigned char *end;

  crc = ~crc & 0xffffffff;
  for (end = buf + len; buf < end; ++buf)
    crc = crc32_table[(crc ^ *buf) & 0xff] ^ (crc >> 8);
  return ~crc & 0xffffffff;
}

此计算不适用于“构建ID”方法。
