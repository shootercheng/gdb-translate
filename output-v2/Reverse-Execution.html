6 反向运行程序 ¶

在调试程序时，你可能会发现自己已经走得太远，一些感兴趣的事件已经发生，这并不罕见。如果目标环境支持， GDB 可以让你通过反向运行程序来“倒带”。

支持反向执行的目标环境应该能够“撤销”程序正常执行时发生的机器状态变化。变量、寄存器等应该恢复到它们之前的值。显然，这需要目标环境具备很高的复杂度；并非所有目标环境都能支持反向执行。

当程序反向执行时，最近执行的指令会按相反顺序“反执行”。程序计数器反向运行，沿着之前的执行线程反向进行。当每条指令被“反执行”时，该指令所改变的内存和/或寄存器的值会恢复到它们之前的状态。反向执行一段源代码后，该代码的所有副作用都应该被“撤销”，所有变量都应该恢复到它们之前的值。7.

在某些平台上， GDB 内置了对反向执行的支持，可通过 record 或 record btrace 命令激活。请参阅 记录被调试程序的执行并回放. 一些远程目标，通常是全系统模拟器，直接支持反向执行，无需任何特殊命令。

如果你在支持反向执行的目标环境中进行调试， GDB 提供以下命令。

reverse-continue [ignore-count] ¶
rc [ignore-count]

从程序最后停止的点开始，开始反向执行。反向执行会像正常执行一样，在断点和同步异常（信号）处停止。异步信号的行为取决于目标环境。

reverse-step [count] ¶

反向运行程序，直到控制到达不同源代码行的开头；然后停止程序，并将控制权返回给 GDB.

与 step 命令类似， reverse-step 只会在源代码行的开头停止。它“反执行”之前执行的源代码行。如果之前的源代码行包含对可调试函数的调用， reverse-step 将（反向）进入被调用的函数，停在 最后 一条语句（通常是返回语句）的开头。

此外，与 step 命令一样，如果调用了不可调试的函数， reverse-step 将反向运行通过它们而不停止。

reverse-stepi [count] ¶

反向执行一条机器指令。请注意，要反向执行的指令 不是 程序计数器所指向的指令，而是在它之前执行的指令。例如，如果最后一条指令是跳转指令， reverse-stepi 将带你从跳转的目标位置返回到跳转指令本身。

reverse-next [count] ¶

反向运行到当前（最内层）栈帧中之前执行的行的开头。如果该行包含函数调用，它们将被“反执行”而不停止。从函数的第一行开始， reverse-next 将带你返回到该函数的调用者， 在 函数被调用之前，就像正常的 next 命令会带你从函数的最后一行返回到其返回给调用者的位置一样。 8.

reverse-nexti [count] ¶

与 nexti, reverse-nexti 反向执行一条指令，不同的是被调用的函数会被原子性地“反执行”。也就是说，如果之前执行的指令是从另一个函数返回， reverse-nexti 将继续反向执行，直到到达从当前栈帧对该函数的调用位置。

reverse-finish ¶

就像 finish 命令会带你到当前函数返回的点一样， reverse-finish 会带你到它被调用的点。你不会停在当前函数调用的末尾，而是停在开头。

set exec-direction ¶

设置目标执行的方向。

set exec-direction reverse ¶

GDB 将以反向方式执行所有执行命令，直到执行方向模式更改为“正向”。受影响的命令包括 step, stepi, next, nexti, continue, and finish. 不能在反向模式下使用 return 命令。

set exec-direction forward

GDB 将以正常方式执行所有执行命令。这是默认设置。

脚注

(7)

请注意，有些副作用比其他副作用更容易撤销。例如，内存和寄存器相对容易处理，但设备 I/O 则比较困难。有些目标可能能够撤销设备 I/O 之类的操作，而有些则可能不行。

之间的约定 GDB 和反向执行目标只要求目标在 GDB 要求它反向执行时做一些合理的事情，然后将结果报告回 GDB. 无论目标报告给 GDB, GDB 都会报告给用户。 GDB 假定目标报告的内存和寄存器处于一致状态，但 GDB 会接受所提供的任何内容。

(8)

除非代码经过了过度优化。
