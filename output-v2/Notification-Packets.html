E.10 通知包 ¶

远程 GDB 远程串行协议包括 通知, 不需要确认的包。 GDB和存根都可以发送通知（尽管目前定义的唯一通知是由存根发送的）。 通知在不产生确认往返延迟的情况下传递信息，因此对于偶尔丢包不是问题的低影响通信很有用。

通知包的形式为‘% 数据 # 校验和’，其中 数据 是通知的内容，并且 校验和 是 数据的校验和，计算和格式化方式与普通 GDB 包相同。 通知的 数据 永远不包含‘$’、‘%’或‘#’字符。 收到通知后，接收方不会发送‘+’或‘-’来确认通知的接收或报告其损坏情况。

每个通知的 数据 以一个不包含冒号字符的名称开头，后面跟着一个冒号字符。

接收方应默默忽略损坏的通知和他们不理解的通知。 接收方在收到格式良好的通知时应重新启动超时周期，无论他们是否理解该通知。

发送方只有在本协议描述规定允许的情况下才应发送此处描述的通知。 将来，我们可能会扩展协议以允许在新的上下文中使用现有的通知；此规则有助于旧的发送方避免使新的接收方感到困惑。

（旧版本的 GDB 会忽略收到的字节，直到看到普通包开头的‘$’字节，因此新的存根可以发送通知而不用担心会使旧的客户端感到困惑。 目前没有定义 GDB 要发送的通知，但我们假设大多数旧的存根也会忽略它们。）

每个通知由三部分组成：

‘名称:事件’

通知包由发起交换的一方（目前只有存根这样做）发送，其中 事件 携带有关通知的具体信息，并且 名称 指定通知的名称。

‘确认’

由另一方发送的确认，通常 GDB, 用于确认交换并请求事件。

异步通知机制的目的是向 GDB 报告远程存根中发生了有趣的事情。

远程存根可以随时发送通知 名称:事件 ，但 GDB 在适当的时候确认通知。 在 GDB 确认之前，通知事件处于待处理状态。 一次只能有一个通知处于待处理状态；如果在 GDB 确认上一个通知之前发生了其他事件，存根必须将它们排队，以便稍后根据 确认 包从 GDB. 进行同步传输。 由于通知机制不可靠，如果存根认为 GDB 可能没有收到通知，则允许重新发送通知。

具体来说，当 GDB 没有从存根读取输入时，或者当 GDB 期望读取正常的同步响应或对其发送的包的+’/‘-’确认时，可能会出现通知。 通知包与存根的任何其他通信不同，因此不会产生歧义。

收到通知后， GDB 应通过向存根发送一个 确认 包作为常规的同步请求来确认该通知。 这种确认不需要立即进行，因为 GDB 可以先向存根发送其他不相关的包，存根应正常处理这些包。

收到 确认 包后，如果存根有其他排队的事件要报告给 GDB, 它应通过发送一个正常的 事件. GDB 然后应发送另一个 确认 包以请求更多响应；同样，也允许发送其他不相关的包，存根应正常处理这些包。

如果存根收到一个 确认 包，并且没有其他 事件 要报告，存根应返回一个‘OK’响应。 此时， GDB 已完成对通知的处理，存根已完成发送所有排队的事件。 GDB 在收到最终的‘OK’之前，不会接受任何新的通知。 如果发生更多的通知事件，存根应发送一个新的通知， GDB 应接受该通知，并且该过程应重复进行。

异步通知的过程可以通过以下示例说明： 

< - %Stop:T0505:98e7ffbf;04:4ce6ffbf;08:b1b6e54c;thread:p7526.7526;core:0;
...
-> vStopped
< - T0505:68f37db7;04:40f37db7;08:63850408;thread:p7526.7528;core:0;
-> vStopped
< - T0505:68e3fdb6;04:40e3fdb6;08:63850408;thread:p7526.7529;core:0;
-> vStopped
< - OK

定义了以下通知：

通知确认事件描述
停止vStopped回复. 该 回复 具有停止回复的形式，如 停止回复包中所述。 有关 非停止模式的远程协议支持如何确认这些通知的信息，请参考 GDB.报告非停止模式下的异步停止事件。
