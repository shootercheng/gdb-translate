5.1.6 断点条件 ¶

最简单的断点类型是每次程序到达指定位置时都会中断。 您还可以为断点指定一个 条件 。 条件只是您编程语言中的一个布尔表达式（请参阅 表达式）。 带有条件的断点每次程序到达该断点时都会计算该表达式，并且只有当条件为 真.

这与使用断言进行程序验证相反；在那种情况下，您希望在断言被违反时停止——即当条件为假时。 在 C 语言中，如果您想测试由条件 断言表示的断言，您应该在适当的断点上设置条件‘! 断言’。

观察点也接受条件；您可能不需要它们，因为观察点无论如何都会检查表达式的值——但例如，可能更简单的做法是只对变量名设置观察点，并指定一个条件来测试新值是否是一个有趣的值。

断点条件可能会有副作用，甚至可能调用您程序中的函数。 例如，这可能很有用，可以激活记录程序进度的函数，或者使用您自己的打印函数来格式化特殊的数据结构。 除非在同一地址还有另一个启用的断点，否则这些效果是完全可预测的。 （在那种情况下， GDB 可能会先看到另一个断点并停止您的程序，而不检查这个断点的条件。） 请注意，当断点被命中时，断点命令通常比分断点条件更方便、更灵活，用于执行副作用（请参阅 断点命令列表).

如果目标支持，断点条件也可以在目标端进行计算。 不是在本地计算条件， GDB 将表达式编码为适合在目标上执行的代理表达式（请参阅 GDB 代理表达式机制），独立于 GDB. 全局变量变成原始内存位置，局部变量变成栈访问，等等。

在这种情况下， GDB 只有当断点条件计算结果为真时才会收到断点触发的通知。 这种机制可能会根据目标的性能特征提供更快的响应时间，因为它不需要让 GDB 了解每个断点触发情况，即使是条件为假的情况。

断点条件可以在设置断点时通过在if’来指定。 请参阅 break 命令的参数中使用‘ 设置断点. 它们也可以在任何时候使用 condition 命令进行更改。

您还可以在 if 命令中使用 watch 关键字。 catch 命令不识别 if 关键字； condition 是对捕获点施加进一步条件的唯一方法。

condition bnum expression ¶

指定 表达式 作为断点、观察点或捕获点编号为 bnum的断点条件。 设置条件后，断点 bnum 只有当 表达式 的值为真（在 C 语言中为非零）时才会停止您的程序。 当您使用 condition, GDB 会立即检查 表达式 的语法正确性，并确定其中的符号在断点上下文中是否有引用。 如果 表达式 使用了断点上下文中未引用的符号， GDB 会打印一条错误消息： 

当前上下文中没有符号 "foo"。

GDB 实际上不会在给出 表达式。 请参阅 命令（或像 condition break if …这样设置带有条件的断点的命令）时计算 表达式.

condition -force bnum expression

当使用 -force 标志时，即使 表达式 在断点 bnum的所有当前位置都无效，也会定义该条件。 这类似于 -force-condition 命令的 break 选项。

condition bnum

移除断点编号为 bnum的条件。 它将变成一个普通的无条件断点。

断点条件的一种特殊情况是仅在断点被命中一定次数时才停止。 这非常有用，因此有一种特殊的方法来实现它，即使用断点的 忽略次数 。 每个断点都有一个忽略次数，它是一个整数。 大多数情况下，忽略次数为零，因此没有效果。 但是，如果您的程序到达一个忽略次数为正的断点，那么它不会停止，而是将忽略次数减一并继续执行。 结果，如果忽略次数的值为 n, 则断点在您的程序接下来 n 次到达该断点时不会停止。

ignore bnum count ¶

将断点编号为 bnum 的忽略次数设置为 次数. 接下来，当断点被命中 次数 次时，您的程序执行不会停止；除了减少忽略次数外， GDB 不采取任何行动。

要使断点在下次被命中时停止，请指定忽略次数为零。

当您使用 continue 从断点恢复程序执行时，您可以直接将忽略次数作为参数指定给 continue，而不是使用 ignore。 请参阅 继续和单步执行.

如果断点有一个正的忽略次数和一个条件，则不检查该条件。 一旦忽略次数达到零， GDB 将恢复检查该条件。

您可以使用一个条件来实现忽略次数的效果，例如‘$foo-- <= 0’ 使用一个每次都会递减的调试器便利变量。 请参阅 便利变量.

忽略次数适用于断点、观察点、跟踪点和捕获点。
