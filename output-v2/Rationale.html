F.5 基本原理 ¶

上述一些明显的设计决策是有争议的。

栈溢出/下溢怎么办？

GDB 应该能够查询目标以发现其栈大小。 有了这些信息，GDB 可以在翻译时确定给定的表达式是否会使栈溢出。但本规范并不涉及 GDB 应该进行何种错误检查。

为什么所有操作都使用 LONGEST 类型？

速度并不重要，但代理代码大小很重要；使用 LONGEST 会引入一堆支持代码来执行除法等操作。所以这是一个严重的问题。

首先，请注意，对于不同的操作数大小，不需要不同的字节码。你可以在不 知道 目标上栈元素实际大小的情况下生成代码。如果目标只支持 32 位整数，并且你不发送任何 64 位字节码，一切都能正常工作。这里的观察结果是，MIPS 和 Alpha 只有固定大小的寄存器，即使大多数指令只对全尺寸字进行操作，你仍然可以获得 C 语言的语义。你只需要确保在正确的时间对所有内容进行正确的符号扩展。所以不需要字节码的 32 位和 64 位变体。只需使用你支持的最大尺寸来实现所有操作。

GDB 当然应该检查目标支持哪些尺寸，这样用户可以更早而不是更晚地得到错误信息。但此信息对于正确性来说并非必需。

为什么没有 > 或 <= 运算符？

我想保持解释器的小型化，而且我们不需要它们。我们可以将 less_ 操作码与 log_not结合，并交换操作数的顺序，从而得到所有四种不对称比较运算符。 例如， (x <= y) 是 ! (x > y)，也就是 ! (y < x)。

为什么要有 log_not？
为什么要有 ext？
为什么要有 zero_ext？

这些都可以很容易地从其他指令合成，但我预计它们会被频繁使用，而且它们很简单，所以我将它们包含进来以保持字节码字符串简短。

log_not 等同于 const8 0 equal；它在一半的关系运算符中使用。

ext n 等同于 const8 s-n lsh const8 s-n rsh_signed，其中 s 是栈元素的大小；当值应该是有符号数时，它遵循 refm 和 寄存器 字节码。见下一个项目符号项。

zero_ext n 等同于 constm mask log_and；每当我们压入寄存器的值时都会使用它，因为我们不能假设寄存器的高位不是垃圾数据。

为什么没有 ref 运算符的符号扩展变体？

因为这会使 ref 运算符的数量翻倍，而且无论如何我们都需要 ext 字节码来访问位域。

为什么没有 ref 运算符的常量地址变体？

因为这会再次使 ref 运算符的数量翻倍，而且 const32 address ref32 只长一个字节。

为什么 refn 运算符必须支持未对齐的读取？

只要可执行文件的调试信息指示，GDB 就会生成在未对齐地址处读取多字节值的字节码。 此外，GDB 在生成字节码时不知道指针的值，所以它无法确定特定的读取是否对齐。

特别是，结构位域可能有几个字节长，但没有对齐规则；打包结构的成员也不一定对齐。

一般来说，在正确的 C 代码中，未对齐引用会在很多情况下出现，要么是程序员明确要求的，要么是编译器自行决定的。因此，让 GDB 代理字节码在所有情况下都能正确工作，比让 GDB 在每种情况下猜测编译器是否做了常规操作要简单。

为什么没有有副作用的运算符？

是因为我们当前的客户端不想要它们吗？这是个简单的答案。我认为真正的答案是我害怕实现函数调用。我们应该在当前合同交付后重新审视这个问题。

为什么 goto 操作不是相对于程序计数器（PC）的？

解释器无论如何都有基地址用于 PC 边界检查，这样似乎更简单。

为什么 goto 操作只有一种偏移量大小？

当前偏移量是 16 位。我对这种情况也不满意：

假设我们有多个具有不同偏移量大小的分支操作。当我从左到右生成代码时，所有的跳转都是向前跳转（表达式中没有循环），所以我在发出跳转操作码时永远不知道目标地址。因此，我要么总是假设最大的偏移量大小，要么在生成代码后对代码进行跳转松弛处理，这似乎很浪费时间。

我可以想象一个合理的表达式长度超过 256 字节。但我无法想象一个表达式长度超过 64k。因此，我们需要 16 位偏移量。这种推理很牵强，但松弛处理也很糟糕。

另一种方法是从右到左生成代码。这样我就总是知道我的偏移量大小。这可能会很有趣。

函数调用字节码在哪里？

当我们添加副作用时，应该添加这个。

为什么 reg 字节码采用 16 位寄存器编号？

英特尔的 IA - 64 架构有 128 个通用寄存器，128 个浮点寄存器，而且我确定它还有一些随机控制寄存器。

为什么我们需要 trace 和 trace_quick？

因为 GDB 需要记录表达式所涉及的所有内存内容和寄存器。如果用户想计算一个表达式 x->y->z, 代理必须记录 x 和 x->y 的值以及 x->y->z的值。

难道 trace 字节码不会使解释器的通用性降低吗？

它们确实意味着解释器包含专用代码，但这并不意味着解释器只能用于该目的。如果一个表达式不使用 trace 字节码，它们不会妨碍它。

为什么 trace_quick 不像其他操作那样消耗其参数？

一般来说，你希望运算符消耗其参数；这是一致的，并且通常可以减少必要的栈重排。然而， trace_quick 是为了节省空间的一种权宜之计；它的存在只是为了让我们不必在每个内存引用前都写 dup const8 SIZE trace 。因此，它不消耗其参数是可以的；它是为特定的上下文设计的，在这个上下文中我们确切知道它应该如何处理栈。如果我们要采用一种权宜之计，它应该是一种有效的权宜之计。

为什么 trace16 存在？

那个操作码是由与 Cygnus 签订数据跟踪工作合同的客户添加的。我个人认为它是不必要的；这么大的对象非常罕见，所以在那些情况下使用 dup const16 size trace 就可以了。

无论我们决定对……做什么 trace16，我们至少应该保留 操作码 0x30，以保持与添加它的客户的兼容性。
