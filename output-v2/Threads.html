4.10 调试多线程程序 ¶

在某些操作系统中，如 GNU/Linux 和 Solaris，一个程序可能有多个 线程 执行。 线程的确切语义在不同的操作系统中有所不同，但一般来说，一个程序的线程类似于多个进程 —— 除了它们共享一个地址空间（即它们都可以检查和修改相同的变量）。 另一方面，每个线程都有自己的寄存器和执行栈，可能还有私有内存。

GDB 提供了这些用于调试多线程程序的工具：

  • 新线程的自动通知
  • ‘线程 线程 ID’，一个用于在线程之间切换的命令
  • ‘info threads’，一个用于查询现有线程的命令
  • ‘thread apply [线程 ID 列表 | all] 参数’， 一个将命令应用于线程列表的命令
  • 特定线程的断点
  • set print thread-events’，它控制线程启动和退出消息的打印。
  • set libthread-db-search-path 路径’，它允许用户指定如果默认选择与程序不兼容时要使用的 libthread_db 如果默认选择与程序不兼容时使用。

GDB 线程调试工具允许你在程序运行时观察所有线程 —— 但每当 GDB 控制时，特定的一个线程始终是调试的焦点。 这个线程称为 当前线程. 调试命令从当前线程的角度显示程序信息。

每当 GDB 检测到程序中有新线程时，它会以 ‘[新 系统标签]’ 的形式显示目标系统对该线程的标识，其中 系统标签 是一个线程标识符，其形式因特定系统而异。 例如，在 GNU/Linux 上，你可能会看到 

[新线程 0x41e02940 (LWP 25582)]

当 GDB 注意到新线程时。 相比之下，在其他系统上 系统标签 只是类似于 ‘进程 368’ 这样的东西，没有进一步的限定。

为了调试目的 GDB 为下级进程的每个线程关联自己的线程编号 —— 始终是一个单一的整数。 这个编号在一个下级进程的所有线程之间是唯一的，但在不同下级进程的线程之间不是唯一的。

你可以使用限定的 下级进程编号.线程编号 语法，也称为 限定线程 ID, 其中 下级进程编号 是下级进程编号， 线程编号 是给定下级进程的线程编号。 例如，线程 2.3 指的是下级进程 2 的线程编号 3。 如果你省略 下级进程编号 （例如 thread 3），那么 GDB 推断你指的是当前下级进程的线程。

在你创建第二个下级进程之前 GDB 不会显示线程 ID 的 下级进程编号 部分，即使你始终可以使用完整的 下级进程编号.线程编号 形式来引用初始下级进程 1 的线程。

一些命令接受以空格分隔的 线程 ID 列表 作为参数。 列表元素可以是：

  1. 如 ‘info threads’ 显示的第一个字段所示的线程 ID，可以带有或不带有下级进程限定符。 例如，‘2.1’ 或 ‘1’。
  2. 线程编号的范围，同样可以带有或不带有下级进程限定符，如下所示 下级进程.线程 1-线程 2 或 线程 1-线程 2. 例如，‘1.2 - 4’ 或 ‘2 - 4’。
  3. 一个下级进程的所有线程，使用星号通配符指定，可以带有或不带有下级进程限定符，如下所示 下级进程.* （例如， ‘1.*’）或 *. 前一种形式指的是给定下级进程的所有线程，而后一种没有下级进程限定符的形式指的是当前下级进程的所有线程。

例如，如果当前下级进程是 1，并且下级进程 7 有一个 ID 为 7.1 的线程，线程列表 ‘1 2 - 3 4.5 6.7 - 9 7.*’ 包括下级进程 1 的线程 1 到 3、下级进程 4 的线程 5、下级进程 6 的线程 7 到 9 以及下级进程 7 的所有线程。 也就是说，以展开的限定形式表示，与 ‘1.1 1.2 1.3 4.5 6.7 6.8 6.9 7.1’ 相同。

除了每个下级进程有一个 编号外，每个线程还被分配了一个唯一的 全局 编号，也称为 全局 线程 ID, 一个单一的整数。 与线程 ID 中的线程编号部分不同，即使在调试多个下级进程时，没有两个线程有相同的全局 ID。

从 GDB的角度来看，一个进程至少有一个 线程。 换句话说， GDB 即使程序不是多线程的，也会给程序的 “主线程” 分配一个线程编号。

调试器便利变量 ‘$_thread’ 和 ‘$_gthread’ 分别包含当前线程的每个下级进程的线程编号和全局线程编号。 你可能会发现这在编写断点条件表达式、命令脚本等方面很有用。 便利变量 ‘$_inferior_thread_count’ 包含当前下级进程中活动线程的数量。 请参阅 便利变量, 以获取有关便利变量的一般信息。

当以非停止模式运行时（请参阅 非停止模式），在这种模式下，新线程可以随时创建，现有线程也可以随时退出， ‘$_inferior_thread_count’ 每次计算时可能会返回不同的值。

如果 GDB 检测到程序是多线程的，它会在通常的在断点处停止的消息中增加命中断点的线程的 ID 和名称。 

线程 2 "客户端" 命中断点 1，在 client.c:68 处调用 send_message ()

同样，当程序收到信号时： 

线程 1 "主程序" 收到信号 SIGINT，中断。
info threads [-gid] [-stopped] [-running] [thread-id-list] ¶

显示一个或多个线程的信息。 如果没有参数，则显示所有线程的信息。 你可以使用线程 ID 列表语法指定要显示的线程列表（请参阅 线程 ID 列表）。

GDB 为每个线程显示（按此顺序）：

  1. 由 GDB
  2. 由 GDB, 如果指定了 ‘-gid’ 选项
  3. 目标系统的线程标识符（系统标签)
  4. 线程的名称（如果已知）。 线程可以由用户命名（请参阅 thread name, 如下），或者在某些情况下，由程序本身命名。
  5. 该线程的当前栈帧摘要

星号 ‘*’ 在 GDB 线程编号 左侧表示当前线程。

例如， 

(gdb) info threads
  Id   目标 Id             帧
* 1    进程 35 线程 13  主函数 (argc = 1, argv = 0x7ffffff8)
  2    进程 35 线程 23  0x34e5 处调用 sigpause ()
  3    进程 35 线程 27  0x34e5 处调用 sigpause ()
    在 threadtest.c:68 处

如果你正在调试多个下级进程， GDB 使用限定的 下级进程编号.线程编号 格式显示线程 ID。 否则，只显示 线程编号 。

如果你指定 ‘-stopped’ 选项， GDB 过滤命令的输出，只打印停止的线程。 同样， 如果你指定 ‘-running’ 选项， GDB 过滤输出，只打印正在运行的线程。 如果有大量线程，这些选项有助于减少输出列表。 如果你同时指定这两个选项， GDB 将打印停止和正在运行的线程。

如果你指定 ‘-gid’ 选项， GDB 显示一列，指示每个线程的全局线程 ID： 

(gdb) info threads
  Id   GId  目标 Id             帧
  1.1  1    进程 35 线程 13  主函数 (argc = 1, argv = 0x7ffffff8)
  1.2  3    进程 35 线程 23  0x34e5 处调用 sigpause ()
  1.3  4    进程 35 线程 27  0x34e5 处调用 sigpause ()
* 2.1  2    进程 65 线程 1   主函数 (argc = 1, argv = 0x7ffffff8)

在 Solaris 上，你可以使用特定于 Solaris 的命令显示有关用户线程的更多信息：

maint info sol-threads ¶

显示 Solaris 用户线程的信息。

thread thread-id ¶

使线程 ID 线程 ID 成为当前线程。 命令参数 线程 ID 是 GDB 线程 ID，如 ‘info threads’ 显示的第一个字段所示，可以带有或不带有下级进程限定符（例如，‘2.1’ 或 ‘1’）。

GDB 通过显示你选择的线程的系统标识符及其当前栈帧摘要来响应： 

(gdb) thread 2
[切换到线程 2 (线程 0xb7fdab70 (LWP 12747))]
#0  某个函数 (ignore = 0x0) 在 example.c:8 处
8	    printf ("hello\n");

与 ‘[新 …]’ 消息一样，‘切换到’ 后面的文本形式取决于你系统识别线程的约定。

thread apply [thread-id-list | all [-ascending]] [flag]… command ¶

该 thread apply 命令允许你将指定的 命令 应用于一个或多个线程。 使用线程 ID 列表语法指定你要影响的线程（请参阅 线程 ID 列表），或者指定 all 应用于所有线程。 要按降序将命令应用于所有线程，请输入 thread apply all 命令. 要按升序将命令应用于所有线程，请输入 thread apply all -ascending 命令.

该 标志 参数控制产生什么输出以及在将 命令 应用于线程时如何处理错误。 标志 必须以一个 - 直接后跟一个字母开始 qcs. 如果提供了多个标志，它们必须单独给出，例如 -c -q.

默认情况下， GDB 在 命令产生的输出之前显示一些线程信息，并且在执行 命令 期间引发的错误将导致中止 thread apply. 可以使用以下标志来微调此行为：

-c

标志 -c, 代表 ‘继续’，会导致 命令 中的任何错误被显示，然后执行继续。 thread apply

-s

标志 -s, 代表 ‘静音’，会导致 命令 产生的任何错误或空输出被静默忽略。 也就是说，执行继续，但不打印线程信息和错误。

-q

标志 -q (‘安静’) 禁用打印线程信息。

标志 -c 和 -s 不能一起使用。

taas [option]… command ¶

的快捷方式。 将 thread apply all -s [option]… command 命令 应用于所有线程，忽略错误和空输出。

该 taas 命令接受与 thread apply all 命令相同的选项。 请参阅 thread apply all.

tfaas [option]… command ¶

的快捷方式。 将 thread apply all -s -- frame apply all -s [option]… command 命令 应用于所有线程的所有帧，忽略错误和空输出。 请注意，标志 -s 被指定了两次： 第一次 -s 确保 thread apply 只显示 frame apply 产生一些输出的线程的线程信息。 第二次 -s 是为了确保 frame apply 只显示 命令 成功产生一些输出的帧的帧信息。

例如，它可以用于在不知道变量或参数所在的线程或帧的情况下打印局部变量或函数参数，使用： 

(gdb) tfaas p 我不记得在哪里的某个局部变量

该 tfaas 命令接受与 frame apply 命令相同的选项。 请参阅 frame apply.

thread name [name] ¶

此命令为当前线程分配一个名称。 如果没有给出参数，则删除任何现有的用户指定的名称。 线程名称会出现在 ‘info threads’ 显示中。

在某些系统上，如 GNU/Linux GDB 能够确定操作系统给出的线程名称。 在这些系统上，使用 ‘thread name’ 指定的名称将覆盖系统给出的名称，并且删除用户指定的名称将导致 GDB 再次显示系统指定的名称。

thread find [regexp] ¶

搜索并显示名称或 系统标签 与提供的正则表达式匹配的线程 ID。

除了是 ‘thread name’ 命令的补充外， 此命令还允许你通过其目标 系统标签. 例如，在 GNU/Linux 上，目标 系统标签 是 LWP ID。 

(gdb) thread find 26688
线程 4 的目标 ID 为 '线程 0x41e02940 (LWP 26688)'
(gdb) info thread 4
  Id   目标 Id         帧 
  4    线程 0x41e02940 (LWP 26688) 0x00000031ca6cd372 处调用 select ()
set print thread-events ¶
set print thread-events on
set print thread-events off

该 set print thread-events 命令允许你启用或禁用在 GDB 注意到新线程启动或线程退出时打印消息。 默认情况下，如果目标支持检测这些事件，则会打印这些消息。 请注意，并非所有目标都可以禁用这些消息。

show print thread-events ¶

显示在 GDB 检测到线程启动和退出时是否会打印消息。

请参阅 停止和启动多线程程序, 以获取有关 GDB 在停止和启动多线程程序时的行为的更多信息。

请参阅 设置观察点, 以获取有关多线程程序中观察点的信息。

set libthread-db-search-path [path] ¶

如果设置了此变量 路径 是一个以冒号分隔的目录列表 GDB 将用于搜索 libthread_db. 如果你省略 路径, ‘libthread-db-search-path’ 将重置为其默认值（$sdir:$pdir 在 GNU/Linux 和 Solaris 系统上）。 在内部，默认值来自 LIBTHREAD_DB_SEARCH_PATH 宏。

在 GNU/Linux 和 Solaris 系统上 GDB 使用一个 “辅助” libthread_db 库来获取下级进程中线程的信息。 GDB 将使用 ‘libthread-db-search-path’ 来查找 libthread_db. GDB 如果通过 ‘set auto-load libthread-db’ 启用了下级进程特定的线程调试库加载，也会首先进行查询（请参阅 自动加载线程调试库)。

‘$sdir’ 对于 ‘libthread-db-search-path’ 的一个特殊条目指的是通常用于加载共享库的默认系统目录。 ‘$sdir’ 条目是唯一不需要通过 ‘set auto-load libthread-db’ 启用的类型（请参阅 自动加载线程调试库)。

‘$pdir’ 对于 ‘libthread-db-search-path’ 的一个特殊条目指的是下级进程中 libpthread 被加载的目录。

对于 libthread_db 库 GDB 在上述目录中找到的任何库 GDB 尝试使用当前下级进程对其进行初始化。 如果此初始化失败（这可能是由于 libthread_db 和 libpthread之间的版本不匹配） GDB 将卸载 libthread_db, 并继续下一个目录。 如果没有一个 libthread_db 库成功初始化 GDB 将发出警告并禁用线程调试。

设置 libthread-db-search-path 目前仅在某些平台上实现。

show libthread-db-search-path ¶

显示当前的 libthread_db 搜索路径。

set debug libthread-db ¶
show debug libthread-db

打开或关闭与 libthread_db相关事件的显示。 使用 1 启用， 0 禁用。

set debug threads [on|off] ¶
show debug threads

当 ‘开启’ GDB 将在创建和删除线程时打印额外的消息。

对于某些调试目标 GDB 支持访问位于线程本地存储（TLS) 中的变量。 TLS 变量类似于全局变量，不同之处在于每个线程都有自己的变量副本。 虽然通常在多线程程序中使用 TLS 变量也可以在没有线程的程序中使用。 C 库变量 errno 可能是最突出的一个 TLS 变量的例子，它经常在非线程程序中使用。 对于 GDB 没有良好 TLS 支持的目标，打印或更改 errno 的值可能无法直接实现。

GNU/Linux 和 FreeBSD 目标支持访问 TLS 变量。 在 GNU/Linux 上，辅助库 libthread_db, 用于帮助解析 TLS 变量的地址。 一些 FreeBSD 和一些 GNU/Linux 目标也有 GDB内部 TLS 解析代码。 GNU/Linux 目标将尝试使用 TLS 地址查找功能，但在 libthread_db, 不可用时将回退到使用其内部 TLS 支持。 这可能在交叉调试场景中发生，或者在调试以某种方式链接的程序时发生，例如 libthread_db libthread_db 支持不可用 —— 这包括静态链接的程序、链接早于 2.34 版本的 GLIBC ，但不包括 libpthread, 以及使用其他（非GLIBC) C 库。

maint set force-internal-tls-address-lookup ¶
maint show force-internal-tls-address-lookup

打开或关闭强制使用 GDB内部 TLS 地址查找代码。 使用 on 启用，使用 off 禁用。 此设置的默认值是 off.

禁用时 GDB 将尝试使用辅助 libthread_db 库（如果可能），但必要时将回退到使用其自己的内部 TLS 地址查找机制。

启用时 GDB 将仅使用 GDB的内部 TLS 地址查找机制（如果存在）。

此命令仅适用于 GNU/Linux 目标。 它的主要用途是用于测试 —— GDB 测试套件中的某些测试使用此命令强制使用内部 TLS 地址查找。
