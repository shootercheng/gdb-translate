5.1.2 设置观察点 ¶

你可以使用观察点在表达式的值发生变化时停止执行，而不必预测这种变化可能发生的特定位置。（这有时被称为 数据断点.）表达式可以像单个变量的值一样简单，也可以像由运算符组合的多个变量一样复杂。示例包括：

  • 对单个变量值的引用。
  • 将地址强制转换为适当的数据类型。例如，‘*(int *)0x12345678’将监视指定地址处的一个 4 字节区域（假设一个 int 占用 4 字节）。
  • 一个任意复杂的表达式，例如‘a*b + c/d’。该表达式可以使用程序原生语言中有效的任何运算符（请参阅 使用 GDB 与不同语言）。

即使表达式尚未求值，你也可以为其设置观察点。例如，你可以在‘*global_ptr’在‘global_ptr’初始化之前设置观察点。 GDB 当你的程序设置‘global_ptr’并且表达式产生有效值时，程序将停止。如果表达式以除改变变量之外的其他方式变得有效（例如，如果‘*global_ptr’所指向的内存由于 malloc 调用而变得可读）， GDB 可能直到表达式下一次改变时才会停止。

根据你的系统，观察点可以用软件或硬件实现。 GDB 通过单步执行你的程序并每次测试变量的值来实现软件观察点，这比正常执行慢数百倍。（但这可能仍然值得，以便捕获你不知道程序哪部分是罪魁祸首的错误。）

在某些系统上，例如大多数基于 PowerPC 或 x86 的目标系统， GDB 支持硬件观察点，这不会减慢程序的运行速度。

watch [-l|-location] expr [thread thread-id] [mask maskvalue] [task task-id] ¶

为一个表达式设置观察点。 GDB 当程序写入表达式 表达式 并且其值发生变化时，程序将中断。此命令最简单（也是最常用）的用法是监视单个变量的值： 

(gdb) watch foo

如果命令包含一个 [thread thread-id] 参数 GDB ，则仅当由 线程 ID 标识的线程改变了 表达式的值时，程序才会中断。如果任何其他线程改变了 表达式的值， GDB 将不会中断。请注意，以这种方式限制到单个线程的观察点仅适用于硬件观察点。

类似地，如果给出了 task 参数，那么观察点将特定于指定的 Ada 任务（请参阅 Ada 任务扩展）。

通常，观察点会考虑 表达式 中变量的作用域（见下文）。 -location 参数告诉 GDB 改为监视 表达式所引用的内存。在这种情况下， GDB 将计算 表达式，获取结果的地址，并监视该地址处的内存。结果的类型用于确定被监视内存的大小。如果表达式的结果没有地址，那么 GDB 将打印一个错误。

该 [mask maskvalue] 参数允许创建掩码观察点，如果当前架构支持此功能（例如，PowerPC 嵌入式架构，请参阅 PowerPC 嵌入式）。一个 掩码观察点 除了要监视的地址之外，还指定一个掩码。掩码指定在将被调试程序访问的地址与观察点地址进行匹配时，应忽略地址的某些位（掩码中被置零的位）。因此，掩码观察点可以同时监视多个地址——那些未掩码位与观察点地址中未掩码位相同的地址。该 mask 参数意味着 -location。示例： 

(gdb) watch foo mask 0xffff00ff
(gdb) watch *0xdeadbeef mask 0xffffff00
rwatch [-l|-location] expr [thread thread-id] [mask maskvalue] ¶

设置一个观察点，当程序读取 表达式 的值时，程序将中断。

awatch [-l|-location] expr [thread thread-id] [mask maskvalue] ¶

设置一个观察点，当程序读取或写入 表达式 时，程序将中断。

info watchpoints [list…] ¶

此命令使用与 info break 相同的格式打印观察点列表（请参阅 设置断点）。

如果你监视一个以数字形式输入的地址的变化，你需要对其进行解引用，因为地址本身只是一个常量数字，永远不会改变。 GDB 拒绝创建监视永远不会改变的值的观察点： 

(gdb) watch 0x600850
无法监视常量值 0x600850。
(gdb) watch *(int *) 0x600850
观察点 1: *(int *) 6293584

GDB 设置一个 硬件观察点 如果可能的话。硬件观察点执行速度非常快，并且调试器会在值发生变化的确切指令处报告值的变化。如果 GDB 无法设置硬件观察点，它会设置一个软件观察点，该观察点执行速度较慢，并且会在变化发生后的下一个 语句，而不是指令处报告值的变化。

你可以强制 GDB 仅使用软件观察点，通过 set can-use-hw-watchpoints 0 命令。将此变量设置为零后， GDB 即使底层系统支持硬件观察点，也永远不会尝试使用它们。（请注意，在 之前 设置 can-use-hw-watchpoints 为零之前设置的硬件辅助观察点仍将使用监视表达式值的硬件机制。）

set can-use-hw-watchpoints ¶

设置是否使用硬件观察点。

show can-use-hw-watchpoints ¶

显示当前使用硬件观察点的模式。

对于远程目标，你可以限制 GDB 将使用的硬件观察点数量，请参阅 set remote hardware-breakpoint-limit。

当你发出 watch 命令时， GDB 会报告 

硬件观察点 编号： 表达式

如果它能够设置一个硬件观察点。

目前， awatch 和 rwatch 命令只能设置硬件观察点，因为在不检查每一条正在执行的指令的情况下，无法检测到对不改变被监视表达式值的数据的访问，并且 GDB 目前不会这样做。如果 GDB 发现使用 awatch 或 rwatch 命令无法设置硬件断点，它会打印一条类似这样的消息： 

表达式无法用读/访问观察点实现。

有时， GDB 无法设置硬件观察点是因为被监视表达式的数据类型比目标机器上的硬件观察点所能处理的范围更宽。例如，某些系统只能监视最多 4 字节宽的区域；在这样的系统上，你无法为产生双精度浮点数（通常为 8 字节宽）的表达式设置硬件观察点。作为一种解决方法，可能可以将大区域拆分成一系列较小的区域，并使用单独的观察点来监视它们。

如果你设置了太多的硬件观察点， GDB 在恢复程序执行时可能无法插入所有观察点。由于在程序即将恢复执行之前，活动观察点的确切数量是未知的， GDB 在你设置观察点时可能无法就此向你发出警告，而警告只会在程序恢复执行时打印出来： 

硬件观察点 编号：无法插入观察点

如果发生这种情况，请删除或禁用一些观察点。

监视引用多个变量的复杂表达式也会耗尽硬件辅助观察点可用的资源。这是因为 GDB 需要使用单独分配的资源来监视表达式中的每个变量。

如果你使用 print 或 call交互式调用一个函数，你设置的任何观察点在 GDB 到达另一种断点或调用完成之前都将处于非活动状态。

GDB 当局部（自动）变量或涉及此类变量的表达式超出作用域时，即当执行离开定义这些变量的代码块时，会自动删除监视这些变量的观察点。特别是，当被调试的程序终止时， 所有 局部变量都会超出作用域，因此只有监视全局变量的观察点仍然有效。如果你重新运行程序，你需要再次设置所有这些观察点。一种方法是在 main 函数的入口处设置一个代码断点，当程序在此处中断时，设置所有观察点。

在多线程程序中，观察点将检测到每个线程对被监视表达式的更改。

警告： 在多线程程序中，软件观察点的作用有限。如果 GDB 创建一个软件观察点，它只能监视一个表达式的值 在单个线程中。如果你确定表达式的变化仅由当前线程的活动引起（并且你也确定没有其他线程可以成为当前线程），那么你可以像往常一样使用软件观察点。然而， GDB 可能不会注意到非当前线程的活动何时改变了表达式。（相比之下，硬件观察点会监视所有线程中的表达式。）

请参阅 set remote hardware-watchpoint-limit。
