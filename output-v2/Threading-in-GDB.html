23.3.2.2 GDB 中的线程处理 ¶

GDB 不是线程安全的。如果你的 Python 程序使用多个 线程，你必须小心，仅在 GDB特定的 函数中调用 GDB 线程。 GDB 提供了一些 函数来帮助处理这个问题。

函数： gdb.blocked_signals () ¶

如前面所述（请参阅 基础 Python），某些信号必须被 传递到 GDB 主线程。 blocked_signals 函数返回一个上下文管理器，该管理器将在进入时阻塞这些信号。这可以在启动新线程时使用，以确保在那里阻塞这些信号，例如： 

with gdb.blocked_signals():
   start_new_thread()
类： gdb.Thread ¶

这是 Python 的 threading.Thread 类的子类。它 重写了 start 方法以调用 blocked_signals, 使得 这成为一个易于使用的替代方案，用于创建在 GDB中能很好工作的线程。

函数： gdb.interrupt () ¶

这会使 GDB 做出反应，就好像用户在终端上键入了控制 - C 字符一样。也就是说，如果被调试程序正在运行，它将被中断；如果一个 GDB 命令正在执行，它将被停止； 如果一个 Python 命令正在运行， KeyboardInterrupt 将被 引发。

与 GDB中的大多数 Python API 不同 interrupt 是 线程安全的。

函数： gdb.post_event (event) ¶

将 事件，一个不接受参数的可调用对象，放入 GDB的内部事件队列。这个可调用对象将在稍后的某个时间，在 GDB的事件处理期间被调用。使用 post_event 发布的事件将按发布的顺序运行；但是，无法知道它们相对于 GDB内的其他事件何时会被处理。

与 GDB中的大多数 Python API 不同 post_event 是 线程安全的。例如： 

(gdb) python
>import threading
>
>class Writer():
> def __init__(self, message):
>        self.message = message;
> def __call__(self):
>        gdb.write(self.message)
>
>class MyThread1 (threading.Thread):
> def run (self):
>        gdb.post_event(Writer("Hello "))
>
>class MyThread2 (threading.Thread):
> def run (self):
>        gdb.post_event(Writer("World\n"))
>
>MyThread1().start()
>MyThread2().start()
>end
(gdb) Hello World
