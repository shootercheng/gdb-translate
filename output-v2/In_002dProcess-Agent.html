31 进程内代理 ¶

传统的调试模型在概念上速度较慢，但工作正常，因为大多数错误可以在调试模式执行中重现。然而，随着多核或众核处理器成为主流，多线程程序越来越受欢迎，应该会有越来越多的错误只在正常模式执行时才会显现出来，例如线程竞争，因为调试器对程序计时的干扰可能会掩盖这些错误。另一方面，在某些应用程序中，调试器长时间中断程序执行以使开发人员了解其行为的有用信息是不可行的。如果程序的正确性取决于其实时行为，调试器引入的延迟可能会导致程序失败，即使代码本身是正确的。能够在不中断程序的情况下观察其行为是很有用的。

因此，传统的调试模型过于侵入性，无法重现一些错误。为了减少对程序的干扰，我们可以减少调试器执行的操作数量。 进程内代理，一个共享库，与被调试进程在同一进程中运行，并且能够自行执行一些调试操作。因此，调试器仅在必要时介入，相应地可以提高调试性能。请注意，对程序的干扰可以减少但不能完全消除，因为进程内代理仍会停止或减慢程序的运行。

进程内代理可以在执行调试操作期间解释和执行代理表达式（见 GDB 代理表达式机制)。代理表达式可用于不同的目的，例如在跟踪点收集数据，以及在断点进行条件评估。

您可以使用以下命令控制是否使用进程内代理作为调试辅助工具：

set agent on ¶

使进程内代理代表调试器执行一些操作。用户请求的哪些操作将由进程内代理执行取决于其能力。例如，如果您请求在进程内代理中评估断点条件，并且进程内代理也具备此能力，那么断点条件将在进程内代理中进行评估。

set agent off ¶

禁用进程内代理执行调试操作。所有操作将由 GDB执行。

show agent ¶

显示进程内代理执行调试操作的当前设置。
