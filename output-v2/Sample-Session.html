1 一个示例 GDB 会话 ¶

您可以在闲暇时阅读本手册，了解关于 GDB. 然而，有几个命令足以开始使用调试器。本章将介绍这些命令。

早期版本之一的 GNU m4 (一个通用宏处理器) 存在以下错误：有时，当我们将其引号字符串从默认值更改时，用于在另一个宏定义中捕获一个宏定义的命令会停止工作。在以下简短的 m4 会话中，我们定义了一个宏 foo 它扩展为 0000；然后我们使用 m4 内置函数 defn 来定义 bar 为相同的内容。然而，当我们将开始引号字符串更改为 <QUOTE> ，结束引号字符串更改为 <UNQUOTE>, 相同的过程无法定义一个新的同义词 baz: 

$ cd gnu/m4
$ ./m4
define(foo,0000)

foo
0000
define(bar,defn(`foo'))

bar
0000
changequote(<QUOTE>,<UNQUOTE>)

define(baz,defn(<QUOTE>foo<UNQUOTE>))
baz
Ctrl-d
m4: 输入结束: 0: 致命错误: 字符串中遇到 EOF

让我们使用 GDB 来尝试查看发生了什么。 

$ gdb m4
GDB 是自由软件，您可以在某些条件下分发它的副本；输入 "show copying" 查看条件。
对于 GDB绝对没有任何保证；输入 "show warranty" 了解详情。

GDB 17.2，版权所有 1999 自由软件基金会，Inc...
(gdb)

GDB 只读取足够的符号数据，以便在需要时知道在哪里找到其余部分；因此，第一个提示符很快就会出现。 现在我们告诉 GDB 使用比平时更窄的显示宽度，以便示例能适合本手册。 

(gdb) set width 70

我们需要了解 m4 内置函数 changequote 是如何工作的。 查看源代码后，我们知道相关的子例程是 m4_changequote, 所以我们使用 GDB break 命令在那里设置一个断点。 

(gdb) break m4_changequote
断点 1 在 0x62f4 处：文件 builtin.c，第 879 行。

使用 run 命令，我们开始让 m4 在 GDB 控制下运行；只要控制没有到达 m4_changequote 子例程，程序就会像往常一样运行： 

(gdb) run
开始执行程序: /work/Editorial/gdb/gnu/m4/m4
define(foo,0000)

foo
0000

为了触发断点，我们调用 changequote. GDB 暂停 m4的执行，显示它停止时的上下文信息。 

changequote(<QUOTE>,<UNQUOTE>)

断点 1，m4_changequote (argc=3, argv=0x33c70)
    在 builtin.c:879 处
879         if (bad_argc(TOKEN_DATA_TEXT(argv[0]),argc,1,3))

现在我们使用命令 n (next) 将执行推进到当前函数的下一行。 

(gdb) n
882         set_quotes((argc >= 2) ? TOKEN_DATA_TEXT(argv[1])\
 : nil,

set_quotes 看起来是一个有希望的子例程。我们可以使用命令 s (step) 而不是 next. step 跳转到 任何 子例程中要执行的下一行，所以它会进入 set_quotes. 

(gdb) s
set_quotes (lq=0x34c78 "<QUOTE>", rq=0x34c88 "<UNQUOTE>")
    在 input.c:530 处
530         if (lquote != def_lquote)

显示显示了子例程所在的位置 m4 现在暂停（及其参数）的操作被称为栈帧显示。它 显示栈的摘要。我们可以使用 backtrace 命令（也可以拼写为 bt），来查看我们在整个栈中的位置： backtrace 命令为每个活动子例程显示一个栈帧。 

(gdb) bt
#0  set_quotes (lq=0x34c78 "<QUOTE>", rq=0x34c88 "<UNQUOTE>")
    在 input.c:530 处
#1  0x6344 处的 m4_changequote (argc=3, argv=0x33c70)
    在 builtin.c:882 处
#2  0x8174 处的 expand_macro (sym=0x33320) 在 macro.c:242 处
#3  0x7a88 处的 expand_token (obs=0x0, t=209696, td=0xf7fffa30)
    在 macro.c:71 处
#4  0x79dc 处的 expand_input () 在 macro.c:40 处
#5  0x2930 处的 main (argc=0, argv=0xf7fffb20) 在 m4.c:195 处

我们再逐行执行几行代码，看看会发生什么。前两次，我们可以使用 ‘s’；接下来的两次我们使用 n 以避免陷入 xstrdup 子例程。 

(gdb) s
0x3b5c  532         if (rquote != def_rquote)
(gdb) s
0x3b80  535         lquote = (lq == nil || *lq == '\0') ?  \
def_lquote : xstrdup(lq);
(gdb) n
536         rquote = (rq == nil || *rq == '\0') ? def_rquote\
 : xstrdup(rq);
(gdb) n
538         len_lquote = strlen(rquote);

显示的最后一行看起来有点奇怪；我们可以检查变量 lquote 和 rquote ，看看它们是否实际上是我们指定的新左右引号。我们使用命令 p (print) 来查看它们的值。 

(gdb) p lquote
$1 = 0x35d40 "<QUOTE>"
(gdb) p rquote
$2 = 0x35d50 "<UNQUOTE>"

lquote 和 rquote 确实是新的左右引号。 为了查看一些上下文，我们可以使用 l (list) 命令显示当前行周围的十行源代码。 

(gdb) l
533             xfree(rquote);
534
535         lquote = (lq == nil || *lq == '\0') ? def_lquote\
 : xstrdup (lq);
536         rquote = (rq == nil || *rq == '\0') ? def_rquote\
 : xstrdup (rq);
537
538         len_lquote = strlen(rquote);
539         len_rquote = strlen(lquote);
540     }
541
542     void

让我们跳过设置 len_lquote 和 len_rquote, 然后检查这些变量的值。 

(gdb) n
539         len_rquote = strlen(lquote);
(gdb) n
540     }
(gdb) p len_lquote
$3 = 9
(gdb) p len_rquote
$4 = 7

这看起来肯定不对，假设 len_lquote 和 len_rquote 分别是 lquote 和 rquote 的长度。我们可以使用 p 命令将它们设置为更好的值，因为它可以打印任何表达式的值 —— 并且该表达式可以包括子例程调用和赋值。 

(gdb) p len_lquote=strlen(lquote)
$5 = 7
(gdb) p len_rquote=strlen(rquote)
$6 = 9

这是否足以解决使用新引号与 m4 内置函数 defn的问题？我们可以使用 m4 继续执行 c (continue) 命令，然后尝试最初导致问题的示例： 

(gdb) c
继续执行。

define(baz,defn(<QUOTE>foo<UNQUOTE>))

baz
0000

成功！新引号现在的效果和默认引号一样好。问题似乎只是定义错误长度的两个拼写错误。我们通过给它输入 EOF 让 m4 退出： 

Ctrl-d
程序正常退出。

消息 ‘程序正常退出。’ 来自 GDB；它 表示 m4 已经执行完毕。我们可以使用 GDB 命令结束我们的 GDB quit 会话。 

(gdb) quit
