23.3.2.41 Python 中的指令反汇编 ¶

GDB的内置反汇编器可以使用 Python API 进行扩展，甚至替换。与反汇编器相关的功能包含在 gdb.disassembler 模块：

类： gdb.disassembler.DisassembleInfo ¶

反汇编由此类的实例驱动。每次 GDB 需要反汇编一条指令时，将创建此类的一个实例并传递给注册的反汇编器。然后反汇编器负责反汇编指令并返回结果。

此类型的实例通常在 GDB, 但是，可以创建此类型实例的副本，请参阅 __init__ 有关更多详细信息。

此类具有以下属性和方法：

一个只读整数，包含 GDB 要反汇编单条指令的地址。

此 gdb.Architecture （请参阅 架构的 Python 表示) 对于 GDB 当前正在反汇编，此属性是只读的。

返回 True 如果 DisassembleInfo 对象有效， False 如果无效。一个 DisassembleInfo 对象在为其创建的反汇编调用返回后将变为无效。调用其他 DisassembleInfo 创建的反汇编调用返回后，调用其他方法或访问属性将引发 DisassembleInfo 方法，或访问 DisassembleInfo 属性，如果对象无效将引发 RuntimeError 异常。

函数： DisassembleInfo.read_memory (长度, 偏移量) ¶

此方法允许反汇编器读取要反汇编的指令的字节。该方法从 长度 字节。 偏移量 开始读取 DisassembleInfo.address.

重要的是，反汇编器应使用此方法读取指令字节，而不是直接读取 inferior 内存，因为在某些情况下 GDB 从内部缓冲区而不是直接从 inferior 内存进行反汇编，调用此方法会处理此细节。

返回一个缓冲区对象，其行为类似于数组或字符串，就像 Inferior.read_memory 一样 （请参阅 Inferior.read_memory）。返回的缓冲区长度将始终恰好为 长度.

如果 GDB 无法读取所需内存，则会 gdb.MemoryError 引发异常（请参阅 异常处理）。

子类可以重写此方法，以控制在从内存读取时 GDB 看到的内容 （请参阅 内置反汇编）。重写此方法时，重要的是要理解 builtin_disassemble 如何使用 此方法。

在反汇编单条指令时，可能会多次调用 此方法，并且相同的字节可能会被多次读取。任何 单次调用可能只读取总指令字节的一部分。

如果 read_memory 的实现无法读取 请求的内存内容，例如，如果有从无效内存地址读取的请求，则会 gdb.MemoryError 引发

引发一个 MemoryError 在 read_memory 内部并不 自动意味着 MemoryError 会引发 builtin_disassemble。有可能 GDB的内置 反汇编器正在探测可用的字节数。当 read_memory 引发 MemoryError 内置 反汇编器可能能够使用其可用的字节进行完整的反汇编，在这种情况下 builtin_disassemble 本身不会引发 MemoryError。

在 read_memory 中引发的任何其他异常类型将传播 回去，并由 builtin_disassemble重新引发。

反汇编器部分在创建 DisassemblerResult 时用于表示指令内的样式 （请参阅 反汇编器结果类）。

类： gdb.disassembler.Disassembler ¶

这是所有用户实现的反汇编器必须继承的基类。

函数： Disassembler.__init__ (名称) ¶

构造函数接受 名称, 一个字符串，应该是此反汇编器的简短 名称。

函数： Disassembler.__call__ (信息) ¶

此 __call__ 方法必须由子类重写以 执行反汇编。调用 __call__ 此基类上的方法将 引发 NotImplementedError 异常。

此 信息 参数是 DisassembleInfo的一个实例，并且 描述了 GDB 要反汇编的指令。

如果此函数返回 None, 这向 GDB 表明此类不希望反汇编请求的 指令。 GDB 然后将使用其内置反汇编器来 执行反汇编。

或者，此函数可以返回一个 DisassemblerResult 表示反汇编后的指令，下面将更详细地描述此类型。

此 __call__ 方法可以引发 gdb.MemoryError 异常（请参阅 异常处理）以向 GDB 表明访问所需内存时出现问题，然后这将由 GDB 在反汇编器输出中显示。

理想情况下，调用 __call__ 的唯一三种结果是返回 None, 成功反汇编后返回一个 DisassemblerResult, 或者 MemoryError 表示读取内存时出现问题。

然而，由于 __call__ 的实现可能由于其他原因失败，例如，执行反汇编所需的某些外部资源暂时不可用，那么，如果 __call__ 引发 GdbError, 异常将转换为字符串 并打印在反汇编输出的末尾，反汇编 请求将停止。

由 __call__ 方法引发的任何其他异常类型被视为用户代码中的错误，异常将根据 set python print-stack 设置 （请参阅 set python print-stack）。

无法对 DisassemblerResult 类进行子类化。

类： gdb.disassembler.DisassemblerAddressPart ¶

此 DisassemblerAddressPart 类表示反汇编指令中的绝对 地址。使用 DisassemblerAddressPart 而不是 DisassemblerTextPart 与 STYLE_ADDRESS 一起使用是首选的， GDB 将地址显示为绝对地址，并将查找合适的符号显示在地址旁边。使用 DisassemblerAddressPart 还确保尊重用户设置，例如 set print max-symbolic-offset 。

以下是一个 x86 - 64 指令的示例： 

call   0x401136 <foo>

在此指令中， 0x401136 <foo> 是从单个 DisassemblerAddressPart生成的。 0x401136 将使用 STYLE_ADDRESS样式化，并且 foo 将使用 STYLE_SYMBOL样式化。 < 和 > 将使用 STYLE_TEXT样式化。

如果不需要包含符号名称，则可以使用 DisassemblerTextPart 样式的 STYLE_ADDRESS 代替。

此类的实例不能直接创建，而是调用 DisassembleInfo.address_part 创建此类的新实例（请参阅 DisassembleInfo 类）。

除了其父类的属性外， DisassemblerAddressPart 还有以下附加属性：

下表列出了所有可用的反汇编器样式。 GDB 将这些样式常量映射到其样式设置（请参阅 输出样式）。在某些情况下，几个样式常量会产生相同的样式设置，因此在屏幕上会产生相同的视觉效果。这在 GDB的未来版本中可能会改变，因此应谨慎选择正确的样式常量，以确保在 GDB的未来版本中输出样式正确。

gdb.disassembler.STYLE_TEXT ¶

这是 GDB 在样式化反汇编器输出时使用的默认样式。此样式应用于指令中不适合下面列出的任何其他样式的任何部分。 GDB 使用其默认样式对具有此样式的文本进行样式化。

gdb.disassembler.STYLE_MNEMONIC ¶

此样式用于样式化主要指令助记符，通常出现在反汇编指令字符串的开头或附近。

GDB 使用 disassembler mnemonic 样式设置对具有此样式的文本进行样式化。

gdb.disassembler.STYLE_SUB_MNEMONIC ¶

此样式用于样式化反汇编指令中的任何子助记符。子助记符是指令中控制指令功能但与主要助记符不连续的任何文本（主要助记符将具有 STYLE_MNEMONIC样式）。

例如，考虑以下 AArch64 指令： 

add	w16, w7, w1, lsl #1

此 add 是主要指令助记符，将被赋予 STYLE_MNEMONIC样式，而 lsl 是子助记符，将被赋予 STYLE_SUB_MNEMONIC样式。

GDB 使用 disassembler mnemonic 样式设置对具有此样式的文本进行样式化。

gdb.disassembler.STYLE_ASSEMBLER_DIRECTIVE ¶

有时一系列字节无法解码为有效的指令。在这种情况下，反汇编器可能会选择使用汇编指令来表示反汇编的结果，例如： 

.word	0x1234

在这种情况下， .word 将被赋予 STYLE_ASSEMBLER_DIRECTIVE 样式。汇编指令在很多方面类似于助记符，但不是架构指令集的一部分。

GDB 使用 disassembler mnemonic 样式设置对具有此样式的文本进行样式化。

gdb.disassembler.STYLE_REGISTER ¶

此样式用于样式化反汇编指令中表示寄存器名称或寄存器编号的任何文本。

GDB 使用 disassembler register 样式设置对具有此样式的文本进行样式化。

gdb.disassembler.STYLE_ADDRESS ¶

此样式用于样式化反汇编指令中表示绝对地址的数值。

在创建具有此样式的 DisassemblerTextPart 时，应考虑是否 DisassemblerAddressPart 更合适。请参阅 反汇编器样式部分 以了解每个部分的功能描述。

GDB 使用 disassembler address 样式设置对具有此样式的文本进行样式化。

gdb.disassembler.STYLE_ADDRESS_OFFSET ¶

此样式用于样式化反汇编指令中表示地址偏移量的数值。当指令本身要访问内存，并且该值用于偏移要访问的地址时，该值被视为地址偏移量。

例如，架构可能有一条指令，使用寄存器中的地址从内存加载。如果该指令还允许将立即偏移量编码到指令中，这将是一个地址偏移量。类似地，分支指令可能跳转到寄存器中的地址加上编码到指令中的地址偏移量。

GDB 使用 disassembler immediate 样式设置对具有此样式的文本进行样式化。

gdb.disassembler.STYLE_IMMEDIATE ¶

使用 STYLE_IMMEDIATE 对于反汇编指令中的任何数值，当这些值不是地址、地址偏移量或寄存器编号时（在这些情况下可以使用 STYLE_ADDRESS, STYLE_ADDRESS_OFFSET, 或 STYLE_REGISTER 样式）。

GDB 使用 disassembler immediate 样式设置对具有此样式的文本进行样式化。

gdb.disassembler.STYLE_SYMBOL ¶

此样式用于样式化反汇编指令中包含的符号的文本名称。符号名称通常包含在反汇编指令中的绝对地址旁边，以便用户更容易理解该地址所指的内容。例如： 

call   0x401136 <foo>

这里 foo 是符号的名称，应被赋予 STYLE_SYMBOL 样式。

像这样在绝对地址旁边添加符号由 DisassemblerAddressPart 类自动处理 （请参阅 反汇编器样式部分）。

GDB 使用 disassembler symbol 样式设置对具有此样式的文本进行样式化。

gdb.disassembler.STYLE_COMMENT_START ¶

此样式用于在反汇编输出中开始行注释。与其他仅应用于单个 DisassemblerTextPiece 的样式不同，注释样式是粘性的，并且会覆盖此指令中任何后续部分的样式。

这意味着，在看到 STYLE_COMMENT_START 部分后， GDB 将应用注释样式直到行尾，忽略部分内的特定样式。

GDB 使用 disassembler comment 样式设置对具有此样式的文本进行样式化。

以下函数也包含在 gdb.disassembler 模块中：

函数： register_disassembler (反汇编器, 架构) ¶

此 反汇编器 必须是 gdb.disassembler.Disassembler 或 None的子类。

可选的 架构 可以是字符串，也可以是值 None. 如果是字符串，则应该是 GDB已知的架构名称，如从 gdb.Architecture.name 返回的（请参阅 gdb.Architecture.name），或从 gdb.architecture_names 返回的（请参阅 gdb.architecture_names）。

此 反汇编器 将为 架构命名的架构安装，如果 架构 是 None, 则 反汇编器 将作为全局反汇编器安装，供所有架构使用。

GDB 仅为每个架构记录一个反汇编器，以及一个全局反汇编器。调用 register_disassembler 为某个架构或全局反汇编器注册，将替换该 架构 值的任何现有反汇编器。返回之前的反汇编器。

如果 反汇编器 是 None , 则当前为 架构 注册的任何反汇编器将被注销并返回。

当 GDB 寻找要使用的反汇编器时， GDB 首先寻找特定架构的反汇编器。如果没有注册，则 GDB 寻找全局反汇编器（使用 架构 设置为 None注册的）。只调用一个反汇编器来执行反汇编，因此，如果既有特定架构的反汇编器，又有全局反汇编器注册，则将使用特定架构的反汇编器。

GDB 分别跟踪特定架构和全局反汇编器，因此反汇编器的创建或注册顺序无关紧要；如果存在特定架构的反汇编器，总是优先使用它而不是全局反汇编器。

可以使用 maint info python - disassemblers 命令 （请参阅 maint info python - disassemblers) 查看已注册的反汇编器。

函数： builtin_disassemble (信息) ¶

此函数回调到 GDB的内置反汇编器，以反汇编由 信息, 一个实例或子类的 DisassembleInfo标识的指令。

当内置反汇编器需要读取内存时，将调用 read_memory 方法在 信息 上。通过对 DisassembleInfo 进行子类化并重写 read_memory 方法，可以拦截内置反汇编器对 read_memory 的调用，并修改返回的值。

重要的是要理解，即使 DisassembleInfo.read_memory 引发 gdb.MemoryError, 也是内部反汇编器本身向 GDB报告内存错误。原因是反汇编器可能会探测内存以查看字节是否可读；如果字节无法读取，则反汇编器可能选择不报告错误，而是反汇编其可用的字节。

如果内置反汇编器成功，则从 DisassemblerResult 返回一个 builtin_disassemble的实例，或者，如果出现问题，将引发异常。

如果 MemoryError 无法读取正确执行反汇编所需的某些内存，将引发 builtin_disassemble 。

在调用 MemoryError时引发的任何不是 read_memory的异常将通过 builtin_disassemble, 并对调用者可见。

最后，在某些情况下， GDB的内置反汇编器可能由于 MemoryError未涵盖的原因失败。在这些情况下，将引发 GdbError 。异常的内容将是一个描述反汇编器遇到的问题的字符串。

以下是一个注册全局反汇编器的示例。新的反汇编器调用内置反汇编器，然后在每行反汇编输出中添加一个注释 ## Comment: 

class ExampleDisassembler(gdb.disassembler.Disassembler):
    def __init__(self):
        super().__init__("ExampleDisassembler")

    def __call__(self, info):
        result = gdb.disassembler.builtin_disassemble(info)
        length = result.length
        text = result.string + "\t## Comment"
        return gdb.disassembler.DisassemblerResult(length, text)

gdb.disassembler.register_disassembler(ExampleDisassembler())

以下示例创建了 DisassembleInfo 的一个子类，以拦截 read_memory 调用，在 read_memory 中，从内存读取的任何字节的两个 4 位半字节将交换位置。这不是一个非常有用的调整，但作为一个示例。 

class MyInfo(gdb.disassembler.DisassembleInfo):
    def __init__(self, info):
        super().__init__(info)

    def read_memory(self, length, offset):
        buffer = super().read_memory(length, offset)
        result = bytearray()
        for b in buffer:
            v = int.from_bytes(b, 'little')
            v = (v << 4) & 0xf0 | (v >> 4)
            result.append(v)
        return memoryview(result)
