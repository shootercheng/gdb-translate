18.7 读取符号文件时的错误 ¶

在读取符号文件时， GDB 偶尔会遇到问题，例如它不识别的符号类型，或者编译器输出中已知的错误。默认情况下， GDB 不会通知你这些问题，因为它们相对常见，主要是对调试编译器的人有吸引力。如果你有兴趣查看有关构造不良的符号表的信息，你可以要求 GDB 只打印关于每种此类问题的一条消息，无论该问题发生多少次；或者你可以要求 GDB 打印更多消息，以查看问题发生的次数，使用 set complaints 命令（请参阅 可选警告和消息）。

当前打印的消息及其含义包括：

inner block not inside outer block in symbol

符号信息显示符号作用域的开始和结束位置（例如在函数或语句块的开头）。此错误表明内部作用域块未完全包含在其外部作用域块中。

GDB 通过将内部块视为与外部块具有相同的作用域来解决此问题。在错误消息中， 符号 如果外部块不是函数，可能会显示为 “(don't know)”。

block at address out of order

符号作用域块的符号信息应按地址递增的顺序出现。此错误表明并非如此。

GDB 不会解决此问题，并且在定位正在读取其符号的源文件中的符号时会遇到问题。（你通常可以通过指定 set verbose on来确定受影响的源文件。请参阅 可选警告和消息。）

bad block start address patched

符号作用域块的符号信息的起始地址小于前一个源行的地址。已知在 SunOS 4.1.1（及更早版本）的 C 编译器中会出现这种情况。

GDB 通过将符号作用域块视为从前一个源行开始来解决此问题。

bad string table offset in symbol n

符号编号 n 包含一个指向字符串表的指针，该指针大于字符串表的大小。

GDB 通过将该符号视为具有名称 foo来解决此问题，如果许多符号最终都使用此名称，可能会导致其他问题。

unknown symbol type 0xnn

符号信息包含新的数据类型， GDB 还不知道如何读取。 0xnn 是以十六进制表示的无法理解的信息的符号类型。

GDB 通过忽略此符号信息来解决此错误。这通常允许你调试程序，尽管某些符号无法访问。如果你遇到此类问题并想调试它，你可以使用自身调试 gdb ，在 complain处设置断点，然后转到函数 read_dbx_symtab 并检查 *bufp 以查看该符号。

stub type has NULL name

GDB 无法找到结构体或类的完整定义。

const/volatile indicator missing (ok if using g++ v1.x), got…

C 的符号信息++ 成员函数缺少编译器最新版本应该为其输出的一些信息。

info mismatch between compiler and debugger

GDB 无法解析编译器输出的类型规范。
