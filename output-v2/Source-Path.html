9.5 指定源目录 ¶

可执行程序有时不会记录它们所编译自的源文件的目录，而只记录文件名。即使记录了目录，在编译和调试会话之间目录也可能被移动。 GDB 有一个用于搜索源文件的目录列表；这被称为 源路径 GDB 需要源文件时，它会按列表中的顺序尝试列表中的所有目录，直到找到具有所需名称的文件。

如果上例中的文件名在可执行文件中记录为相对路径而非绝对路径，那么第一次查找将不会发生，但其余所有步骤将类似。

在 MS - DOS 和 MS - Windows 上搜索源文件时，绝对路径以驱动器盘符开头（例如 C:/project/foo.c） GDB 将在把文件名附加到 源路径中的搜索目录之前，从文件名中移除驱动器盘符；例如，如果可执行文件引用源文件 C:/project/foo.c 并且 源路径 设置为 D:/mnt/cross，那么 GDB 将在以下位置搜索源文件：

  1. C:/project/foo.c
  2. D:/mnt/cross/project/foo.c
  3. D:/mnt/cross/foo.c

请注意，可执行文件搜索路径 不 用于定位源文件。

每当你重置或重新排列源路径时， GDB 会清除它缓存的有关源文件位置以及文件中每行位置的所有信息。

当你启动 GDB时，其源路径仅包含 ‘$cdir’ 和 ‘$cwd’，顺序如此。要添加其他目录，请使用 directory 命令。

搜索路径用于查找程序源文件和 GDB 脚本文件（使用 ‘-command’ 选项和 ‘source’ 命令读取）。

除了源路径之外， GDB 还提供了一组命令来管理源路径替换规则列表。一个 替换规则 指定了如何重写程序调试信息中存储的源目录，以防在编译和调试之间源文件被移动到不同的目录。规则由两个字符串组成，第一个指定路径中需要重写的内容，第二个指定应如何重写。在 set substitute - path中，我们将这两部分分别命名为 from 和 to 。 GDB 会在源文件名的目录部分开头进行简单的字符串替换，将 from 替换为 to ，并使用该结果而不是原始文件名来查找源文件。

使用前面的示例，假设 foo - 1.0 目录树已从 /usr/src 移动到 /mnt/cross，那么你可以告诉 GDB 将所有源路径名中的 /usr/src 替换为 /mnt/cross。然后第一次查找将是 /mnt/cross/foo - 1.0/lib/foo.c ，而不是原始位置 /usr/src/foo - 1.0/lib/foo.c。要定义源路径替换规则，请使用 set substitute-path 命令（参见 set substitute - path）。

为避免意外的替换结果，只有当目录名的 from 部分以目录分隔符结尾时，规则才会应用。例如，将 /usr/source 替换为 /mnt/cross 的规则将应用于 /usr/source/foo - 1.0 ，但不会应用于 /usr/sourceware/foo - 2.0。并且由于替换仅在目录名开头应用，此规则也不会应用于 /root/usr/source/baz.c 。

在许多情况下，你可以使用 directory 命令实现相同的结果。然而， set substitute-path 在源文件组织在具有多个子目录的复杂目录树中的情况下可能更高效。使用 directory 命令时，你需要添加项目的每个子目录。如果你在保留其内部组织的同时移动了整个目录树，那么 set substitute-path 允许你通过一个命令将调试器指向所有源文件。

set substitute-path 不仅仅是一个快捷命令。源路径仅在原始位置的文件不再存在时使用。另一方面， set substitute-path 会修改调试器的行为，使其查看重写后的位置。因此，如果由于任何原因，与你的可执行文件无关的源文件位于原始位置，替换规则是将 GDB 指向新位置的唯一可用方法。

你可以通过使用 ‘ GDB --with - relocated - sources=dir’ 选项配置来配置默认的源路径替换规则。 dir 应该是 GDB配置的前缀（使用 ‘--prefix’ 或 ‘--exec - prefix’ 设置）下的一个目录名，并且如果安装的 dir 被移动到新位置，调试信息中的目录名将自动调整。如果 GDB , 带有调试信息的库或可执行文件以及相应的源代码一起移动，这将很有用。 GDB将目录
