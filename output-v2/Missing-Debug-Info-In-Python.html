23.3.2.42 Python 中缺少调试信息 ¶

当 GDB 遇到一个新的目标文件（参见 Python 中的目标文件），例如主可执行文件，或被调试程序使用的任何共享库时， GDB 将尝试为该目标文件加载相应的调试信息。 调试信息可能在目标文件本身中找到，或者在一个单独的目标文件中，而 GDB 将自动定位并加载该文件。

不过，有时 GDB 可能找不到某个目标文件的任何调试信息，在这种情况下，调试体验将受到限制。

如果 GDB 无法为某个特定的目标文件找到任何调试信息，Python 扩展就有机会介入。 Python 扩展可以通过一些特定于平台或项目的步骤潜在地定位缺失的调试信息，并将其位置告知 GDB 。 或者 Python 扩展可以向用户提供一些特定于平台或项目的建议，说明如何获取缺失的调试信息。

一个缺失调试信息的 Python 扩展由一个处理程序对象组成，该对象具有 name 和 enabled 属性，并实现了 __call__ 方法。 当 GDB 遇到一个无法找到任何调试信息的目标文件时，它会调用 __call__ 方法。 关于如何编写处理程序的详细信息可以在下面找到。

该 gdb.missing_debug 模块 ¶

GDB 附带一个 gdb.missing_debug 模块，其中包含以下类和全局函数：

类： gdb.missing_debug.MissingDebugHandler ¶

MissingDebugHandler 是一个基类，用户创建的处理程序可以从该基类派生，不过处理程序不一定要从这个类派生，只要任何用户创建的处理程序具有 name 和 enabled 属性，并实现了 __call__ 方法。

函数： MissingDebugHandler.__init__ (name) ¶

该 名称 是一个字符串，用于在某些 GDB 命令中引用这个缺失调试信息处理程序。 有效的名称由字符 [-_a-zA-Z0-9]组成，创建一个名称无效的处理程序会引发一个 ValueError 异常。

函数： MissingDebugHandler.__call__ (objfile) ¶

子类必须重写 __call__ 方法。 该 目标文件 参数将是一个 gdb.Objfile, 这是 GDB 无法找到任何调试信息的目标文件。

从 __call__ 方法返回的值指示 GDB 接下来应该做什么。 可能的返回值有：

  • None

    这表明该处理程序无法帮助处理 目标文件, GDB 应该调用任何其他已注册的处理程序。

  • True

    这表明该处理程序已将调试信息安装到 GDB 在查找单独的调试信息文件时通常期望找到的位置（参见 单独文件中的调试信息)。 GDB 将重复正常的查找过程，现在应该能找到单独的调试文件。

    如果 GDB 在第二次尝试后仍然找不到单独的调试信息文件，那么 Python 缺失调试信息处理程序不会被再次调用，这可以防止行为不当的处理程序导致 GDB 陷入循环。 GDB 将在没有任何调试信息的情况下继续处理 目标文件.

  • False

    这表明该处理程序已经对 目标文件做了它打算做的一切，但找不到单独的调试信息。 GDB 不会为 目标文件调用任何其他已注册的处理程序。 GDB 将在没有调试信息的情况下继续处理 目标文件.

  • 一个字符串

    返回的字符串应该包含一个文件名。 GDB 不会再调用任何其他已注册的处理程序，而是将从返回的文件名所标识的文件中加载调试信息。

从这个基类调用 __call__ 方法将引发一个 NotImplementedError 异常。

变量： MissingDebugHandler.name ¶

一个只读属性，是一个字符串，是传递给该处理程序的名称 __init__ 方法。

变量： MissingDebugHandler.enabled ¶

一个包含布尔值的可修改属性；当 True, 该处理程序启用，并将被 GDB使用。 当 False, 该处理程序已禁用，将不会被使用。

函数： gdb.missing_debug.register_handler (locus, handler, replace=False) ¶

向 GDB注册一个新的缺少调试信息处理程序。

处理程序 是 MissingDebugHandler的子类的实例，或者至少是一个具有与 MissingDebugHandler相同属性和方法的对象的实例。

locus 指定要将 处理程序添加到哪个处理程序列表的开头。 它可以是一个 gdb.Progspace （参见 Python 中的程序空间) 或者 None, 在这种情况下，处理程序将全局注册。 新注册的 处理程序 将在同一 locus 中的任何其他处理程序之前被调用。 同一 locus 中的两个处理程序不能有相同的名称，尝试添加一个已存在名称的处理程序会引发异常，除非 replace 为 True, 在这种情况下，旧的处理程序将被删除，新的处理程序将被添加到所选处理程序列表的开头。

GDB 首先调用当前程序空间的处理程序，然后调用全局注册的处理程序。 一旦某个处理程序返回的值不是 None, 就不会再为该 objfile 调用其他处理程序。
