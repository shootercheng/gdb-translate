C.3 编译 GDB 在另一个目录中 ¶

如果您想运行 GDB 多个主机或目标机器的版本， 您需要为每个主机和目标的组合分别编译。 gdb configure 旨在通过允许您在单独的子目录中生成每个配置，而不是在源目录中，来简化此过程。 如果您的 make 程序 处理‘VPATH’功能（GNU make 确实支持），在这些目录中的每个目录中运行 make 将构建指定的 gdb 程序。

要在单独的目录中构建 gdb ，运行 configure 时使用‘--srcdir’选项指定在哪里找到源文件。 （您还需要指定一个路径，以便从您的工作目录找到 configure 本身。 如果到 configure 的路径与‘--srcdir’的参数相同，您可以省略 ‘--srcdir’选项；默认会使用该路径。）

例如，对于版本17.2，您可以像这样在单独的目录中为Sun 4构建： GDB 

cd gdb-17.2
mkdir ../gdb-sun4
cd ../gdb-sun4
../gdb-17.2/configure
make

当 configure 使用远程源目录构建配置时，它会创建一个与源目录下的树结构相同（并使用相同名称）的二进制文件树。 在 示例中，您会在目录 libiberty.a 中找到Sun 4库 gdb-sun4/libiberty, 以及 GDB 它本身在 gdb-sun4/gdb。

确保您到 configure 脚本的路径中只有一个 实例的 gdb 。 如果您到 configure 的路径看起来像 ../gdb-17.2/gdb/configure, 您只是在配置 GDB的一个子目录，而不是整个包。 这会导致关于缺少头文件（如 bfd/bfd.h）的构建错误。

在单独的目录中构建多个 GDB 配置的一个常见原因是为交叉编译进行配置（其中 GDB 在一台机器上运行 —— 即 GDB 主机 —— 同时调试在另一台机器上运行的程序 —— 即目标 ）。 您可以通过给configure--target=目标’选项来指定交叉调试目标。 configure.

当您运行 make 来构建程序或库时，您必须在已配置的目录中运行 —— 即您调用 configure 时所在的目录（或其某个子目录）。

由 Makefile configure 在每个源目录中生成的 也会递归运行。 如果您在源目录（如 make gdb-17.2 ，您将构建所有必需的库，然后构建GDB。 （或使用‘--srcdir=目录名/gdb-17.2’配置的单独配置目录）中输入

当您在单独的目录中配置了多个主机或目标时，您可以并行在它们上运行 make （例如，如果它们在每个主机上都通过NFS挂载）；它们不会相互干扰。
