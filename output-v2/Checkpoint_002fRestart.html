4.12 设置一个 书签 以便稍后返回 ¶

在某些操作系统上4, GDB 能够保存一个 快照 程序状态的，称为一个 检查点, 并在稍后返回该状态。

返回检查点实际上会撤销自 checkpoint 保存以来程序中发生的所有事情。这包括内存、寄存器的变化，甚至（在一定范围内）系统状态的变化。实际上，这就像回到保存检查点的那一刻。

因此，如果你正在逐步调试一个程序，并且你认为自己接近出现问题的点，你可以保存一个检查点。然后，如果你不小心走过头了，错过了关键语句，你不必从程序的开头重新启动，只需返回检查点并从那里重新开始。

如果到达你认为出现错误的点需要花费大量时间或步骤，这尤其有用。

要使用 checkpoint/restart 调试方法：

checkpoint ¶

保存被调试程序当前执行状态的快照。 checkpoint 该命令不带参数，但每个检查点都会被分配一个小整数ID，类似于断点ID。

info checkpoints ¶

列出当前调试会话中已保存的检查点。对于每个检查点，将列出以下信息：

Checkpoint ID
Active state indicator
Process ID
Code Address
Source line, or label

检查点ID将显示为非负整数或形式 i.n, 其中 i 是子进程编号，一个正整数，如命令 info inferiors所示，并且 n, 一个非负整数，是该子进程的检查点编号。当只有一个子进程时，使用单个非负整数形式。 i.n 形式用于有多个子进程的情况。

活动状态指示符是一个单字母，要么是 ‘y’ 要么是 ‘n’，表示是或否。每个子进程一次只能有一个检查点处于活动状态。当前子进程中的活动检查点在该行开头也会显示一个 ‘*’。活动状态为 ‘n’ 的检查点可以使用 restart 命令切换为活动状态，或使用 delete checkpoint 命令删除。

restart checkpoint-id ¶

恢复保存为检查点编号 检查点ID. 的程序状态。所有程序变量、寄存器、栈帧等将恢复到保存检查点时的值。本质上，GDB将 “倒回时间” 到保存检查点的时刻。检查点编号 检查点ID 的指定形式与 info checkpoints 命令的输出形式相同。

请注意，断点、 GDB 变量、命令历史记录等不受恢复检查点的影响。一般来说，检查点只恢复被调试程序中的内容，而不是调试器中的内容。

delete checkpoint checkpoint-id ¶

删除由 检查点ID. 标识的先前保存的检查点。

返回先前保存的检查点将恢复被调试程序的用户状态，以及系统（操作系统）状态的一个重要子集，包括文件指针。它不会 “撤销” 文件中的数据写入，但会将文件指针倒回到先前的位置，以便可以覆盖先前写入的数据。对于以只读模式打开的文件，指针也会被恢复，以便可以再次读取先前读取的数据。

当然，已经发送到打印机（或其他外部设备）的字符无法 “取回”，从例如串行设备接收的字符可以从程序内部缓冲区中移除，但无法 “推回” 到串行管道中以便再次接收。同样，已更改文件的实际内容（目前）无法恢复。

然而，在这些限制范围内，你实际上可以将程序 “倒回” 到先前保存的时间点，并重新开始调试 —— 并且你可以改变事件的进程，以便这次调试不同的执行路径。

最后，当你返回检查点时，程序的一个内部状态会有所不同 —— 程序的进程ID。每个检查点都将有一个唯一的进程ID（或 pid），并且每个都将与程序的原始 pid. 不同。如果你程序保存了其进程ID的本地副本，这可能会潜在地带来问题。

  • 使用检查点的一个非显而易见的好处

4.12.1 使用检查点的一个非显而易见的好处 ¶

在某些系统上，如 GNU/Linux，出于安全原因，会对新进程进行地址空间随机化。如果你必须重新启动程序，这会使在绝对地址上设置断点或观察点变得困难或不可能，因为符号的绝对位置在每次执行时都会改变。

然而，检查点是进程的一个 相同 副本。因此，如果你在（例如）main函数开始时创建一个检查点，并且只是返回该检查点而不是重新启动进程，你可以避免地址空间随机化的影响，并且你的符号都将保持在相同的位置。

脚注

(4)

目前，仅支持 GNU/Linux。
