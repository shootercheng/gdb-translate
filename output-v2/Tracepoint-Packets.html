E.7 跟踪点数据包 ¶

这里我们描述用于实现跟踪点的数据包（参见 GDB 跟踪点 跟踪点)。

‘QTDP:n:地址:ena:单步执行:通过[:F帧长度][:S][:X长度,字节][-]’ ¶

创建一个新的跟踪点，编号为 n, 在 地址. 如果 ena 是 ‘E’, 则跟踪点已启用；如果是 ‘D 单步执行 给出跟踪点的单步计数，并且 通过 给出其通过计数。如果有 ‘F’ 存在，则跟踪点将是一个快速跟踪点，并且 帧长度 是目标应该复制到其他地方以为跟踪点腾出空间的字节数。如果有 ‘S’ 存在，跟踪点将是一个静态跟踪点。如果有 ‘X’ 存在，它引入一个跟踪点条件，该条件由一个十六进制长度，后跟一个逗号和十六进制编码的字节组成，方式类似于下面描述的操作编码。如果末尾的 ‘-’ 存在，后续会有更多 ‘QTDP’ 数据包将跟进以指定此跟踪点的操作。

回复：

‘OK

‘qRelocInsn’

参见 重定位指令回复数据包.

‘n:地址:[S]操作…[-]’

定义跟踪点命中时要执行的操作。 n 和 地址 必须与初始的 ‘QTDP’ 数据包用于 此跟踪点。此数据包只能在另一个以 ‘QTDP’ 结尾且末尾为 ‘-’ 的数据包之后立即发送。如果末尾的 ‘-’ 存在，则后续会有更多 ‘QTDP’ 数据包，为该跟踪点指定更多操作。

在给定跟踪点的一系列操作数据包中，最多只能有一个数据包在其第一个S’ 之前有 ‘ 操作’。如果发送了这样的数据包，它和后续的数据包定义 “单步执行时” 的操作。任何先前的数据包定义普通操作 —— 也就是跟踪点首次命中时执行的操作。如果没有操作数据包有 ‘S’，那么系列中的所有数据包都指定普通跟踪点操作。

数据包的 ‘操作…’ 部分是一系列操作，无分隔符连接。每个操作有以下形式之一：

‘R 掩码’

收集掩码中对应位被置位的寄存器， 掩码是一个十六进制数，若第 i 位被置位，则表示寄存器编号为 i 的寄存器应被收集。（最低有效位编号为零）。请注意， 掩码 可以是任意位数；它可能无法放入 32 位字中。

‘M 基寄存器,偏移量,长度’

收集从寄存器编号为 长度 的地址开始的 基寄存器 加上 偏移量 处的内存字节。如果 基寄存器 是 ‘-1’，那么该范围有一个固定地址： 偏移量 是要收集的最低字节的地址。 基寄存器, 偏移量, 和 长度 参数都是无符号十六进制值（‘-1’ 对于 基寄存器 是一个特殊情况）。

‘X 长度,表达式’

计算长度为 表达式 的 长度，并按照其指示收集内存。代理表达式 表达式 如 GDB 代理表达式机制 中所述。表达式的每个字节在数据包中编码为两位十六进制数； 长度 是表达式中的字节数（因此是数据包中十六进制数字的一半）。

‘L’

收集静态跟踪数据。

只要数据包不超过最大数据包长度（对于许多存根来说是 400 字节），任意数量的操作都可以打包在一个 ‘QTDP’ 数据包中。每个跟踪点只能有一个 ‘R’ 操作，并且它必须在任何 ‘M’ 或 ‘X’ 操作之前。‘M’ 和 ‘X’ 操作引用的任何寄存器必须由前面的 ‘R’ 操作收集。（就这些限制而言，“单步执行时” 的操作被视为附加到一个单独的跟踪点）。

回复：

‘OK’

数据包已被理解并执行。

‘qRelocInsn’

参见 重定位指令回复数据包.

‘QTDPsrc:n:地址:类型:起始位置:总长度:字节’ ¶

指定跟踪点 n 在地址 地址 处的源字符串。这对于准确重现跟踪运行开始时最初下载的跟踪点很有用。 类型 是跟踪点部分的名称，例如 ‘cond’ 表示跟踪点的条件表达式（以下列出了类型列表），而 字节 是十六进制编码的字符串。

起始位置 是 字节 在整个源字符串中的偏移量，而 总长度 是源字符串的总长度。这用于处理比单个数据包能容纳的更长的源字符串。

可用的字符串类型有 ‘at’ 表示位置，‘cond’ 表示条件，以及 ‘cmd’ 表示操作命令。 GDB 为操作列表中的每个命令发送一个单独的数据包，顺序与命令在列表中的存储顺序相同。

目标除了将源字符串作为对 ‘qTfP’/‘qTsP’ 查询数据包的回复的一部分报告回来之外，不需要对源字符串做任何处理。

虽然这个数据包是可选的，并且 GDB 只有在目标回复 ‘TracepointSource’ 时才会发送。参见 通用查询数据包, 但它使断开连接的跟踪和跟踪文件的使用变得更加容易。否则，用户必须确保在查看跟踪帧时生效的跟踪点与跟踪运行期间生效的跟踪点相同；即使是很小的差异也可能导致 ‘tdump’ 无法工作，或者找不到特定的跟踪帧。

‘QTDV:n:值:内置标志:名称’ ¶

创建一个新的跟踪状态变量，编号为 n, 初始值为 值, 这是一个 64 位有符号整数。 n 和 值 都编码为十六进制值。 GDB 可以选择不为初始值为零的情况使用此数据包；目标应该在表达式中提到跟踪状态变量时直接创建它们。 内置标志 如果跟踪状态变量是内置的，则应为 1（一），如果不是内置的，则应为 0（零）。 GDB 仅在之前的 ‘ 内置标志 被设置为 1 的 ‘qTfV’ 或 ‘qTsV’ 数据包时才将其设置为 1。 名称 的内容是跟踪状态变量的十六进制编码名称（不带前导 ‘$’）。

‘QTFrame:n’ ¶

从缓冲区中选择第 n 个跟踪点帧，并使用其中记录的寄存器和内存内容来回答来自 GDB 的后续请求数据包。

存根的成功回复表明存根已找到请求的帧。响应是一系列部分，无分隔符连接，描述我们选择的帧。每个部分有以下形式之一：

‘F f’

所选帧是跟踪帧缓冲区中的第 n 个； f 是一个十六进制数。如果 f 是 ‘-1’，则请求数据包中没有符合条件的帧。

‘T t’

所选跟踪帧记录了跟踪点编号为 t 的命中； t 是一个十六进制数。

‘QTFrame:pc:地址’

类似于 ‘QTFrame:n’，但选择当前所选帧之后第一个程序计数器（PC）为 地址 的跟踪点帧； 地址 是一个十六进制数。

‘QTFrame:tdp:t’

类似于 ‘QTFrame:n’，但选择当前所选帧之后第一个命中跟踪点 t 的跟踪点帧； t 是一个十六进制数。

‘QTFrame:range:起始地址:结束地址’

类似于 ‘QTFrame:n’，但选择当前所选帧之后第一个程序计数器（PC）在 起始地址 （包含）和 结束地址 （包含）之间的跟踪点帧； 起始地址 和 结束地址 都是十六进制数。

‘QTFrame:outside:起始地址:结束地址’

类似于 ‘QTFrame:range:起始地址:结束地址’，但选择第一个 在给定地址范围之外（不包含）的帧。

‘qTMinFTPILen’ ¶

此数据包请求可以放置快速跟踪点（参见 设置跟踪点的命令) 的最小指令长度。例如，在 32 位 x86 架构上，可以使用 4 字节跳转，但这取决于目标系统是否能够在内存的前 64K 中创建蹦床，这对于该系统来说可能行得通也可能行不通。因此，如果能够安排，此数据包的回复将是 4。

回复：

‘0’

当前最小指令长度未知。

‘长度’

最小指令长度是 长度, 其中 长度 是一个大于或等于 1 的十六进制数。回复为 1 表示可以在任何指令上放置快速跟踪点，无论其大小如何。

‘E’

发生了错误。

‘QTStart’ ¶

开始跟踪点实验。开始在跟踪帧缓冲区中收集跟踪点命中的数据。此数据包支持 ‘qRelocInsn’ 回复（参见 重定位指令回复数据包)。

‘QTStop’ ¶

结束跟踪点实验。停止收集跟踪帧。

‘QTEnable:n:地址’

在已启动的跟踪点实验中启用跟踪点 n 在地址 地址 处。如果跟踪点之前已禁用，则将恢复从该跟踪点收集数据。

‘QTDisable:n:地址’

在已启动的跟踪点实验中禁用跟踪点 n 在地址 地址 处。除非随后发出 ‘QTEnable:n:地址’，否则将不再从该跟踪点收集数据。

‘QTinit’ ¶

清除跟踪点表，并清空跟踪帧缓冲区。

‘QTro:起始范围1,结束范围1:起始范围2,结束范围2:…’ ¶

将给定的内存范围设置为 “透明”。如果这些范围在跟踪点命中时未被收集，存根将从内存的当前内容中回答对这些范围的请求。

GDB 使用此功能来标记内存的只读区域，例如包含程序代码的区域。由于这些区域永远不会改变，它们在跟踪点命中时的内容应该仍然相同，因此存根没有理由拒绝提供它们的内容。

‘QTDisconnected:值’ ¶

设置当 GDB 与目标断开连接时对跟踪运行的处理方式。 值 为 1 指示目标继续跟踪运行，而 0 告诉目标如果 GDB 不再连接则停止跟踪。

‘qTStatus’ ¶

询问存根当前是否有跟踪实验正在运行。

回复形式如下：

‘T运行状态[;字段]…’

运行状态 是一个单数字符， 1 如果跟踪当前正在运行，或 0 如果没有运行。它后面跟着用分号分隔的可选字段，代理可以使用这些字段报告额外的状态。

如果跟踪没有运行，代理可以将以下几种解释之一作为可选字段报告：

‘tnotrun:0’

尚未运行任何跟踪。

‘tstop[:文本]:0’

跟踪被用户发起的停止命令停止。可选的 文本 字段是作为停止命令一部分提供的用户字符串（例如，手动停止跟踪的原因解释）。它是十六进制编码的。

‘tfull:0’

跟踪因跟踪缓冲区已满而停止。

‘tdisconnected:0’

跟踪因 GDB 与目标断开连接而停止。

‘tpasscount:跟踪点编号’

跟踪因跟踪点 跟踪点编号 超过其通过计数而停止。

‘terror:文本:跟踪点编号’

跟踪因跟踪点 跟踪点编号 出现错误而停止。字符串 文本 可用于描述错误的性质（例如，条件表达式中的除零错误）；它是十六进制编码的。

‘tunknown:0’

跟踪因其他原因停止。

额外的可选字段提供统计和其他信息。虽然不是必需的，但它们对于监控跟踪运行进度的用户非常有用。如果跟踪已停止，并且报告了这些数字，它们必须反映刚刚停止的跟踪的状态。

‘tframes:n’

缓冲区中的跟踪帧数量。

‘tcreated:n’

运行期间创建的跟踪帧总数。如果缓冲区是循环的，这个数字可能大于跟踪帧计数。

‘tsize:n’

跟踪缓冲区的总大小，以字节为单位。

‘tfree:n’

缓冲区中仍未使用的字节数。

‘circular:n’

循环跟踪缓冲区标志的值。 1 表示跟踪缓冲区是循环的，如果需要会丢弃旧的跟踪帧以腾出空间， 0 表示跟踪缓冲区是线性的，可能会填满。

‘disconn:n’

断开连接跟踪标志的值。 1 表示 GDB 断开连接后跟踪将继续， 0 表示跟踪运行将停止。

‘qTP:跟踪点编号:地址’ ¶

向存根询问跟踪点编号为 跟踪点编号 在地址 地址.

回复：

‘V命中次数:使用量’

跟踪点在跟踪运行期间到目前为止已被命中 命中次数 次，并且在跟踪缓冲区中占用 使用量 。请注意，单步执行不被视为单独的命中，但单步执行的空间消耗会计入使用量数字中。 while-stepping

‘qTV:变量编号’ ¶

向存根询问跟踪状态变量编号为 变量编号.

回复：

‘V值’

变量的值是 值. 如果用户正在检查正在运行的目标，这将是变量的当前值；如果变量是在用户正在查看的跟踪帧中收集的，则是保存的值。请注意，多个请求可能会导致不同的回复值，例如在程序运行时请求值时。

‘U’

变量的值未知。例如，如果用户正在检查一个未收集请求变量的跟踪帧，就会出现这种情况。

‘qTfP’ ¶
‘qTsP’

这些数据包请求目标正在使用的跟踪点的数据。 GDB 发送 qTfP 以获取第一部分数据，并多次发送 qTsP 以获取额外的数据。对这些数据包的回复通常采用定义跟踪点的 QTDP 数据包的形式。（待补充详细语法）

‘qTfV’ ¶
‘qTsV’

这些数据包请求目标上的跟踪状态变量的数据。 GDB 发送 qTfV 以获取第一部分数据，并多次发送 qTsV 以获取额外的变量。对这些数据包的回复遵循定义跟踪状态变量的 QTDV 数据包的语法。

‘qTfSTM’
‘qTsSTM’

这些数据包请求目标程序中存在的静态跟踪点标记的数据。 GDB 发送 qTfSTM 以获取第一部分数据，并多次发送 qTsSTM 以获取额外的数据。对这些数据包的回复采用以下形式：

回复：

‘m 地址:标识符:额外信息’

单个标记

‘m 地址:标识符:额外信息,地址:标识符:额外信息…’

一个用逗号分隔的标记列表

‘l’

（小写字母 ‘L’）表示列表结束。

地址 以十六进制编码； 标识符 和 额外信息 是十六进制编码的字符串。

针对每个查询，目标将回复一个或多个标记的列表，用逗号分隔。 GDB 将对每个回复响应一个请求更多标记的请求（使用 ‘qs’ 形式的查询），直到目标回复 ‘l’（小写字母 l，表示 最后一个)。

‘qTSTMat:地址’

此数据包请求目标程序中在 地址 处的静态跟踪点标记的数据。对这个数据包的回复遵循列出静态跟踪点标记的 qTfSTM 和 qTsSTM 数据包的语法。

‘QTSave:文件名’ ¶

此数据包指示目标将跟踪数据保存到目标文件系统中的 文件名 文件中。 文件名 编码为十六进制字符串；文件名的解释（相对路径与绝对路径、通配符等）由目标决定。

‘qTBuffer:偏移量,长度’ ¶

返回跟踪缓冲区当前内容中从 长度 字节开始，从 偏移量 处开始的最多 l 的回复表示没有可用的字节。

‘QTBuffer:circular:值’

此数据包指示目标如果 值 为 1 则使用循环跟踪缓冲区，如果值为 0 则使用线性缓冲区。

‘QTBuffer:size:大小’

此数据包指示目标使跟踪缓冲区的大小为 大小 如果可能的话。值为 -1 告诉目标使用它首选的任何大小。

‘QTNotes:[类型:文本][;类型:文本]…’ ¶

此数据包为跟踪运行添加可选的文本注释。允许的类型包括 user, notes, 和 tstop, 文本 字段是任意字符串，十六进制编码。

E.7.1 重定位指令回复数据包 ¶

在内存中安装快速跟踪点时，目标可能需要将当前位于跟踪点地址的指令重定位到内存中的不同地址。对于大多数指令，简单的复制就足够了，但例如，隐式地将返回地址压入栈的调用指令，以及相对分支或其他与程序计数器（PC）相关的指令需要进行偏移调整，以便在不同地址执行指令的效果与在原始位置执行相同。

作为对几个跟踪点数据包的响应，目标也可能在最终结果数据包之前回复多个中间的 ‘qRelocInsn’ 请求数据包，以便 GDB 处理此重定位操作。如果一个数据包支持此机制，其文档将明确说明。例如，参见上面 ‘QTStart’ 和 ‘QTDP’ 数据包的描述。请求的格式如下：

‘qRelocInsn:从地址;到地址’

此请求 GDB 将地址 从地址 处的指令复制到地址 到地址, 可能进行调整，以便在 到地址 处执行指令的效果与在 从地址. GDB 将调整后的指令写入目标内存，从 到地址.

回复：

‘qRelocInsn:调整后大小’

通知存根重定位已完成。 调整后大小 是重定位后指令序列的字节长度。
