5.1.1 设置断点 ¶

使用以下命令设置断点 break 命令（缩写为 b)。调试器便利变量 ‘$bpnum’ 记录了你最近设置的断点编号： 

(gdb) b main
断点 1 设置在 0x11c6：文件 zeoes.c，第 24 行。
(gdb) p $bpnum
$1 = 1

一个断点可能会映射到多个代码位置，例如内联函数、Ada 泛型、C++ 模板或重载函数名。 GDB 然后在断点命令输出中指示代码位置的数量： 

(gdb) b some_func
断点 2 设置在 0x1179：some_func。（3 个位置）
(gdb) p $bpnum
$2 = 2
(gdb)

当程序在断点处停止时，便利变量 ‘$_hit_bpnum’ 和 ‘$_hit_locno’ 分别设置为遇到的断点编号和断点的代码位置编号： 

线程 1 "zeoes" 命中断点 2.1，some_func () 在 zeoes.c:8
8	  printf("some func\n");
(gdb) p $_hit_bpnum
$5 = 2
(gdb) p $_hit_locno
$6 = 1
(gdb)

请注意，‘$_hit_bpnum’ 和 ‘$bpnum’ 并不等价： ‘$_hit_bpnum’ 设置为断点编号 最后命中, 而 ‘$bpnum’ 设置为断点编号 最后设置。

如果遇到的断点只有一个代码位置，‘$_hit_locno’ 设置为 1： 

断点 1，main (argc=1, argv=0x7fffffffe018) 在 zeoes.c:24
24	  if (argc > 1)
(gdb) p $_hit_bpnum
$3 = 1
(gdb) p $_hit_locno
$4 = 1
(gdb)

‘$_hit_bpnum’ 和 ‘$_hit_locno’ 变量通常可用于断点命令列表中。（请参阅 断点命令列表）。例如，作为断点命令列表的一部分，你可以使用 disable $_hit_bpnum 完全禁用遇到的断点，或使用 disable $_hit_bpnum.$_hit_locno禁用遇到的特定断点位置。 如果断点只有一个位置，‘$_hit_locno’ 设置为 1，并且命令 disable $_hit_bpnum 和 disable $_hit_bpnum.$_hit_locno 都将禁用该断点。

你还可以定义别名，以便轻松禁用最后命中的位置或最后命中的断点： 

(gdb) alias lld = disable $_hit_bpnum.$_hit_locno
(gdb) alias lbd = disable $_hit_bpnum
break locspec

在程序中所有通过解析给定的 位置规范得到的代码位置设置断点。 位置规范 可以指定函数名、行号、指令地址等。请参阅 位置规范，了解 位置规范的各种形式。断点将在程序执行断点代码位置地址处的指令之前停止程序。

当使用允许符号重载的源语言（如 C）时，函数名可能引用多个符号，因此可能有多个断点位置。请参阅++, 对于这种情况的讨论。 模糊表达式，了解有关这种情况的讨论。

也可以插入一个断点，只有当特定线程（请参阅 特定线程的断点）、特定子进程（请参阅 特定子进程的断点）或特定任务（请参阅 Ada 任务扩展）命中该断点时，程序才会停止。

break

如果不带任何参数调用， break 将在所选栈帧中要执行的下一条指令处设置断点（请参阅 检查栈）。在除最内层之外的任何所选帧中，这将使程序在控制权返回该帧时立即停止。这类似于在所选帧内部的帧中使用 finish 命令的效果 —— 除了 finish 不会留下活动断点。如果你在最内层帧中不带参数使用 break ， GDB 将在下次到达当前位置时停止；这在循环内部可能很有用。

GDB 通常在恢复执行时会忽略断点，直到至少执行了一条指令。如果不这样做，你将无法在不禁用断点的情况下越过断点继续执行。此规则适用于程序停止时断点是否已经存在。

break … if cond

设置一个带有条件 条件的断点；每次到达断点时评估表达式 条件 ，只有当值非零（即条件 条件 评估为真）时才停止。…’ 代表上述描述的可能参数之一（或无参数），用于指定在何处设置断点。请参阅 断点条件，了解有关断点条件的更多信息。

断点可能会映射到多个位置。如果断点条件 条件 在某些但并非所有位置无效，则条件无效的位置将被禁用。例如， GDB 以下报告显示三个位置中的两个被禁用。 

(gdb) break func if a == 10
警告：无法在位置 0x11ce 验证条件，已禁用：
  当前上下文中没有符号 "a"。
警告：无法在位置 0x11b6 验证条件，已禁用：
  当前上下文中没有符号 "a"。
断点 1 设置在 0x11b6：func。（3 个位置）

由于条件而被禁用的位置在输出中用大写字母表示 N 在以下命令的输出中： info breakpoints 命令： 

(gdb) info breakpoints
编号     类型           处置 启用 地址            位置
1       断点         保留 y   <MULTIPLE>
        仅当 a == 10 时停止
1.1                         N*  0x00000000000011b6 在 ...
1.2                         y   0x00000000000011c2 在 ...
1.3                         N*  0x00000000000011ce 在 ...
(*): 此位置的断点条件无效。

如果断点条件 条件 在 所有 断点位置的上下文中都无效， GDB 将拒绝定义该断点。例如，如果变量 foo 是一个未定义的变量： 

(gdb) break func if foo
当前上下文中没有符号 "foo"。
break … -force-condition if cond

可能存在这样的情况：条件 条件 在当前所有位置都无效，但用户知道在未来某个位置它将有效；例如，由于加载了库。在这种情况下，通过使用 -force-condition 关键字在 ‘if’ 之前， GDB 可以强制使用给定的条件表达式定义断点，而不是拒绝它。 

(gdb) break func -force-condition if foo
警告：无法在位置 1 验证条件，已禁用：
  当前上下文中没有符号 "foo"。
警告：无法在位置 2 验证条件，已禁用：
  当前上下文中没有符号 "foo"。
警告：无法在位置 3 验证条件，已禁用：
  当前上下文中没有符号 "foo"。
断点 1 设置在 0x1158：test.c:18。（3 个位置）

这会导致原本要插入断点的所有当前位置都被禁用，如上面的示例所示。然而，如果存在条件有效的位置， -force-condition 关键字将不起作用。

tbreak args ¶

设置一个仅在一次停止时启用的断点。 参数 与 break 命令的参数相同，断点的设置方式也相同，但程序第一次在该断点处停止后，该断点将自动删除。请参阅 禁用断点。

hbreak args ¶

设置一个硬件辅助断点。 参数 与 break 命令的参数相同，断点的设置方式也相同，但该断点需要硬件支持，某些目标硬件可能不具备此支持。其主要用途是 EPROM/ROM 代码调试，这样你可以在不更改指令的情况下在某条指令处设置断点。这可以与 SPARClite DSU 和大多数基于 x86 的目标提供的新陷阱生成功能一起使用。当程序访问分配给调试寄存器的某些数据或指令地址时，这些目标将生成陷阱。然而，硬件断点寄存器只能设置有限数量的断点。例如，在 DSU 上，一次只能设置两个数据断点，并且 GDB 如果使用的断点超过两个，将拒绝此命令。在设置新的硬件断点之前，删除或禁用未使用的硬件断点（请参阅 禁用断点）。 请参阅 断点条件。 对于远程目标，你可以限制 GDB 将使用的硬件断点数量，请参阅 set remote hardware-breakpoint-limit。

thbreak args ¶

设置一个仅在一次停止时启用的硬件辅助断点。 参数 与 hbreak 命令的参数相同，断点的设置方式也相同。然而，与 tbreak 命令一样，程序第一次在该断点处停止后，该断点将自动删除。此外，与 hbreak 命令一样，该断点需要硬件支持，某些目标硬件可能不具备此支持。请参阅 禁用断点。 另请参阅 断点条件。

rbreak regex ¶

在所有与正则表达式 正则表达式匹配的函数上设置断点。此命令在所有匹配项上设置无条件断点，并打印出它设置的所有断点的列表。一旦设置了这些断点，它们的处理方式与使用 break 命令设置的断点相同。你可以像处理其他任何断点一样删除、禁用它们或使它们成为条件断点。

在使用不同语言的程序中， GDB 根据 ‘set language’ 的值选择打印它设置的所有断点列表的语法：使用 ‘set language auto’（请参阅 自动设置语言）意味着使用断点函数的语言，其他值意味着使用手动指定的语言（请参阅 手动设置语言）。

正则表达式的语法是与 grep等工具使用的标准语法。请注意，这与 shell 使用的语法不同，例如 foo* 匹配所有包含一个 fo 后跟零个或多个 o的函数。你提供的正则表达式前后有隐式的 .* ，因此要仅匹配以 foo开头的函数，请使用 ^foo。

在调试 C++ 程序时， rbreak 对于在不属于任何特殊类的重载函数上设置断点很有用。

该 rbreak 命令可用于在 所有 程序中的函数上设置断点，如下所示： 

(gdb) rbreak .
rbreak file:regex

如果 rbreak 在调用时带有文件名限定符，它会将搜索与给定正则表达式匹配的函数的范围限制在指定的 文件中。例如，这可用于在给定文件中的每个函数上设置断点： 

(gdb) rbreak file.c:.

分隔文件名限定符和正则表达式的冒号可以选择性地用空格包围。

info breakpoints [list…] ¶
info break [list…]

打印所有已设置且未删除的断点、观察点、跟踪点和捕获点的表格。可选参数 n 表示仅打印指定断点（或观察点、跟踪点或捕获点）的信息。 对于每个断点，将打印以下列：

断点编号
类型

断点、观察点、跟踪点或捕获点。

处置方式

断点命中时是否标记为禁用或删除。

启用或禁用

启用的断点用 ‘y’ 标记。‘n’ 标记未启用的断点。

地址

断点在程序中的位置，以内存地址表示。对于地址尚未确定的待决断点，此字段将包含 ‘<PENDING>’。此类断点在加载包含断点所引用的符号或行的共享库之前不会触发。详情请见下文。具有多个位置的断点在此字段中将有 ‘<MULTIPLE>’ —— 详情请见下文。

位置

断点在程序源代码中的位置，以文件和行号表示。对于待决断点，传递给断点命令的原始字符串将被列出，因为在将来加载适当的共享库之前无法解析它。

如果断点是有条件的，有两种评估模式：“主机”和“目标”。如果模式是“主机”，则断点条件评估由 GDB 在主机端进行。如果是“目标”，则条件由目标进行评估。 info break 命令会在受影响的断点下面的行显示条件，以及括号内的条件评估模式。

如果有断点命令，则会在后面列出。待决断点可以指定条件。在加载允许待决断点解析为有效位置的共享库之前，不会解析条件的有效性。

info break 使用断点编号 n 作为参数仅列出该断点。便利变量 $_ 和 x 命令的默认检查地址将设置为列出的最后一个断点的地址（请参阅 检查内存）。

info break 会显示断点被命中的次数。这与 ignore 命令结合使用特别有用。你可以忽略大量的断点命中，查看断点信息以了解断点被命中的次数，然后再次运行，忽略比该次数少一次的命中。这将使你快速到达该断点的最后一次命中。

对于启用计数（xref）大于 1 的断点， info break 也会显示该计数。

GDB 允许你在程序的同一位置设置任意数量的断点。这并非毫无意义。当断点是有条件的时，这甚至很有用（请参阅 断点条件）。

单个逻辑断点可能会在程序的多个代码位置设置。请参阅 位置规范，了解示例。

具有多个代码位置的断点在断点表中使用多行显示 —— 一行标题行，后面跟着每个代码位置的一行。标题行的地址列中有 ‘<MULTIPLE>’。每个代码位置行包含其代码位置的实际地址、源文件、源行和函数。代码位置的编号列形式为 断点编号。位置编号。

例如： 

编号     类型           处置 启用 地址    位置
1       断点         保留 y    <MULTIPLE>
        仅当 i==1 时停止
        断点已命中 1 次
1.1                         y    0x080486a2 在 void foo<int>() 于 t.cc:8
1.2                         y    0x080486ca 在 void foo<double>() 于 t.cc:8

你不能从断点中删除单个位置。但是，每个位置可以通过传递 断点编号。位置编号 作为参数给 enable 和 disable 命令来单独启用或禁用。也可以 enable 和 disable 一个范围的 位置编号 位置，使用一个 断点编号 和两个 位置编号，按升序排列，用连字符分隔，如 断点编号。位置编号 1-位置编号 2, 在这种情况下， GDB 将作用于该范围内的所有位置（包括两端）。禁用或启用父断点（请参阅 禁用断点）会影响属于该断点的所有位置。

父断点被禁用时仍启用的位置不会触发中断，在 y- 列中用 Enb 表示。例如： 

(gdb) info breakpoints
编号     类型           处置 启用 地址            位置
1       断点         保留 n   <MULTIPLE>
1.1                         y-  0x00000000000011b6 在 ...
1.2                         y-  0x00000000000011c2 在 ...
1.3                         n   0x00000000000011ce 在 ...

在共享库中设置断点是很常见的。共享库可以在程序执行时显式加载和卸载，并且可能会重复进行。为了支持这种用例， GDB 每当加载或卸载任何共享库时，都会更新断点位置。通常，你会在调试会话开始时，当库尚未加载且库中的符号不可用时，在共享库中设置断点。当你尝试设置断点时， GDB 会询问你是否要设置一个所谓的 待决断点 —— 地址尚未解析的断点。

程序运行后，每当加载新的共享库时， GDB 会重新评估所有断点。当新加载的共享库包含某个待决断点所引用的符号或行时，该断点将被解析并成为普通断点。当库被卸载时，所有引用其符号或源行的断点将再次变为待决断点。

这种逻辑也适用于具有多个位置的断点。例如，如果你在一个 C++ 模板函数中有一个断点，并且新加载的共享库有该模板的一个实例化，那么该断点的位置列表中将添加一个新位置。

除了地址未解析之外，待决断点与普通断点没有区别。你可以设置条件或命令，启用和禁用它们，并执行其他断点操作。

GDB 提供了一些额外的命令，用于控制当 ‘break’ 命令无法将位置规范解析到程序中的任何代码位置时的行为（请参阅 位置规范）：

set breakpoint pending auto

这是默认行为。当 GDB 无法解析位置规范时，它会询问你是否应该创建一个待决断点。

set breakpoint pending on

这表示当 GDB 无法解析位置规范时，它应该在不确认的情况下创建一个待决断点。

set breakpoint pending off

这表示不创建待决断点。如果 GDB 无法解析位置规范，它将以错误终止断点创建。此设置不会影响之前创建的任何待决断点。

show breakpoint pending

显示当前创建待决断点的行为设置。

上述设置仅影响 break 命令及其变体。一旦设置了断点，随着共享库的加载和卸载，它将自动更新。

对于某些目标， GDB 可以根据断点地址是只读还是读写，自动决定使用硬件还是软件断点。这适用于使用 break 命令设置的断点，以及由 next 和 finish等命令设置的内部断点。对于使用 hbreak, GDB 将始终使用硬件断点。

你可以使用以下命令控制这种自动行为：

set breakpoint auto-hw on

这是默认行为。当 GDB 设置断点时，它将尝试使用目标内存映射来决定使用软件还是硬件断点。

set breakpoint auto-hw off

这表示 GDB 不应自动选择断点类型。如果目标提供了内存映射， GDB 在尝试在只读地址设置软件断点时会发出警告。

GDB 通常通过将断点地址处的程序代码替换为一条特殊指令来实现断点，当执行该指令时，将控制权交给调试器。默认情况下，只有在程序恢复执行时才会修改程序代码。一旦程序停止， GDB 将恢复原始指令。这种行为可以防止在 GDB 突然断开连接时，目标中仍插入有断点。然而，对于慢速远程目标，插入和移除断点会降低性能。可以使用以下命令控制这种行为：

set breakpoint always-inserted off

所有断点，包括用户新添加的断点，仅在目标恢复执行时插入到目标中。目标停止时，所有断点将从目标中移除。这是默认模式。

set breakpoint always-inserted on

导致所有断点始终插入到目标中。如果用户添加新的断点或更改现有断点，目标中的断点将立即更新。只有当断点本身被删除时，才会从目标中移除断点。

GDB 通过在断点触发时评估这些条件来处理条件断点。如果条件为真，则被调试的进程停止，否则进程继续执行。

如果目标支持在其端评估条件， GDB 可以将断点及其条件下载到目标中。

可以通过以下命令控制此功能：

set breakpoint condition-evaluation host

此选项命令 GDB 在主机端评估断点条件。无条件断点将发送到目标，目标接收触发并将其报告回 GDB 进行条件评估。这是标准评估模式。

set breakpoint condition-evaluation target

此选项命令 GDB 在插入断点条件时将其下载到目标中。目标负责评估条件表达式，并在条件为真时将断点停止事件报告回 GDB 。由于目标端评估的限制，某些条件无法在那里评估，例如依赖于仅主机知道的本地数据的条件。示例包括涉及便利变量的条件表达式、代理表达式解析器无法处理的复杂类型以及太长而无法发送到目标的表达式，特别是当目标是远程系统时。在这些情况下，条件将由 GDB评估。

set breakpoint condition-evaluation auto

这是默认模式。如果目标支持在其端评估断点条件， GDB 将断点条件下载到目标（受前面提到的限制）。如果目标不支持断点条件评估，则 GDB 将回退到在主机端评估所有这些条件。

GDB 本身有时会为特殊目的在程序中设置断点，例如正确处理 longjmp （在 C 程序中）。这些内部断点被分配负编号，从 -1开始；‘info breakpoints’ 不会显示它们。你可以使用 GDB 维护命令 ‘maint info breakpoints’（请参阅 maint info breakpoints）查看这些断点。
