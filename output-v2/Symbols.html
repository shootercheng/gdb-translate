whatis[/flags] [arg] ¶

打印 参数, 它可以是一个表达式或数据类型的名称。如果没有参数，则打印 $, 值历史记录中的最后一个值的数据类型。

如果 参数 是一个表达式（请参阅 表达式），它实际上不会被求值，并且其中的任何有副作用的操作（如赋值或函数调用）都不会发生。

如果 参数 是一个变量或表达式， whatis 则打印其在源代码中使用的字面类型。如果该类型是使用 typedef, whatis 将 不 打印 typedef下面的数据类型。如果变量或表达式的类型是复合数据类型，如 struct 或 class, whatis 永远不会打印它们的字段或方法。它只是打印 struct/class 名称（也称为其 标签）。如果你想查看这种复合数据类型的成员，请使用 ptype.

如果 参数 是使用 typedef, whatis 定义的类型名称，它将展开 该类型的仅一层。展开意味着 typedef将显示在 whatis 中使用的底层类型 typedef 声明的 参数。但是，如果该底层类型也是 typedef, whatis 将不会展开它。

对于C代码，类型名称也可能具有 ‘类 类名’, ‘结构体 结构体标签’, ‘联合体 联合体标签’ 或 ‘枚举 枚举标签’。

标志 可用于修改类型的显示方式。可用的标志有：

r

以 “原始” 形式显示。通常， GDB 在打印类的成员时会替换类中定义的模板参数和 typedef。 /r 标志禁用此功能。

m

不打印类中定义的方法。

M

打印类中定义的方法。这是默认设置，但如果使用 set print type methods. 更改了默认设置，则该标志存在。

t

不打印类中定义的 typedef。请注意，这控制的是 typedef 定义本身是否被打印，而不是在打印其他类型时是否替换 typedef 名称。

T

打印类中定义的 typedef。这是默认设置，但如果使用 set print type typedefs. 更改了默认设置，则该标志存在。

o

打印结构体中字段的偏移量和大小，类似于 pahole 工具的做法。此选项意味着 /tm 标志。

x

在打印结构体中字段的偏移量和大小时使用十六进制表示法。

d

在打印结构体中字段的偏移量和大小时使用十进制表示法。

例如，给定以下声明： 

struct tuv
{
  int a1;
  char *a2;
  int a3;
};

struct xyz
{
  int f1;
  char f2;
  void *f3;
  struct tuv f4;
};

union qwe
{
  struct tuv fff1;
  struct xyz fff2;
};

struct tyu
{
  int a1 : 1;
  int a2 : 3;
  int a3 : 23;
  char a4 : 2;
  int64_t a5;
  int a6 : 5;
  int64_t a7 : 3;
};

执行 ptype /o struct tuv 命令将打印： 

(gdb) ptype /o struct tuv
/* 偏移量      |    大小 */  类型 = struct tuv {
/*      0      |       4 */    int a1;
/* XXX  4 字节空洞      */
/*      8      |       8 */    char *a2;
/*     16      |       4 */    int a3;

                               /* 总大小（字节）:   24 */
                             }

注意注释第一列的格式。在那里，你可以找到由 ‘|’ 字符分隔的两部分： 偏移量, 它指示字段在结构体内部的位置，以字节为单位，以及 字段的大小 。另一个有趣的行是结构体中 空洞 的标记，表明可以通过重新组织其字段来压缩结构体并使其使用更少的空间。

也可以打印联合体内部的偏移量： 

(gdb) ptype /o union qwe
/* 偏移量      |    大小 */  类型 = union qwe {
/*                    24 */    struct tuv {
/*      0      |       4 */        int a1;
/* XXX  4 字节空洞      */
/*      8      |       8 */        char *a2;
/*     16      |       4 */        int a3;

                                   /* 总大小（字节）:   24 */
                               } fff1;
/*                    40 */    struct xyz {
/*      0      |       4 */        int f1;
/*      4      |       1 */        char f2;
/* XXX  3 字节空洞      */
/*      8      |       8 */        void *f3;
/*     16      |      24 */        struct tuv {
/*     16      |       4 */            int a1;
/* XXX  4 字节空洞      */
/*     24      |       8 */            char *a2;
/*     32      |       4 */            int a3;

                                       /* 总大小（字节）:   24 */
                                   } f4;

                                   /* 总大小（字节）:   40 */
                               } fff2;

                               /* 总大小（字节）:   40 */
                             }

在这种情况下，由于 struct tuv 和 struct xyz 占用相同的空间（因为我们处理的是联合体），所以不会为它们打印偏移量。但是，你仍然可以检查这些结构体字段的偏移量。

另一个有用的场景是打印包含位字段的结构体的偏移量： 

(gdb) ptype /o struct tyu
/* 偏移量      |    大小 */  类型 = struct tyu {
/*      0:31   |       4 */    int a1 : 1;
/*      0:28   |       4 */    int a2 : 3;
/*      0: 5   |       4 */    int a3 : 23;
/*      3: 3   |       1 */    signed char a4 : 2;
/* XXX  3 位空洞       */
/* XXX  4 字节空洞      */
/*      8      |       8 */    int64_t a5;
/*     16: 0   |       4 */    int a6 : 5;
/*     16: 5   |       8 */    int64_t a7 : 3;
/* XXX  7 字节填充   */

                               /* 总大小（字节）:   24 */
                             }

注意现在偏移量信息如何扩展到还包括位字段的第一位。

ptype[/flags] [arg] ¶

ptype 接受与 whatis, 相同的参数，但打印类型的详细描述，而不仅仅是类型的名称。请参阅 表达式.

与 whatis, 相反 ptype 总是展开其参数声明中的任何 typedef，无论参数是变量、表达式还是数据类型。这意味着 ptype 变量或表达式的类型不会按源代码中出现的那样逐字打印 — 使用 whatis 来实现这一点。 typedef指针或引用目标处的 typedef也会被展开。只有 class typedef字段、方法和内部 struct的 class和 union即使使用 ptype. 也不会被展开。

例如，对于这个变量声明： 

typedef double real_t;
struct complex { real_t real; double imag; };
typedef struct complex complex_t;
complex_t var;
real_t *real_pointer_var;

这两个命令给出以下输出： 

(gdb) whatis var
type = complex_t
(gdb) ptype var
type = struct complex {
    real_t real;
    double imag;
}
(gdb) whatis complex_t
type = struct complex
(gdb) whatis struct complex
type = struct complex
(gdb) ptype struct complex
type = struct complex {
    real_t real;
    double imag;
}
(gdb) whatis real_pointer_var
type = real_t *
(gdb) ptype real_pointer_var
type = double *

与 whatis, 一样，使用 ptype 没有参数时指的是 $, 值历史记录中的最后一个值的类型。

有时，程序使用不透明数据类型或复杂数据结构的不完整规范。如果程序中包含的调试信息不允许 GDB 显示数据类型的完整声明，它将显示 ‘<不完整类型>’。例如，给定这些声明： 

    struct foo;
    struct foo *fooptr;

但没有 struct foo 本身的定义 GDB 将显示： 

  (gdb) ptype foo
  $1 = <不完整类型>

“不完整类型” 是C术语，用于表示未完全指定的数据类型。

其他时候，程序中包含的调试信息中完全没有关于变量类型的信息。这最常发生在定义变量的程序或库根本不包含调试信息时。 GDB 通过检查链接器/加载器符号表（例如，ELF动态符号表）知道变量存在，但这些符号不包含类型信息。检查没有类型信息的（全局）变量的类型会显示： GDB has no type information shows: 

  (gdb) ptype var
  type = <数据变量，无调试信息>

请参阅 无调试信息的变量, 了解如何打印此类变量的值。

info types [-q] [regexp] ¶

打印所有名称与正则表达式 正则表达式 匹配的类型的简要描述（如果你不提供参数，则打印程序中的所有类型）。每个完整的类型名就像它是完整的一行一样进行匹配；因此，‘i type value’ 提供程序中所有名称包含字符串 value, 的类型的信息，但 ‘i type ^value$’ 仅提供名称完全为 value. 的类型的信息。

在使用不同语言的程序中 GDB 根据set language’ 的值选择打印类型描述的语法：使用 ‘set language auto’ (请参阅 自动设置语言) 意味着使用类型的语言，其他值意味着使用手动指定的语言（请参阅 手动设置语言).

此命令与 ptype 在两个方面不同：首先，像 whatis, 一样，它不打印详细描述；其次，它列出类型定义所在的所有源文件和行号。

‘into types’ 的输出前面有一个标题行，描述正在列出的类型。可选标志 ‘-q’, 它代表 ‘安静’, 禁用打印此标题信息。

info source ¶

显示有关当前源文件的信息 — 即包含当前执行点的函数的源文件的信息：

  • 源文件的名称及其所在的目录
  • 它编译时所在的目录
  • 它的行数
  • 它所使用的编程语言
  • 如果调试信息提供了，编译该文件的程序（可能包括，例如，编译器版本和命令行参数）

  • 调试信息是否包含有关预处理器宏的信息。
info sources [-dirname | -basename] [--] [regexp] ¶

没有选项时，‘info sources’ 打印程序中所有有调试信息的源文件的名称。源文件是根据当前加载到 GDB. 中的目标文件（可执行文件和库）列表呈现的。对于每个目标文件，列出所有关联的源文件。

每个源文件对于每个目标文件只会打印一次，但如果一个源文件是多个目标文件的一部分，则它可能会在输出中重复出现。

如果提供了可选的 正则表达式 ，则只打印与正则表达式匹配的源文件。匹配是区分大小写的，除非在不区分大小写的文件系统（例如，MS-Windows）的操作系统上。‘--’ 可在 正则表达式 之前使用，以防止 GDB 将 正则表达式 解释为命令选项（例如，如果 正则表达式 以 ‘-’ 开头）。

默认情况下， 正则表达式 用于在文件名的任何位置进行匹配。如果 -dirname, 则只显示目录名与 正则表达式 匹配的文件。如果 -basename, 则只显示基本文件名与 正则表达式 匹配的文件。

目标文件可能会在列表中打印出来，但没有关联的源文件。这可能发生在没有源文件与 正则表达式匹配，或者目标文件是在没有调试信息的情况下编译的，因此 GDB 无法找到任何源文件名。

info functions [-q] [-n] ¶

打印所有已定义函数的名称和数据类型。与 ‘info types’ 类似，此命令按源文件对输出进行分组，并为每个函数定义标注其源行号。

在使用不同语言的程序中 GDB 根据set language’ 的值选择打印函数名称和类型的语法：使用 ‘set language auto’ (请参阅 自动设置语言) 意味着使用函数的语言，其他值意味着使用手动指定的语言（请参阅 手动设置语言).

‘-n’ 标志排除 非调试符号 从结果中。非调试符号是来自可执行文件符号表的符号，而不是来自与可执行文件关联的调试信息（例如，DWARF）。

可选标志 ‘-q’, 它代表 ‘安静’, 禁用打印标题信息和解释为什么没有打印函数的消息。

与 ‘info functions’ 类似，但只打印使用提供的正则表达式选择的函数的名称和数据类型。

如果 正则表达式 被提供，则只打印名称与正则表达式 正则表达式. 匹配的函数。因此，‘info fun step’ 查找名称包含 step的所有函数；‘info fun ^step’ 查找名称以 step. 开头的函数。如果函数名称包含与正则表达式语言冲突的字符（例如operator*()’），可以使用反斜杠进行转义。

如果 类型正则表达式 被提供，则只打印其类型（由 whatis 命令打印）与正则表达式 类型正则表达式. 匹配的函数。如果 类型正则表达式 包含空格，则应将其用引号括起来。如果需要，使用反斜杠转义特殊字符或引号的含义。因此，‘info fun -t '^int ('’ 查找返回整数的函数；‘info fun -t '(.*int.*'’ 查找具有包含 int 的参数类型的函数；‘info fun -t '^int (' ^step’ 查找名称以 step 开头且返回 int 的函数。

set print symbol-loading brief
set print symbol-loading off

该 set print symbol-loading 命令允许你控制 GDB 加载符号信息时消息的打印。默认情况下，会为可执行文件和每个共享库打印一条消息，通常这是你想要的。但是，在调试具有大量共享库的应用程序时，这些消息可能会很烦人。当设置为 brief 时，会为每个可执行文件打印一条消息，并且当 GDB 一次加载一组共享库时，无论共享库的数量如何，它只会打印一条消息。当设置为 off 时，不打印任何消息。

show print symbol-loading ¶

显示从键盘输入的 GDB 命令导致加载符号信息时是否会打印消息。

maint print symbols [-pc address] [filename] ¶
maint print symbols [-objfile objfile] [-source source] [--] [filename]
maint print psymbols [-objfile objfile] [-pc address] [--] [filename]
maint print psymbols [-objfile objfile] [-source source] [--] [filename]
maint print msymbols [-objfile objfile] [--] [filename]

将调试符号数据转储到文件 文件名 中，或者如果 文件名 未指定，则转储到终端。如果 -objfile objfile 被指定，则只转储该目标文件的符号。如果 -pc address 被指定，则只转储代码位于该地址的文件的符号。请注意， 地址 可能是像 main. 这样的符号。如果 -source source 被指定，则只转储该源文件的符号。

这些命令用于调试 GDB 符号读取代码。这些命令不会修改内部 GDB 状态，因此 ‘maint print symbols’ 只会打印已经展开的符号表中的符号。你可以使用命令 info sources 找出这些文件是哪些。如果你使用 ‘maint print psymbols’ 代替，转储将显示 GDB 只部分了解的符号的信息 — 即 GDB 已经略读但尚未完全读取的文件中定义的符号。最后，‘maint print msymbols’ 只是转储 “最小符号”，例如 “ELF符号”。

请参阅 指定文件的命令, 了解 GDB 如何读取符号（在 symbol-file. 的描述中）。

maint info symtabs [ regexp ] ¶
maint info psymtabs [ regexp ]

列出 struct symtab 或 struct partial_symtab 名称与 正则表达式匹配的结构。如果 正则表达式 未给出，则列出所有结构。输出包括你可以复制到 GDB 调试中以更详细地检查特定结构的表达式。例如： 

(gdb) maint info psymtabs dwarf2read
{ objfile /home/gnu/build/gdb/gdb
  ((struct objfile *) 0x82e69d0)
  { psymtab /home/gnu/src/gdb/dwarf2read.c
    ((struct partial_symtab *) 0x8474b10)
    readin no
    fullname (null)
    text addresses 0x814d3c8 -- 0x8158074
    globals (* (struct partial_symbol **) 0x8507a08 @ 9)
    statics (* (struct partial_symbol **) 0x40e95b78 @ 2882)
    dependencies (none)
  }
}
(gdb) maint info symtabs
(gdb)

我们看到有一个部分符号表，其文件名包含字符串 ‘dwarf2read’，属于 ‘gdb’ 可执行文件；并且我们看到 GDB 还没有读取任何符号表。如果我们在一个函数上设置断点，这将导致 GDB 读取包含该函数的编译单元的符号表： 

(gdb) break dwarf2_psymtab_to_symtab
Breakpoint 1 at 0x814e5da: file /home/gnu/src/gdb/dwarf2read.c,
line 1574.
(gdb) maint info symtabs
{ objfile /home/gnu/build/gdb/gdb
  ((struct objfile *) 0x82e69d0)
  { symtab /home/gnu/src/gdb/dwarf2read.c
    ((struct symtab *) 0x86c1f38)
    dirname (null)
    fullname (null)
    blockvector ((struct blockvector *) 0x86c1bd0) (primary)
    linetable ((struct linetable *) 0x8370fa0)
    debugformat DWARF 2
  }
}
(gdb)
maint info line-table [ regexp ] ¶

列出所有 struct linetable 名称与 struct symtab 正则表达式 匹配的实例的. 如果 正则表达式 未给出，则列出所有 struct linetable 的 struct symtab. 例如： 

(gdb) maint info line-table
objfile: /home/gnu/build/a.out ((struct objfile *) 0x6120000e0d40)
compunit_symtab: simple.cpp ((struct compunit_symtab *) 0x6210000ff450)
symtab: /home/gnu/src/simple.cpp ((struct symtab *) 0x6210000ff4d0)
linetable: ((struct linetable *) 0x62100012b760):
INDEX  LINE   ADDRESS            IS-STMT PROLOGUE-END EPILOGUE-BEGIN
0      3      0x0000000000401110 Y
1      4      0x0000000000401114 Y       Y            Y
2      9      0x0000000000401120 Y
3      10     0x0000000000401124 Y       Y
4      10     0x0000000000401129 Y                    Y
5      15     0x0000000000401130 Y
6      16     0x0000000000401134 Y       Y
7      16     0x0000000000401139
8      21     0x0000000000401140 Y                    Y
9      22     0x000000000040114f Y       Y
10     22     0x0000000000401154                      Y
11     END    0x000000000040115a Y

‘IS-STMT’ 列指示该地址是否是表示一行或一条语句的推荐断点位置。‘PROLOGUE-END’ 列指示给定地址是在函数序言之后的第一条指令处设置断点的合适位置。‘EPILOGUE-BEGIN’ 列指示给定地址标记了块的帧被销毁的点，使得局部变量难以或无法找到。

set always-read-ctf [on|off] ¶
show always-read-ctf

关闭时，只有在没有DWARF调试信息时才会读取CTF调试信息。打开时，无论是否有DWARF调试信息，都会读取CTF调试信息。默认值为关闭。

maint set symbol-cache-size size ¶

将符号缓存的大小设置为 大小. 默认大小旨在足以调试大多数应用程序。此选项存在是为了允许尝试不同的大小。

maint show symbol-cache-size ¶

显示符号缓存的大小。

maint print symbol-cache ¶

打印符号缓存的内容。这在调试符号缓存问题时很有用。

maint print symbol-cache-statistics ¶

打印符号缓存使用统计信息。这有助于确定缓存的利用情况。

maint flush symbol-cache ¶
maint flush-symbol-cache

刷新符号缓存的内容，移除所有条目。此命令在调试符号缓存时很有用。在收集性能数据时也很有用。该命令 maint flush-symbol-cache 已被弃用，建议使用 maint flush symbol-cache..

maint set ignore-prologue-end-flag [on|off] ¶

启用或禁用使用行表中的 ‘PROLOGUE-END’ 标志。当 ‘off’（默认）时， GDB 使用 ‘PROLOGUE-END’ 标志在函数序言结束后放置断点。当 ‘on’ 时， GDB 忽略该标志并依赖序言分析器跳过函数序言。

maint show ignore-prologue-end-flag ¶

显示 GDB 是否会忽略 ‘PROLOGUE-END’ 标志。
