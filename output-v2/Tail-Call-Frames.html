11.2 尾调用帧 ¶

函数 B 可以在其最后一条语句中调用函数 C 。在未优化的编译中， C 的调用之后紧接着是 B 代码末尾的返回指令。优化编译器可以将函数 B 中的调用和返回替换为跳转到函数 C 的一条指令。这种使用跳转指令的方式称为 尾调用.

在函数执行期间 C，函数调用栈帧中不会有它是从 B尾调用而来的迹象。如果函数 A 通常调用函数 B 又尾调用函数 C，那么 GDB 会将 A 视为 C的调用者。然而，在某些情况下 GDB 可以确定 C 是从 B尾调用而来的，然后它会为其创建一个虚拟的调用帧，返回地址的设置就好像 B 正常调用 C 一样。

目前，此功能仅由DWARF 2调试格式支持，并且编译器必须生成‘DW_TAG_call_site’标签。使用 GCC时，你需要在编译期间指定 -O -g 以获取此信息。

info frame 命令（请参阅 关于帧的信息)将通过文本指示尾调用帧的类型，例如在这个示例 tail call frame 输出中： GDB 

(gdb) x/i $pc - 2
   0x40066b <b(int, double)+11>: jmp 0x400640 <c(int, double)>
(gdb) info frame
Stack level 1, frame at 0x7fffffffda30:
 rip = 0x40066d in b (amd64-entry-value.cc:59); saved rip 0x4004c5
 tail call frame, caller of frame at 0x7fffffffda30
 source language c++.
 Arglist at unknown address.
 Locals at unknown address, Previous frame's sp is 0x7fffffffda30

对所有可能的代码路径执行的检测可能会发现它们存在歧义。没有存储执行历史记录（可能 逆向运行程序 永远不会用于此目的），并且最后已知的调用者可能通过多种不同的跳转序列到达了已知的被调用者。在这种情况下 GDB 仍然会尝试显示至少所有明确的顶部尾调用者和所有明确的底部尾被调用者（如果有的话）。

set debug entry-values ¶

当设置为开启时，启用对函数入口处的帧参数值和尾调用的分析消息的打印。它将显示它所考虑的所有可能的有效尾调用代码路径。它还将打印这些路径与最终明确的（可能是部分的甚至是空的）代码路径结果的交集。

show debug entry-values ¶

显示函数入口处的帧参数值和尾调用的分析消息打印的当前状态。

尾调用的分析消息例如可以显示为什么函数 c 的虚拟尾调用帧未被识别（由于变量的间接引用 x）： 

static void __attribute__((noinline, noclone)) c (void);
void (*x) (void) = c;
static void __attribute__((noinline, noclone)) a (void) { x++; }
static void __attribute__((noinline, noclone)) c (void) { a (); }
int main (void) { x (); return 0; }

Breakpoint 1, DW_OP_entry_value resolving cannot find
DW_TAG_call_site 0x40039a in main
a () at t.c:3
3	static void __attribute__((noinline, noclone)) a (void) { x++; }
(gdb) bt
#0  a () at t.c:3
#1  0x000000000040039a in main () at t.c:5

另一种可能性是模糊的虚拟尾调用帧解析： 

int i;
static void __attribute__((noinline, noclone)) f (void) { i++; }
static void __attribute__((noinline, noclone)) e (void) { f (); }
static void __attribute__((noinline, noclone)) d (void) { f (); }
static void __attribute__((noinline, noclone)) c (void) { d (); }
static void __attribute__((noinline, noclone)) b (void)
{ if (i) c (); else e (); }
static void __attribute__((noinline, noclone)) a (void) { b (); }
int main (void) { a (); return 0; }

tailcall: initial: 0x4004d2(a) 0x4004ce(b) 0x4004b2(c) 0x4004a2(d)
tailcall: compare: 0x4004d2(a) 0x4004cc(b) 0x400492(e)
tailcall: reduced: 0x4004d2(a) |
(gdb) bt
#0  f () at t.c:2
#1  0x00000000004004d2 in a () at t.c:8
#2  0x0000000000400395 in main () at t.c:9

帧 #0 和 #2 是真实的，#1 是虚拟尾调用帧。代码可能有执行路径 main→a→b→c→d→f 或 main→a→b→e→f, GDB 无法从程序状态中确定是哪一个。

initial: 状态显示了一些随机的可能调用序列 GDB 已经找到。然后它又找到了另一个可能的调用序列 - 那个序列以 compare:为前缀。这两个序列的明确交集被打印为 reduced: 调用序列。只要还有任何明确的序列项，这个序列可能还有很多 compare: 和 reduced: 语句。

对于函数 b 的帧，在两种情况下都有不同的可能 $pc 值（0x4004cc 或 0x4004ce），因此这个帧也是模糊的。唯一明确的帧是函数 a的帧，因此会将这个帧显示给用户，而模糊的帧则会被省略。

函数入口处的帧参数值打印可能失败的原因也可能有： 

int v;
static void __attribute__((noinline, noclone)) c (int i) { v++; }
static void __attribute__((noinline, noclone)) a (int i);
static void __attribute__((noinline, noclone)) b (int i) { a (i); }
static void __attribute__((noinline, noclone)) a (int i)
{ if (i) b (i - 1); else c (0); }
int main (void) { a (5); return 0; }

(gdb) bt
#0  c (i=i@entry=0) at t.c:2
#1  0x0000000000400428 in a (DW_OP_entry_value resolving has found
function "a" at 0x400420 can call itself via tail calls
i=<optimized out>) at t.c:6
#2  0x000000000040036e in main () at t.c:7

GDB 无法从程序状态中确定函数 a 是否以及多少次通过函数 b进行了自调用，因为这些调用将是尾调用。这样的尾调用会修改 i 变量，因此 GDB 不能确定它所知道的值是否正确 - GDB 会打印 <optimized out> 代替。
