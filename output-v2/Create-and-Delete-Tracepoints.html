13.1.1 创建和删除跟踪点 ¶

trace locspec ¶

该 trace 命令与 break 命令非常相似。 其参数 locspec 可以是任何有效的位置规范。 请参阅 位置规范。 该 trace 命令定义一个跟踪点， 这是目标程序中的一个点，调试器将在此处短暂停止， 收集一些数据，然后允许程序继续。 如果远程存根支持 ‘InstallInTrace’ 功能（请参阅 在跟踪中安装跟踪点），设置跟踪点或更改其动作将立即生效。 如果远程存根不支持 ‘InstallInTrace’ 功能，所有这些更改直到下一次 tstart 命令才会生效，并且一旦跟踪实验正在运行，在下次跟踪实验开始之前，进一步的更改将不会有任何效果。 此外， GDB 支持 待决跟踪点——其地址尚未解析的跟踪点。 （这类似于待决断点。） 待决跟踪点在解析之前不会下载到目标并安装。 待决跟踪点的解析需要 GDB 支持 —— 在使用远程目标进行调试时，并且 GDB 与远程存根断开连接（请参阅 断开连接的跟踪），当 GDB 断开连接时，待决跟踪点无法解析（并下载到远程存根）。

以下是使用 trace 命令的一些示例： 

(gdb) trace foo.c:121    // 一个源文件和行号

(gdb) trace +2           // 向前 2 行

(gdb) trace my_function  // 函数的第一行源代码

(gdb) trace *my_function // 函数的确切起始地址

(gdb) trace *0x2117c4    // 一个地址

您可以将 trace 缩写为 tr。

trace locspec if cond

设置带有条件 cond的跟踪点；每次到达跟踪点时计算表达式 cond ，并且仅当值非零 —— 即当 cond 计算结果为真时收集数据。 请参阅 跟踪点条件，以获取有关跟踪点条件的更多信息。

ftrace locspec [ if cond ] ¶

该 ftrace 命令设置一个快速跟踪点。 对于支持它们的目标，快速跟踪点将使用更高效但可能不太通用的技术来触发数据收集，例如使用跳转指令而不是陷阱，或者某种硬件支持。 可能无法在所需位置创建快速跟踪点，在这种情况下，命令将退出并给出解释性消息。

GDB 处理 ftrace 的参数与处理 trace的参数完全相同。

在 32 位 x86 架构系统上，快速跟踪点通常需要放置在 5 字节或更长的指令处，但如果目标程序的低 64K 内存可用于安装跳板，则可以放置在 4 字节指令处。 一些 Unix 类型的系统，例如 GNU/Linux，将低地址排除在程序的地址空间之外；但例如对于 Linux 内核，可以通过执行 GDB 命令来设置 sysctl 内核参数，如下所示 mmap_min_addr 

sudo sysctl -w vm.mmap_min_addr=32768

这将低地址设置为 32K，为跳板留下了足够的空间。 最小地址应设置为页面边界。

strace [locspec | -m marker] [ if cond ] ¶

该 strace 命令设置一个静态跟踪点。 对于支持它的目标，设置静态跟踪点会探测在解析 locspec后得到的代码位置处找到的静态插桩点或标记。 可能无法在所需的代码位置设置静态跟踪点，在这种情况下，命令将退出并给出解释性消息。

GDB 处理 strace 的参数与处理 trace的参数完全相同，此外用户还可以指定 -m marker 而不是位置规范。 这将探测由 marker 字符串标识符标识的标记。 此标识符取决于您的程序正在使用的静态跟踪点后端库。 您可以在ID字段中找到所有标记标识符。 info static-tracepoint-markers 命令输出。 请参阅 列出静态跟踪点标记。 例如，在以下使用 UST 跟踪引擎的小程序中： 

main ()
{
  trace_mark(ust, bar33, "str %s", "FOOBAZ");
}

标记 ID 由调用的前两个参数用斜杠连接组成，即： trace_mark 

(gdb) info static-tracepoint-markers
Cnt Enb ID         Address            What
1   n   ust/bar33  0x0000000000400ddc in main at stexample.c:22
         Data: "str %s"
[etc...]

因此您可以使用以下命令探测上述标记： 

(gdb) strace -m ust/bar33

静态跟踪点接受一个额外的收集动作 —— collect $_sdata。 这将收集在探测点调用跟踪库时传递的任意用户数据。 在上面的 UST 示例中，您会看到 trace_mark 的第三个参数是一个类似 printf 的格式字符串。 用户数据就是将该格式字符串应用于后续参数的结果。 请注意 info static-tracepoint-markers 命令输出在 ‘数据：字段。

在分析跟踪缓冲区时，您可以通过像打印任何其他可用变量一样打印 $_sdata 变量来检查此数据。 GDB请参阅 跟踪点动作列表。

便利变量 $tpnum 记录最近设置的跟踪点的编号。

delete tracepoint [num] ¶

永久删除一个或多个跟踪点。如果没有参数，默认是删除所有跟踪点。请注意，常规 delete 命令也可以删除跟踪点。

示例： 

(gdb) delete trace 1 2 3 // 删除三个跟踪点

(gdb) delete trace       // 删除所有跟踪点

您可以将此命令缩写为 del tr。
