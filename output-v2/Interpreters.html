24 命令解释器 ¶

GDB 支持多个命令解释器，以及一些命令基础设施，允许用户或用户界面编写者在解释器之间切换，或在其他解释器中运行命令。

GDB 目前支持两种命令解释器，控制台解释器（有时也称为命令行解释器或 CLI) 以及机器接口解释器（或 GDB/MI）。本手册详细描述了这两种接口。

默认情况下， GDB 将以控制台解释器启动。 但是，用户可以通过指定 GDB 使用另一个 解释器启动 -i 或 --interpreter 启动选项。已定义的解释器包括：

console ¶

传统的控制台或命令行解释器。这是使用 GDB时最常用的解释器。如果在运行时未指定解释器， GDB 将使用此解释器。

dap ¶

当 GDB 以支持 Python 的方式构建时，它还支持 调试器适配器协议。此协议可由调试器 GUI 或 IDE 用于与 GDB进行通信。此 协议的文档位于 https://microsoft.github.io/debug-adapter-protocol/. 请参阅 调试器适配器协议, 了解有关 GDB 该协议扩展的信息。

mi ¶

最新的 GDB/MI 接口（目前 mi3)。主要用于希望使用 GDB 作为调试器 GUI 或 IDE 后端的程序。有关更多信息，请参阅 GDB/MI 接口.

mi3 ¶

GDB/MI 接口于 GDB 9.1 版本引入。

mi2 ¶

GDB/MI 接口于 GDB 6.0 版本引入。

您可以使用适当的命令从当前解释器在任何解释器中执行命令。如果您正在运行控制台解释器，只需使用 interpreter-exec 命令： 

interpreter-exec mi "-data-list-register-names"

GDB/MI 有类似的命令，不过它仅在支持 GDB 版本 2（或更高版本）的 GDB/MI 版本中可用。

请注意， interpreter-exec 仅在指定命令的执行期间更改解释器。它不会永久更改解释器。

虽然您在启动时只能选择一个解释器，但可以在指定的输入/输出设备（通常是 tty）上运行一个独立的解释器。

例如，考虑一个希望提供 GDB 控制台视图的调试器 GUI 或 IDE。它可以通过在其 GUI 中嵌入一个终端仿真器小部件来实现这一点，以传统 命令行模式启动 GDB ，将标准输入/标准输出/标准错误重定向到该终端，然后创建一个在指定输入/输出设备上运行的 MI 解释器。 GDB 在启动时创建的控制台解释器处理用户在终端小部件中输入的命令，而 GUI 使用单独的 MI 解释器与 GDB 进行控制和同步状态。

要启动一个新的二级 用户界面 运行 MI，请使用 new-ui 命令： 

new-ui 解释器 tty

解释器 参数指定要运行的解释器。此参数接受与 interpreter-exec 命令相同的值。例如，‘console’，‘mi’，‘mi2’等。 tty 参数指定解释器用于输入/输出的双向文件的名称，通常是 Unix 系统上伪终端从设备的名称。例如： 

(gdb) new-ui mi /dev/pts/9

在 /dev/pts/9.
