5.5.1 全停止模式 ¶

在全停止模式下，无论何时您的程序因任何原因停止， GDB 所有 执行线程都会停止，而不仅仅是当前线程。这允许您检查程序的整体状态，包括在线程之间切换，而不必担心情况可能会在您操作时发生变化。 相反，无论何时您重新启动程序，

所有 线程都会开始执行。 即使在单步执行时也是如此， 使用像 这样的命令 step 或 next。

特别是， GDB 不能同步单步执行所有线程。由于线程调度由您的调试目标操作系统决定（不受 GDB控制），当当前线程完成单步执行时，其他线程可能会执行多条语句。此外，一般来说，当程序停止时，其他线程会在语句中间停止，而不是在清晰的语句边界处停止。

您甚至可能会发现在继续执行甚至单步执行后，程序在另一个线程中停止。只要其他某个线程在第一个线程完成您所请求的操作之前遇到断点、信号或异常，就会发生这种情况。

每当 GDB 因断点或信号停止您的程序时，它会自动选择发生该断点或信号的线程。 GDB 会通过一条消息提醒您上下文切换，例如‘[切换到线程 n]’来标识该线程。

在某些操作系统上，您可以通过锁定操作系统调度器来修改 GDB的默认行为，以只允许单个线程运行。

set scheduler-locking mode ¶

设置调度器锁定模式。它适用于正常执行、记录模式和重放模式。 模式 可以是以下之一：

off

不进行锁定，任何线程可以在任何时间运行。

on

当被调试程序恢复运行时，只有当前线程可以运行。由恢复运行的线程创建的新线程会在其入口点停止，在它们执行任何指令之前。

step

在单步执行时表现得像 on ， off 在其他情况下。当您单步执行时，除当前线程外的其他线程永远没有机会运行，而当您使用像‘继续’、‘直到’或‘完成’这样的命令时，它们可以完全自由地运行。

此模式针对单步执行进行了优化；它可以防止其他线程在您单步执行时抢占当前线程，从而使调试焦点不会意外改变。然而，除非另一个线程在其时间片内遇到断点， GDB 不会将当前线程从您正在调试的线程切换走。

replay

在重放模式下表现得像 on ， off 在记录模式或正常执行期间。这是默认模式。

show scheduler-locking

显示当前的调度器锁定模式。

默认情况下，当您发出诸如 continue, next 或 step, GDB 之类的执行命令时，只允许当前被调试程序的线程运行。例如，如果 GDB 附加到两个被调试程序，每个被调试程序有两个线程，那么 continue 命令只会恢复当前被调试程序的两个线程。例如，当您调试一个进行了 fork 操作的程序，并且您希望让父进程停止（这样，例如，它不会运行到退出），而您调试子进程时，这很有用。在其他情况下，您可能对检查 GDB 所附加的任何进程的当前状态不感兴趣，并且您可能希望让它们全部继续运行，直到遇到某个断点。在后一种情况下，您可以指示 GDB 允许所有被调试程序的所有线程运行，使用 set schedule-multiple 命令。

set schedule-multiple ¶

设置在发出执行命令时允许多个进程的线程恢复运行的模式。当 on, 所有进程的所有线程都被允许运行。当 off, 只有当前进程的线程会恢复运行。默认值是 off. 当设置为 scheduler-locking 模式时，该模式优先，或者当您单步执行且设置为 on. step显示当前用于恢复多个进程线程执行的模式。

show schedule-multiple
