5.1.3 设置捕获点 ¶

您可以使用 捕获点 使调试器在某些类型的程序事件发生时停止，例如 C++ 异常或共享库的加载。使用 catch 命令来设置捕获点。

catch event ¶

当 事件 发生时停止。 事件 可以是以下任何一种：

throw [regexp] ¶
rethrow [regexp]
catch [regexp]

C++ 异常的抛出、重新抛出或捕获。++ 异常。

如果 正则表达式 被给出，那么只有类型与正则表达式匹配的异常才会被捕获。

便利变量 $_exception 在某些系统上，在与异常相关的捕获点处可用。它保存着正在抛出的异常。

目前在++ 中对 C++ 异常处理存在一些限制： GDB:

  • 对这些命令的支持依赖于系统。目前，仅支持使用 ‘gnu - v3’ C++ ABI（请参阅++ 配置当前 ABI)的系统。
  • 正则表达式功能和 $_exception 便利变量依赖于 libstdc++中存在一些 SDT 探针。如果这些探针不存在，那么这些功能将无法使用。这些探针在 GCC 4.8 版本中首次可用，但您的 GCC 中是否有这些探针还取决于它的构建方式。
  • 该 $_exception 便利变量仅在设置了与异常相关的捕获点的指令处有效。
  • 当与异常相关的捕获点被触发时， GDB 会在系统库中的某个位置停止，该位置实现了 C++ 的运行时异常支持，通常是++, 通常 libstdc++. 您可以使用 up （请参阅 选择帧)回到您的代码。
  • 如果您交互式地调用一个函数， GDB 通常在函数执行完毕后将控制权返回给您。然而，如果调用引发了异常，该调用可能会绕过将控制权返回给您的机制，导致您的程序要么中止，要么继续运行，直到遇到断点、捕获到 GDB 正在监听的信号，或者退出。即使您为该异常设置了捕获点，情况也是如此；在交互式调用中，异常捕获点会被禁用。请参阅 调用程序函数, 了解如何使用 set unwind-on-terminating-exception. 进行控制的信息。
  • 您不能交互式地引发异常。
  • 您不能交互式地安装异常处理程序。
exception [name] ¶

Ada 异常被引发。如果在命令末尾指定了异常名称（例如 catch exception Program_Error），调试器将仅在引发此特定异常时停止。否则，调试器将在引发任何 Ada 异常时停止执行。

当为用户定义的异常插入异常捕获点，且该异常的名称与语言定义的异常之一相同时，必须使用完全限定名称作为异常名称。否则， GDB 将假设它应该在预定义的异常而不是用户定义的异常处停止。例如，假设在包 Constraint_Error 中定义了一个名为 Pck的异常，那么用于捕获此类异常的命令是 catch exception Pck.Constraint_Error.

便利变量 $_ada_exception 保存着正在抛出的异常的地址。这在为这样的捕获点设置条件时可能会有用。

exception unhandled ¶

程序引发但未处理的异常。便利变量 $_ada_exception 的设置与 catch exception相同。

handlers [name] ¶

Ada 异常被处理。如果在命令末尾指定了异常名称（例如 catch handlers Program_Error），调试器将仅在处理此特定异常时停止。否则，调试器将在处理任何 Ada 异常时停止执行。

当为用户定义的异常插入处理程序捕获点，且该异常的名称与语言定义的异常之一相同时，必须使用完全限定名称作为异常名称。否则， GDB 将假设它应该在预定义的异常而不是用户定义的异常处停止。例如，假设在包 Constraint_Error 中定义了一个名为 Pck的异常，那么用于捕获此类异常处理的命令是 catch handlers Pck.Constraint_Error.

便利变量 $_ada_exception 的设置与 catch exception相同。

assert ¶

Ada 断言失败。请注意，此捕获点不会设置便利变量 $_ada_exception 。 不 设置。

exec ¶

调用 exec.

syscall ¶
syscall [name | number | group:groupname | g:groupname] …

系统调用的调用或返回，也称为 系统调用. 系统调用是应用程序向操作系统（OS）或其中一个操作系统服务请求服务的一种机制。 GDB 可以捕获被调试程序发出的部分或全部系统调用，并显示每个系统调用的相关信息。如果未指定参数，将捕获所有系统调用的调用和返回。

名称 可以是底层操作系统有效的任何系统调用名称。哪些系统调用有效取决于操作系统。在 GNU 和 Unix 系统上，您可以在以下位置找到有效系统调用名称的完整列表 /usr/include/asm/unistd.h.

通常， GDB 预先知道每个操作系统的哪些系统调用是有效的，因此您可以使用 GDB 命令行补全功能（请参阅 命令补全)来列出可用选项。

您也可以用数字指定系统调用。系统调用的编号是传递给操作系统的系统调用调度程序以标识所请求服务的值。当您通过名称指定系统调用时， GDB 使用其系统调用数据库将名称转换为相应的数字代码，但如果 GDB的数据库没有您系统上完整的系统调用列表（例如，因为 GDB 落后于操作系统升级），直接使用编号可能会有用。

您可以使用以下语法一次指定一组相关的系统调用进行捕获 group: 语法（g: 是其更简短的等效形式）。例如，在某些平台上 GDB 允许您通过传递参数来捕获所有与网络相关的系统调用 group:network 到 catch syscall. 请注意，并非所有系统调用组在每个系统中都可用。您可以使用命令补全功能（请参阅 命令补全)来列出您的环境中可用的系统调用组。

下面的示例说明了如果您不向此命令提供参数，它是如何工作的： 

(gdb) catch syscall
捕获点 1（系统调用）
(gdb) r
开始执行程序：/tmp/catch - syscall

捕获点 1（调用系统调用 'close'）， \
	   0xffffe424 在 __kernel_vsyscall () 中
(gdb) c
继续执行。

捕获点 1（从系统调用 'close' 返回）， \
	0xffffe424 在 __kernel_vsyscall () 中
(gdb)

以下是按名称捕获系统调用的示例： 

(gdb) catch syscall chroot
捕获点 1（系统调用 'chroot' [61]）
(gdb) r
开始执行程序：/tmp/catch - syscall

捕获点 1（调用系统调用 'chroot'）， \
		   0xffffe424 在 __kernel_vsyscall () 中
(gdb) c
继续执行。

捕获点 1（从系统调用 'chroot' 返回）， \
	0xffffe424 在 __kernel_vsyscall () 中
(gdb)

按数字指定系统调用的示例。在以下情况下，系统调用编号在 XML 文件中有对应的条目，因此 GDB 找到其名称并打印出来： 

(gdb) catch syscall 252
捕获点 1（系统调用 'exit_group'）
(gdb) r
开始执行程序：/tmp/catch - syscall

捕获点 1（调用系统调用 'exit_group'）， \
		   0xffffe424 在 __kernel_vsyscall () 中
(gdb) c
继续执行。

程序正常退出。
(gdb)

以下是捕获系统调用组的示例： 

(gdb) catch syscall group:process
捕获点 1（系统调用 'exit' [1] 'fork' [2] 'waitpid' [7]
'execve' [11] 'wait4' [114] 'clone' [120] 'vfork' [190]
'exit_group' [252] 'waitid' [284] 'unshare' [310]）
(gdb) r
开始执行程序：/tmp/catch - syscall

捕获点 1（调用系统调用 fork），0x00007ffff7df4e27 在 open64 () 中
   来自 /lib64/ld - linux - x86 - 64.so.2

(gdb) c
继续执行。

但是，可能会出现这样的情况：对于该系统调用编号，XML 文件中没有对应的名称。在这种情况下， GDB 会打印一条警告消息，说明无法找到系统调用名称，但捕获点仍会被设置。请参阅以下示例： 

(gdb) catch syscall 764
警告：数字 '764' 不代表已知的系统调用。
捕获点 2（系统调用 764）
(gdb)

如果您使用 GDB 使用 ‘--without - expat’ 选项进行配置，它将无法显示系统调用名称。此外，如果您的架构没有描述其系统调用的 XML 文件，您将无法看到系统调用名称。重要的是要注意，这两个功能用于访问系统调用名称数据库。在任何一种情况下，您都会看到如下警告： 

(gdb) catch syscall
警告：无法打开 "syscalls/i386 - linux.xml"
警告：无法加载系统调用 XML 文件 'syscalls/i386 - linux.xml'。
GDB 将无法显示系统调用名称。
捕获点 1（系统调用）
(gdb)

当然，文件名会根据您的架构和系统而变化。

仍然使用上面的示例，您也可以尝试按编号捕获系统调用。在这种情况下，您会看到类似如下的内容： 

(gdb) catch syscall 252
捕获点 1（系统调用 252）

同样，在这种情况下 GDB 将无法显示系统调用的名称。

fork ¶

调用 fork.

vfork ¶

调用 vfork.

load [regexp] ¶
unload [regexp]

共享库的加载或卸载。如果 正则表达式 被给出，那么只有当正则表达式与受影响的库之一匹配时，捕获点才会使程序停止。

signal [signal… | ‘all’] ¶

信号的传递。

如果没有参数，此捕获点将捕获任何未被 GDB内部使用的信号，具体来说，除了 ‘SIGTRAP’ 和 ‘SIGINT’ 之外的所有信号。

使用参数 ‘all’，将捕获所有信号，包括被 GDB使用的信号。此参数不能与其他信号名称一起使用。

否则，参数是传递给 handle 的信号名称列表（请参阅 信号)。只有此列表中指定的信号才会被捕获。

使用 catch signal 比 handle 更有用的一个原因是，您可以为捕获点附加命令和条件。

当信号被捕获点捕获时，信号的 stop 和 print 设置（由 handle指定）将被忽略。然而，信号是否仍会传递给被调试程序取决于 pass 设置；这可以在捕获点的命令中更改。

tcatch event ¶

设置一个仅在一次停止时启用的捕获点。该捕获点在首次捕获到事件后会自动删除。

使用 info break 命令列出当前的捕获点。
