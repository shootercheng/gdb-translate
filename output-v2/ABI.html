22.7 配置当前ABI ¶

GDB 可以自动确定 ABI （应用程序二进制接口）。不过，有时你需要推翻它的结论。使用这些命令来管理 GDB对当前ABI的看法。

一种 GDB 配置可以通过远程调试或本地仿真为多个操作系统目标调试二进制文件。 GDB 将自动检测 操作系统ABI （操作系统应用程序二进制接口），但你可以使用 set osabi 命令来推翻它的结论。一个有用的例子是调试使用替代C库（例如 UCLIBC 用于 GNU/Linux）的二进制文件，该库没有你的平台标准C库所提供的相同标识。

当 GDB 调试AArch64架构时，它提供了一个“Newlib”操作系统ABI。这对于处理 setjmp 和 longjmp 在调试使用 NEWLIB C库的二进制文件时很有用。 可以通过 set osabi Newlib选择“Newlib”操作系统ABI。

show osabi

显示当前正在使用的操作系统ABI。

set osabi

不带参数时，显示已注册的可用操作系统ABI列表。

set osabi abi

将当前操作系统ABI设置为 abi。

一般来说，类型为 float 的参数传递给函数的方式取决于函数是否有原型。对于有原型（即ANSI/ISO风格）的函数， float 参数将按照架构的约定原封不动地传递。 float对于没有原型（即K&R风格）的函数， float 参数首先会被提升为类型 double 然后再传递。

不幸的是，某些形式的调试信息不能可靠地表明函数是否有原型。如果 GDB 调用一个没有标记为有原型的函数，它会参考 设置coerce - float - to - double。

set coerce-float-to-double ¶
set coerce-float-to-double on

类型为 float 的参数在传递给没有原型的函数时将被提升为 double 。这是默认设置。

set coerce-float-to-double off

类型为 float 的参数将直接传递给没有原型的函数。

show coerce-float-to-double ¶

显示将 float 提升为 double的当前设置。

GDB 需要知道你的程序的C++ 对象所使用的ABI。正确的C++ ABI取决于用于构建你的应用程序的C++ 编译器。 GDB 仅完全支持使用单一C++ ABI的程序；如果你的程序包含使用多个C++ ABI的代码，或者 GDB 无法正确识别你的程序的ABI，你可以告诉 GDB 使用哪个ABI。 目前支持的ABI包括“gnu - v2”，适用于 g++ 3.0之前的版本，“gnu - v3”，适用于 g++ 3.0及以后的版本，以及适用于HP ANSI C++ 编译器的“hpaCC”。其他C++ 编译器也可能使用“gnu - v2”或“gnu - v3”ABI。默认设置是“自动”。

show cp-abi

显示当前正在使用的C++ ABI。

set cp-abi

不带参数时，显示支持的C++ ABI列表。

set cp-abi abi
set cp-abi auto

将当前C++ ABI设置为 abi，或返回自动检测。
