4.2 启动你的程序 ¶

run ¶
r

使用 run 命令在 GDB. 你必须首先使用参数指定程序名称给 GDB (见 进入和退出 GDB file 或 exec-file 命令（见 指定文件的命令).

如果你在支持进程的执行环境中运行程序， run 会创建一个被调试进程，并使该进程运行你的程序。在一些不支持进程的环境中， run 会跳转到你的程序的开始处。其他目标，如‘remote’，总是处于运行状态。如果你收到如下错误消息： 

The "remote" target does not support "run".
Try "help target" or "continue".

那么使用 continue 来运行你的程序。你可能需要先 load (见 load).

程序的执行会受到它从上级接收的某些信息的影响。 GDB 提供了指定这些信息的方法，你必须在 启动程序之前 进行指定。（你可以在启动程序后更改这些信息，但这些更改只会在下一次启动程序时影响程序。）这些信息可以分为四类：

程序的 参数。

将传递给程序的参数指定为 run 命令的参数。如果目标系统上有可用的 shell，会使用 shell 来传递参数，这样你可以使用正常的约定（如通配符展开或变量替换）来描述参数。在 Unix 系统上，你可以使用 SHELL 环境变量来控制使用哪个 shell。如果你未定义 SHELL, GDB 会使用默认的 shell（/bin/sh)。你可以使用 set startup-with-shell 命令禁用使用任何 shell（详情见下文）。

程序的 环境。

你的程序通常从 GDB继承其环境，但你可以使用 GDB 命令 set environment 和 unset environment 来更改影响程序的部分环境。见 你的程序的环境.

程序的 工作目录。

你可以使用命令 set cwd.来设置程序的工作目录。如果你未使用此命令设置任何工作目录，你的程序将在进行原生调试时继承 GDB的工作目录，或者在进行远程调试时继承远程服务器的工作目录。见 你的程序的工作目录.

程序的 标准输入和输出。

你的程序通常使用与 GDB 相同的设备进行标准输入和标准输出。你可以在 run 命令行中重定向输入和输出，或者使用 tty 命令为你的程序设置不同的设备。见 你的程序的输入和输出.

警告： 虽然输入和输出重定向可以正常工作，但你不能使用管道将正在调试的程序的输出传递给另一个程序；如果你尝试这样做， GDB 很可能最终会调试错误的程序。

当你发出 run 命令时，你的程序会立即开始执行。见 停止和继续, 讨论如何安排程序停止。一旦你的程序停止，你可以使用 print 或 call 命令调用程序中的函数。见 检查数据.

如果自上次 GDB 读取符号文件以来，符号文件的修改时间发生了变化， GDB 会丢弃其符号表并重新读取。在这样做时， GDB 会尝试保留你当前的断点。

start ¶

主程序的名称可能因语言而异。在 C 或 C++ 中，主程序的名称总是++, 但其他语言（如 Ada）不要求主程序有特定的名称。调试器提供了一种方便的方法来启动程序的执行，并根据使用的语言在主程序的开始处停止。 main

‘start’命令相当于在主程序的开始处设置一个临时断点，然后调用‘run’命令。

有些程序包含一个 程序准备阶段 在调用主程序之前执行一些启动代码的阶段。这取决于编写程序所使用的语言。例如，在 C 语言中++，静态和全局对象的构造函数会在 main 被调用之前执行。因此，调试器有可能在到达主程序之前停止。然而，临时断点仍将用于暂停执行。

将传递给程序的参数指定为‘start’命令的参数。这些参数将原封不动地传递给底层的‘run’命令。请注意，如果在后续调用‘start’或‘run’时未提供参数，将重用相同的参数。

有时需要在程序准备阶段进行调试。在这些情况下，使用 start 命令会使程序执行停止得太晚，因为程序可能已经完成了准备阶段。在这种情况下，可以在运行程序之前在准备代码中插入断点，或者使用 starti 命令。

starti ¶

‘starti’命令相当于在程序执行的第一条指令处设置一个临时断点，然后调用‘run’命令。对于包含准备阶段的程序， starti 命令将在准备阶段开始时停止执行。

set exec-wrapper wrapper ¶
show exec-wrapper
unset exec-wrapper

当设置了‘exec-wrapper’时，将使用指定的包装器来启动要调试的程序。 GDB 以如下形式的 shell 命令启动你的程序 exec wrapper program。会对 program 及其参数添加引号，但不会对 wrapper添加引号，因此如果你的 shell 需要，你应该自行添加引号。包装器会一直运行，直到执行你的程序，然后 GDB 接管控制权。

你可以使用任何最终调用 execve 及其参数的程序作为包装器。一些标准的 Unix 实用程序可以做到这一点，例如 env 和 nohup。任何以 exec "$@" 结尾的 Unix shell 脚本也可以。

例如，你可以使用 env 将一个环境变量传递给被调试的程序，而无需在你的 shell 环境中设置该变量： 

(gdb) set exec-wrapper env 'LD_PRELOAD=libtest.so'
(gdb) run

此命令在大多数目标系统（不包括 DJGPP、Cygwin 和 MS Windows）上进行本地调试时可用。

set startup-with-shell
set startup-with-shell on
set startup-with-shell off
show startup-with-shell

在 Unix 系统上，默认情况下，如果目标系统上有可用的 shell， GDB) 会使用它来启动你的程序。 run 命令的参数会传递给 shell，shell 会进行变量替换、展开通配符并执行 I/O 重定向。在某些情况下，禁用使用 shell 可能会很有用，例如，在调试 shell 本身或诊断启动失败时，如： 

(gdb) run
Starting program: ./a.out
During startup program terminated with signal SIGSEGV, Segmentation fault.

这表明 shell 或使用‘exec-wrapper’指定的包装器崩溃了，而不是你的程序。大多数情况下，这是由 shell 的非交互式模式初始化文件中的异常情况引起的，例如 .cshrc 对于 C-shell，$.zshenv 对于 Z shell，或者 BASH 环境变量指定的文件。 BASH_ENV

set auto-connect-native-target ¶
set auto-connect-native-target on
set auto-connect-native-target off
show auto-connect-native-target

默认情况下，如果当前的被调试程序尚未连接到任何目标（例如，使用 target remote）， run 命令会在你的本地机器上以原生进程的方式启动你的程序。如果你确定不想在本地机器上调试程序，你可以使用 GDB, 在你的本地机器上。如果你确定不想在本地机器上调试程序，你可以告诉 GDB 不要自动连接到原生目标，使用 set auto-connect-native-target off 命令。

如果 on（默认情况），并且当前的被调试程序尚未连接到目标， run 命令会自动连接到可用的原生目标。

如果 off, 并且如果当前的被调试程序尚未连接到目标， run 命令会出错失败： 

(gdb) run
Don't know how to run.  Try "help target".

如果当前的被调试程序已经连接到目标， GDB 总是会在 run 命令中使用该连接。

在任何情况下，你都可以使用 target native 命令显式地连接到原生目标。例如， 

(gdb) set auto-connect-native-target off
(gdb) run
Don't know how to run.  Try "help target".
(gdb) target native
(gdb) run
Starting program: ./a.out
[Inferior 1 (process 10421) exited normally]

如果你显式地连接到了 native 目标， GDB 即使所有被调试程序退出，连接仍然保持，准备好进行下一次 run 命令。使用 disconnect 命令断开连接。

其他同样遵循 auto-connect-native-target 设置的命令示例： attach, info proc, info os.

set disable-randomization ¶
set disable-randomization on

此选项（在 GDB中默认启用）将关闭启动程序的虚拟地址空间的原生随机化。此选项对于多个调试会话很有用，可以使执行更具可重复性，并使内存地址在不同调试会话中可重用。

此功能仅在某些目标系统上实现，包括 GNU/Linux。在 GNU/Linux 上，你可以使用 

(gdb) set exec-wrapper setarch `uname -m` -R
set disable-randomization off

保持启动的可执行文件的行为不变。有些 bug 只有在程序加载到特定地址时才会出现。如果你在 GDB下运行程序时 bug 消失了，可能是因为 GDB 默认情况下在某些平台（如 GNU/Linux）上禁用了地址随机化，而这些平台在独立程序中会进行地址随机化。使用 set disable-randomization off 来尝试重现这些难以捉摸的 bug。

在支持虚拟地址空间随机化的目标系统上，它可以保护程序免受某些类型的安全攻击。在这些情况下，攻击者需要知道具体可执行代码的确切位置。随机化其位置使得利用预期地址处的代码注入跳转变得不可能。

预链接共享库可以提供启动性能优势，但它会使特权进程通过无特权访问目标系统就能预测这些库中的地址。读取共享库二进制文件可以获得足够的信息来组装滥用它的恶意代码。不过，即使是预链接的共享库也可以在启动时加载到新的随机地址，只需在启动期间进行常规的重定位过程。未预链接的共享库总是加载到随机选择的地址。

位置无关可执行文件（PIE）包含类似于共享库的位置无关代码，因此此类可执行文件在启动时会加载到随机选择的地址。PIE 可执行文件总是将即使已经预链接的共享库加载到随机地址。你可以使用 gcc -fPIE -pie.

堆（malloc 存储）、栈和自定义 mmap 区域总是随机放置（只要启用了随机化）。

show disable-randomization

显示启动程序的虚拟地址空间的原生随机化的显式禁用的当前设置。
