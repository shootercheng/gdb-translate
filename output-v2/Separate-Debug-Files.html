18.3 分离文件中的调试信息 ¶

GDB 允许你将程序的调试信息放在一个与可执行文件本身分离的文件中，这样可以 GDB 自动查找并加载调试信息。由于调试信息可能非常大——有时甚至比可执行代码本身还要大——一些系统将其可执行文件的调试信息分发到单独的文件中，用户只有在需要调试问题时才安装这些文件。

GDB 支持两种指定分离调试信息 文件的方式：

  • 可执行文件包含一个 调试链接 它指定了分离调试信息文件的名称。分离调试文件的名称通常是 .debug, 其中 可执行文件 是相应可执行文件的名称，不包含前导目录（例如， ls.debug 对应 /usr/bin/ls）。此外，调试链接指定了一个 32 位的 循环冗余校验 (CRC) 调试文件的校验和，用于验证可执行文件和调试文件是否来自同一构建。 GDB 可执行文件包含一个
  • 构建 ID , 一个唯一的位串，该位串也存在于相应的调试信息文件中。（仅在某些操作系统上支持，当二进制文件使用 ELF 或 PE 文件格式以及Binutils 时。）有关此功能的更多详细信息，请参阅 GNU --build-id 命令行选项的描述，该选项在 命令行选项 中，位于 。调试信息文件的名称不是由构建 ID 明确指定的，但可以从构建 ID 计算得出，见下文。 GNU 链接器。

根据调试信息文件的指定方式， GDB 使用两种不同的方法来查找调试文件：

  • 对于“调试链接”方法， GDB 首先在可执行文件的目录中查找指定名称的文件，然后在该目录的名为 .debug的子目录中查找，最后在每个全局调试目录下，在一个名称与可执行文件绝对文件名的前导目录相同的子目录中查找。（在 MS - Windows/MS - DOS 上，可执行文件前导目录的驱动器字母会转换为一个单字母的子目录，即 d:/usr/bin/ 会转换为 /d/usr/bin/, 因为 Windows 文件系统不允许文件名中包含冒号。）
  • 对于“构建 ID”方法， GDB 在每个全局调试目录的 .build - id 子目录中查找名为 nn/nnnnnnnn.debug, 其中 nn 是构建 ID 位串的前 2 个十六进制字符， nnnnnnnn 是位串的其余部分。（实际的构建 ID 字符串有 32 个或更多十六进制字符，而不是 10 个。） GDB 可以自动查询 debuginfod 服务器，使用构建 ID 下载本地找不到的分离调试文件。有关更多信息，请参阅 使用 Debuginfod 下载调试资源。

例如，假设你要求 GDB 调试 /usr/bin/ls, 它有一个调试链接，指定文件为 ls.debug, 并且构建 ID 的十六进制值为 abcdef1234. 如果全局调试目录列表包括 /usr/lib/debug, 那么 GDB 将按以下顺序查找以下调试信息文件：

  • /usr/lib/debug/.build - id/ab/cdef1234.debug
  • /usr/bin/ls.debug
  • /usr/bin/.debug/ls.debug
  • /usr/lib/debug/usr/bin/ls.debug.

如果仍然没有找到调试文件，并且 debuginfod （见 使用 Debuginfod 下载调试资源）已启用， GDB 将尝试从 debuginfod 服务器下载该文件。

全局调试信息目录默认为由 GDB 配置选项 --with - separate - debug - dir 设置的值，并通过 GDB 配置选项 --additional - debug - dirs提供的以冒号分隔的目录列表进行扩充。在 GDB 运行期间，你还可以设置全局调试信息目录，并查看当前正在使用的列表。 GDB

调试链接是可执行文件中名为 .gnu_debuglink的特殊节。该节必须包含：

  • 一个文件名，去掉任何前导目录组件，后面跟着 一个零字节，
  • 根据需要填充零到三个字节，以达到该节内的下一个四字节 边界，以及
  • 一个四字节的 CRC 校验和，存储的字节序与可执行文件本身使用的字节序相同。校验和是通过下面给出的函数对调试 信息文件的全部内容进行计算的，将零作为 crc 参数传入。

任何可执行文件格式都可以携带调试链接，只要它可以 包含一个名为 .gnu_debuglink 且内容如上所述的节。

构建 ID 是可执行文件（以及其他 ELF 二进制文件中可能会考虑的）中的一个特殊节。这个节通常名为 GDB ，但这个名称不是必需的。 它包含构建文件的唯一标识——在同一构建树的多次构建中，该 ID 保持不变。默认算法 SHA1 为构建 ID 字符串生成 160 位（40 个十六进制字符）的内容。在带有符号的原始构建二进制文件、其剥离版本以及分离的调试信息文件中，都存在具有相同值的相同节。 .note.gnu.build-id调试信息文件本身应该是一个普通的 可执行文件，包含完整的链接器符号、节和 调试信息。调试信息文件的节应该与原始文件具有相同的名称、地址和大小，但它们不需要包含任何数据——就像普通可执行文件中的

节一样。 .bss 节。

GNU 二进制工具集（Binutils）包包含“objcopy”工具，它可以使用以下命令生成分离的可执行文件/调试信息文件对： 

objcopy --only-keep-debug foo foo.debug
strip -g foo

这些命令从可执行文件中移除调试 信息 foo 并将其放置在文件 foo.debug中。你可以使用第一种、第二种或两种方法来链接这两个文件：

  • 调试链接方法需要以下附加命令，以便在 foo中也留下一个调试链接： 
    objcopy --add-gnu-debuglink=foo.debug foo

    乌尔里希·德雷珀（Ulrich Drepper）的 elfutils 包从 0.53 版本开始，包含一个版本的 strip 命令，使得命令 strip foo -f foo.debug 具有与上述两个 objcopy 命令以及 ln -s 命令相同的功能。

  • 构建 ID 使用 ld --build-id 或 GCC 对应的 gcc -Wl,--build-id嵌入到主可执行文件中。自 2.18 版本起， GNU 二进制工具集（Binutils）包中包含了对构建 ID 的支持以及对调试文件分离的兼容性修复。

中使用的 CRC 是 .gnu_debuglink 由 IEEE 802.3 定义的 CRC - 32，使用多项式： 

 x32 + x26 + x23 + x22 + x16 + x12 + x11
 + x10 + x8 + x7 + x5 + x4 + x2 + x + 1

该函数逐字节计算，首先取每个字节的最低有效位。使用初始模式 0xffffffff 以确保前导零影响 CRC，并且最终结果取反以确保尾随零也影响 CRC。

注意： 这与处理 远程串行协议 qCRC 数据包（见 qCRC 数据包）时使用的 CRC 多项式相同。 然而，在远程串行协议的情况下，CRC 是从最高有效位开始计算的，并且结果不取反，因此尾随零对 CRC 值没有影响。
