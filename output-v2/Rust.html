15.4.8 Rust ¶

GDB 支持 Rust 编程语言。类型和值的打印以及表达式 解析相当完善。不过，有一些 特殊情况和漏洞需要注意。

  • 行规范（请参阅 位置规范）绝不是相对于 当前 crate 的。相反，它们的行为就好像存在一个全局的 crate 命名空间，有点类似于 extern crate 的行为方式。

    也就是说，如果 GDB 在 crate ‘A’、模块 ‘B’ 中的一个函数的断点处停止，那么 break B::f 将尝试 在名为 ‘f’ 的函数中设置一个断点，该函数位于名为 ‘B’ 的 crate 中。

    由于这种方法，行规范也不能使用 ‘self::’ 或 ‘super::’ 来引用项。

  • 因为 GDB 在表达式中实现了 Rust 的名称查找语义，它有时会在名称前加上当前 crate。 例如，如果 GDB 在 crate ‘K’ 的断点处停止，那么 print ::x::y 将尝试查找符号 ‘K::x::y’。

    然而，由于在调试时能够引用其他 crate 很有用， GDB 提供了 extern 扩展来绕过这个问题。要使用该扩展，只需在 路径表达式前加上 extern 来引用原本不可用的 “全局” 作用域。

    在上面的例子中，如果你想引用符号 ‘y’，它位于 crate ‘x’ 中，你可以使用 print extern x::y.

  • Rust 表达式求值器不支持 “类似语句” 的 表达式，例如 if 或 match, 或者 lambda 表达式。
  • 元组表达式未实现。
  • Rust 表达式求值器目前未实现 Drop trait。求值器可能创建的对象永远不会被销毁。
  • GDB 不实现泛型的类型推断。为了调用泛型函数或以其他方式引用泛型项，你 必须手动指定类型参数。
  • GDB 目前使用 C++ 反编译器来处理 Rust。在大多数 情况下，这不会造成任何问题。然而，在表达式 上下文中，补全泛型函数名会给出语法上无效的结果。这是因为 Rust 要求在函数名和其泛型参数之间使用 ‘::’ 运算符。例如， GDB 可能会提供类似 crate::f<u32>, 但解析器需要 crate::f::<u32>.
  • 在撰写本文时，Rust 编译器（版本 1.8）在其生成的调试信息中存在一些漏洞。这些漏洞导致某些 功能无法由 GDB:
    • 无法通过 trait 进行方法调用。
    • 运算符重载未实现。
    • 在单态化函数中调试时，不能使用泛型 类型名。
    • 类型 Self 不可用。
    • use 语句不可用，因此 crate 中的某些名称可能不可用。
