15.4.10.11 Ada 模式的已知特性 ¶

除了前面列出的遗漏（见“ Ada 的遗漏）之外，我们还知道 Ada 模式存在一些问题和限制， GDB其中一些问题将在调试器和 GNU Ada 编译器的未来计划版本中修复。

  • 编译器选择不在存储中实例化为对象的静态常量对调试器不可见。
  • 函数参数列表中的命名参数关联被忽略（参数列表被视为按位置排列）。
  • 许多有用的库包目前对调试器不可见。
  • 定点算术、转换、输入和输出使用浮点算术进行，可能会给出仅近似主机机器上结果的结果。
  • GNAT 编译器从不为 Ada 语言定义的任何标准符号生成前缀 Standard 。 GDB 知道这一点：当你使用它时，它会从名称中去除前缀，并且永远不会在本地符号中查找你如此限定的名称，也不会与其他包或子程序中的符号匹配。如果你在程序中的任何位置定义了除参数和局部变量之外的实体，其简单名称与 Standard中的名称匹配， Standard GNAT 在此处缺乏限定可能会导致混淆。当这种情况发生时，你通常可以通过使用包

显式限定有问题的名称来解决混淆。

set ada trust-PAD-over-XVS on

较旧版本的编译器有时会生成错误的调试信息，导致调试器错误地打印受影响实体的值。在某些情况下，调试器能够自动解决问题。在其他情况下，调试器能够解决问题，但必须专门启用解决方法。 PAD 配置 GDB 在计算 Ada 实体的值时严格遵循 GNAT 编码，特别是当 PAD___XVS 和 ada/exp_dbug.ads 类型涉及到时（有关 GNAT 编译器使用的编码的完整描述，请参阅 GCC 源中的

set ada trust-PAD-over-XVS off

这与 GNAT 编译器使用的编码有关。如果 GDB 有时会为某些实体打印错误的值，将 ada trust-PAD-over-XVS 更改为 off 会激活一个可能解决问题的解决方法。将 ada trust-PAD-over-XVS 设置为 off总是安全的，但这会带来轻微的性能损失，因此建议除非必要，否则将此设置保留为 on 。

在内部，调试器还依赖于编译器遵循一些称为‘GNAT 编码’的约定，所有这些都记录在 GCC 源中的 gcc/ada/exp_dbug.ads 中。此编码描述了应为某些类型生成调试信息的方式。特别是，此约定使用 描述性类型，

这些是纯粹为帮助调试器而生成的人工类型。这些编码是在当时使用的调试信息格式不够强大以描述 Ada 中可用的一些更复杂类型的时候定义的。由于 DWARF 允许我们表达几乎所有 Ada 特性，长期目标是慢慢地用它们的纯 DWARF 等效物替换这些描述性类型。为了促进这种过渡，有一个新的维护选项可用于强制调试器忽略那些描述性类型。它允许用户快速评估 GDB 在没有它们的情况下的工作情况。

maintenance ada set ignore-descriptive-types [on|off] ¶

控制调试器是否应忽略描述性类型。默认值是不忽略描述性类型（off）。

maintenance ada show ignore-descriptive-types ¶

显示是否被 GDB忽略描述性类型。
