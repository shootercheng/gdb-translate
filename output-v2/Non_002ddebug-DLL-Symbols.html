21.1.4.1 对无调试符号的DLL的支持 ¶

在Windows系统中，你的程序所依赖的一些DLL文件常常不包含符号调试信息（例如 kernel32.dll）。当 GDB 无法识别DLL中的任何调试符号时，它会依赖DLL导出表中包含的最少符号信息。本节将介绍如何使用这些符号， GDB 内部将其称为 “最小符号”。

请注意，在被调试的程序开始执行之前，不会加载任何DLL文件。解决此问题的最简单方法是启动程序 —— 可以通过设置断点，或者让程序运行一次直至结束。

21.1.4.2 DLL名称前缀 ¶

为了遵循Microsoft调试工具使用的命名约定，DLL导出符号会以基于DLL名称的前缀形式提供，例如 KERNEL32!CreateFileA。普通名称也会被录入符号表，因此 CreateFileA 通常就足够了。在某些情况下，程序中可能会出现名称冲突（特别是当可执行文件本身包含完整的调试符号时），这就需要在引用DLL内容时使用完全限定名称。使用单引号将名称括起来，以避免感叹号（“!”）被解释为语言运算符。

请注意，DLL的内部名称可能全为大写，即使DLL的文件名是小写的，反之亦然。由于 GDB 中的符号 区分大小写 这可能会造成一些混淆。如果有疑问，可以尝试使用 info functions 和 info variables 命令，甚至 maint print msymbols （请参阅 检查符号表）。以下是一个示例： 

(gdb) info function CreateFileA
所有与正则表达式 “CreateFileA” 匹配的函数：

非调试符号：
0x77e885f4  CreateFileA
0x77e885f4  KERNEL32!CreateFileA

(gdb) info function !
所有与正则表达式 “!” 匹配的函数：

非调试符号：
0x6100114c  cygwin1!__assert
0x61004034  cygwin1!_dll_crt0@0
0x61004240  cygwin1!dll_crt0(per_process *)
[等等...]

21.1.4.3 使用最小符号 ¶

从DLL导出表中提取的符号不包含太多类型信息。 GDB 所能做的只是根据包含该符号的链接器段来猜测一个符号是指函数还是变量。还要注意，除非程序正在运行，否则DLL中包含的内存的实际内容是不可用的。这意味着如果程序没有运行，你就无法检查DLL中变量的内容或反汇编DLL中的函数。

变量通常被视为指针并自动解引用。因此，在命令中通常需要在变量名前加上取地址运算符（“&”）并提供明确的类型信息。以下是这类问题的一个示例： 

(gdb) print 'cygwin1!__argv'
'cygwin1!__argv' 类型未知；将其转换为声明的类型

(gdb) x 'cygwin1!__argv'
'cygwin1!__argv' 类型未知；将其转换为声明的类型

以下是两种可能的解决方案： 

(gdb) print ((char **)'cygwin1!__argv')[0]
$2 = 0x22fd98 "/cygdrive/c/mydirectory/myprogram"

(gdb) x/2x &'cygwin1!__argv'
0x610c0aa8 <cygwin1!__argv>:    0x10021608      0x00000000
(gdb) x/x 0x10021608
0x10021608:     0x0022fd98
(gdb) x/s 0x0022fd98
0x22fd98:        "/cygdrive/c/mydirectory/myprogram"

即使在程序开始执行之前，也可以在DLL中设置断点。然而，在这种情况下， GDB 无法检查函数的初始指令以跳过函数的帧设置代码。你可以通过使用 “*&” 在原始内存地址处设置断点来解决这个问题： 

(gdb) break *&'python22!PyOS_Readline'
断点 1 设置在 0x1e04eff0

这些扩展的作者并不完全确信在像 kernel32.dll 这样的共享DLL中设置断点是完全安全的。
