23.3.2.43 Python 中缺少目标文件 ¶

当 GDB 打开一个核心文件时，例如使用 核心文件 命令（请参阅 核心文件命令), GDB

如果 GDB 无法找到特定文件时，Python 扩展就有机会介入。Python 扩展可以使用一些特定于平台或项目的步骤来定位缺失的文件，并将其位置告知 GDB 。或者，Python 扩展可以为用户提供一些特定于平台或项目的建议，说明如何获取缺失的文件。

一个缺少目标文件的 Python 扩展由一个处理程序对象组成，该对象具有 name 和 enabled 属性，并实现了 __call__ 方法。当 GDB 遇到找不到文件的情况，但已知缺失文件的构建 ID（请参阅 构建 ID) 时，将调用 __call__ 方法来尝试查找该文件。有关如何编写处理程序的详细信息可以在下面找到。

gdb.missing_objfile 模块 ¶

GDB 附带一个 gdb.missing_objfile 模块，其中包含以下类和全局函数：

类： gdb.missing_objfile.MissingObjfileHandler ¶

MissingObjfileHandler 是一个基类，用户创建的处理程序可以从该基类派生，不过处理程序不一定要从这个类派生，只要任何用户创建的处理程序具有 name 和 enabled 属性，并实现了 __call__ 方法即可。

函数： MissingObjfileHandler.__init__ (name) ¶

name 是一个字符串，用于在某些 GDB 命令中引用此缺少目标文件处理程序。有效的名称由字符 ‘[-_a-zA-Z0-9]’ 组成，创建一个名称无效的处理程序会引发一个 ValueError 异常。

函数： MissingObjfileHandler.__call__ (pspace, build_id, filename) ¶

子类必须重写 __call__ 方法。 pspace 参数将是一个 gdb.Progspace （请参阅 Python 中的程序空间），这是 GDB 正在其中查找缺失文件的程序空间。

build_id 参数是一个字符串，包含缺失文件的构建 ID，其格式与 Objfile.build_id 返回的格式相同（请参阅 Objfile.build_id)。

filename 参数包含 GDB 正在查找的文件的名称。提供此信息是为了让处理程序能够为用户生成有用的消息。处理程序不一定要将缺失的文件放在此位置。此位置可能已经有一个文件，但它可能与所需的构建 ID 不匹配，在这种情况下 GDB 会忽略它。在某些有限的情况下 GDB 可能无法确定 filename 它正在搜索的文件的名称，在这种情况下 GDB 将使用字符串 ‘带有构建 ID build_id’ 作为替代。

__call__ 方法的返回值指示 GDB 接下来应该做什么。可能的返回值如下：

  • None

    这表明此处理程序无法定位缺失的文件，并且 GDB 应该调用任何其他已注册的处理程序。

  • True

    这表明此处理程序已将缺失的文件安装到 GDB 通常期望找到它的位置。 GDB 将查找的唯一位置是在 .build-id 子目录中，该子目录位于 调试文件目录 （请参阅 调试文件目录) 内。

    GDB 将重复正常的查找过程，此时应该能找到之前缺失的文件。

    如果 GDB 在第二次尝试后仍然找不到文件，那么 Python 缺少目标文件处理程序不会再被调用第二次，这可以防止行为不良的处理程序导致 GDB 陷入循环。 GDB 将在缺少文件的情况下继续，不过这会降低调试体验。

  • False

    这表明此处理程序已完成其打算做的所有事情，但未找到缺少的文件。 GDB 不会调用任何其他已注册的处理程序来查找缺少的文件。 GDB 将在缺少文件的情况下继续，不过这会降低调试体验。

  • 一个字符串

    返回的字符串应包含一个文件名。 GDB 不会再调用任何其他已注册的处理程序，而是将返回的文件名用作缺少的文件。

调用 __call__ 此基类的方法将引发一个 NotImplementedError 异常。

变量： MissingObjfileHandler.name ¶

一个只读属性，是一个字符串，即传递给 __init__ 方法的此处理程序的名称。

变量： MissingObjfileHandler.enabled ¶

一个可修改的属性，包含一个布尔值；当 True时，处理程序已启用，并将由 GDB使用。 当 False时，处理程序已禁用，将不会被使用。

函数： gdb.missing_objfile.register_handler (locus, handler, replace=False) ¶

向 GDB注册一个新的缺少目标文件处理程序。

处理程序 是 MissingObjfileHandler的子类的实例，或者至少是一个具有与 MissingObjfileHandler相同属性和方法的对象的实例。

locus 指定要将 处理程序添加到哪个处理程序列表的开头。它可以是一个 gdb.Progspace （请参阅 Python 中的程序空间) 或 None，在这种情况下，处理程序将全局注册。新注册的 处理程序 将在同一位置的任何其他处理程序之前被调用。同一位置的两个处理程序不能有相同的名称，尝试添加一个已存在名称的处理程序会引发异常，除非 replace 为 True，在这种情况下，旧的处理程序将被删除，新的处理程序将被添加到所选处理程序列表的开头。

GDB 首先调用当前程序空间的处理程序，然后调用全局注册的处理程序。一旦有一个处理程序返回的值不是 None，就不会再调用其他处理程序。

管理缺少目标文件处理程序 ¶

GDB 定义了以下命令来管理已注册的缺少目标文件处理程序：

info missing-objfile-handlers [ locus [ name-regexp ] ] ¶

列出所有已注册的缺少目标文件处理程序。参数 locus 和 name-regexp 都是可选的，可用于过滤要列出的处理程序。

locus 参数应该是 全局, 程序空间, 或者一个目标文件的名称。只会列出为指定位置注册的处理程序。

name-regexp 是一个正则表达式，用于与处理程序名称进行匹配。

disable missing-objfile-handler [ locus [ name-regexp ] ] ¶

locus 和 name-regexp 的解释与 info missing-objfile-handlers 上面相同，但不是列出匹配的处理程序，而是禁用所有匹配的处理程序。每个匹配处理程序的 enabled 字段将被设置为 False.

enable missing-objfile-handler [ locus [ name-regexp ] ] ¶

locus 和 name-regexp 的解释与 info missing-objfile-handlers 上面相同，但不是列出匹配的处理程序，而是启用所有匹配的处理程序。每个匹配处理程序的 enabled 字段将被设置为 True.
