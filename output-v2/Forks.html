4.11 调试分叉 ¶

在大多数系统上， GDB 对于使用 fork 函数创建额外进程的程序调试没有特殊支持。 当程序分叉时， GDB 将继续调试父进程，而子进程将不受阻碍地运行。 如果你在子进程随后执行的任何代码中设置了断点，子进程将收到一个 SIGTRAP 信号（除非它捕获该信号） 这将导致它终止。

但是，如果你想调试子进程，有一个不太麻烦的解决方法。 在子进程在分叉后执行的代码中调用 sleep 。 仅在设置了某个环境变量或存在某个文件时进行休眠可能会很有用， 这样在你不想对该子进程运行 GDB 时就不需要有延迟。 当子进程处于休眠状态时，使用 ps 程序获取其进程 ID。 然后告诉 GDB （如果你也在调试父进程，则是一个新的 GDB 实例）附加到子进程（请参阅 调试已经运行的进程)。 从那时起，你就可以像调试任何其他附加的进程一样调试子进程。

在某些系统上， GDB 为使用 fork 或 vfork 函数创建额外进程的程序调试提供支持。 在 GNU/Linux 平台上，内核版本 2.5.46 及更高版本支持此功能。

分叉调试命令在原生模式下以及连接到 gdbserver 时，无论是在 target remote 模式还是 target extended-remote 模式下都受支持。

默认情况下，当程序分叉时， GDB 将继续调试父进程，而子进程将不受阻碍地运行。

如果你想跟踪子进程而不是父进程，请使用命令 set follow-fork-mode.

set follow-fork-mode mode ¶

设置调试器对程序调用 fork 或 vfork的响应。 对 fork 或 vfork 的调用会创建一个新进程。 模式 参数可以是：

parent

分叉后调试原始进程。 子进程不受阻碍地运行。 这是默认设置。

child

分叉后调试新进程。 父进程不受阻碍地运行。

show follow-fork-mode ¶

显示调试器当前对 fork 或 vfork 调用的响应。

在 Linux 上，如果你想同时调试父进程和子进程，请使用命令 set detach-on-fork.

set detach-on-fork mode ¶

告诉 gdb 在分叉后是否分离其中一个进程，或者保留对它们两者的调试器控制。

on

子进程（或父进程，取决于 follow-fork-mode) 的值）将被分离并允许独立运行。 这是默认设置。

off

两个进程都将由 GDB控制。 一个进程（子进程或父进程，取决于 follow-fork-mode) 的值）将像往常一样被调试，而另一个将被暂停。

show detach-on-fork ¶

显示分叉时分离模式是开启还是关闭。

如果你选择关闭‘分叉时分离’模式，那么 GDB 将保留对所有分叉进程（包括嵌套分叉）的控制。 你可以使用 GDB 命令列出受 info inferiors 控制的分叉进程，并使用 inferior 命令从一个分叉切换到另一个分叉（请参阅 调试多个子进程连接和程序)。

要停止调试其中一个分叉进程，你可以使用 detach inferiors 命令将其分离（允许它独立运行），或者使用 kill inferiors 命令将其终止。 请参阅 调试 多个子进程连接和程序.

如果你要求调试子进程，并且一个 vfork 之后跟着一个 exec, GDB 会执行新目标直到新目标中的第一个断点。 如果你在原始程序的 main 上设置了断点，该断点也将设置在子进程的 main上。

在某些系统上，当子进程由 vfork产生时，在 exec 调用完成之前，你无法调试子进程或父进程。

如果你在 run 发出 GDB 调用执行后向 exec 命令，新目标将重新启动。 要重启父进程，请使用 file 命令并将父可执行文件的名称作为其参数。 默认情况下，在 exec 调用执行后， GDB 会丢弃之前可执行映像的符号。 你可以使用 set follow-exec-mode 命令更改此行为。

set follow-exec-mode mode ¶

设置调试器对程序调用的响应 exec. 一次 exec 调用会替换进程的程序映像。

follow-exec-mode 可以是：

new

GDB 创建一个新的子进程并将该进程重新绑定到这个 新的子进程。 在 exec 调用之前进程正在运行的程序可以在之后通过重启 原始子进程来重新启动。

例如： 

(gdb) info inferiors
(gdb) info inferior
  Id   描述   可执行文件
* 1    <空>        prog1
(gdb) run
进程 12020 正在执行新程序：prog2
程序正常退出。
(gdb) info inferiors
  Id   描述   可执行文件
  1    <空>        prog1
* 2    <空>        prog2
same

GDB 使进程保持绑定到同一个子进程。 新的 可执行映像会替换子进程中之前加载的可执行文件。 在 exec 调用之后重启子进程，例如使用 run 命令，会重启进程在 exec 调用之后正在运行的可执行文件。 这是默认模式。

例如： 

(gdb) info inferiors
  Id   描述   可执行文件
* 1    <空>        prog1
(gdb) run
进程 12020 正在执行新程序：prog2
程序正常退出。
(gdb) info inferiors
  Id   描述   可执行文件
* 1    <空>        prog2

follow-exec-mode 在原生模式和 target extended-remote 模式下受支持。

你可以使用 catch 命令来让 GDB 在每次进行 fork, vfork, 或者 exec 调用时停止。 请参阅 设置捕获点.
