E.15 库列表格式 ¶

在某些平台上，动态加载器（例如 ld.so) 与您的应用程序在同一进程中运行以管理库。在这种情况下， GDB 可以使用加载器的符号表和正常的内存操作来维护共享库列表。在其他平台上，操作系统管理已加载的库。 GDB 无法通过内存操作检索当前已加载的库列表，因此它使用 ‘qXfer:libraries:read’ 数据包（请参阅 qXfer库列表读取) 代替。远程存根查询目标的操作系统并报告哪些库已加载。

‘qXfer:libraries:read’ 数据包返回一个XML文档，该文档列出了已加载的库及其偏移量。每个库都有一个关联的名称和一个或多个段或节基地址，这些地址报告库在内存中的加载位置。

对于完全链接的二进制库的常见情况，库应该有一个段列表。如果目标支持可重定位目标文件的动态链接，其库XML元素应该包含一个已分配节的列表。段或节基地址是起始地址，而不是重定位偏移量；它们不依赖于库的链接时基地址。

GDB 必须与Expat库链接以支持XML库列表。请参阅 Expat.

一个简单的内存映射，其中一个已加载的库通过单个偏移量进行重定位，如下所示： 

<library-list>
  <library name="/lib/libc.so.6">
    <segment address="0x10000000"/>
  </library>
</library-list>

另一个简单的内存映射，其中一个已加载的库有三个已分配的节（.text、.data、.bss），如下所示： 

<library-list>
  <library name="sharedlib.o">
    <section address="0x10000000"/>
    <section address="0x20000000"/>
    <section address="0x30000000"/>
  </library>
</library-list>

库列表的格式由以下DTD描述： 

<!-- library-list: 带有版本信息的根元素 -->
<!ELEMENT library-list  (library)*>
<!ATTLIST library-list  version CDATA   #FIXED  "1.0">
<!ELEMENT library       (segment*, section*)>
<!ATTLIST library       name    CDATA   #REQUIRED>
<!ELEMENT segment       EMPTY>
<!ATTLIST segment       address CDATA   #REQUIRED>
<!ELEMENT section       EMPTY>
<!ATTLIST section       address CDATA   #REQUIRED>

此外，段和节描述符不能在单个库元素中混合使用，并且每个库必须至少提供一个段或节。
