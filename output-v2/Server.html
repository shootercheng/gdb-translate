20.3 使用 gdbserver 程序 ¶

gdbserver 是类Unix系统的控制程序，它允许你将你的程序与远程 GDB 通过 target remote 或 target extended-remote—但不链接通常的调试桩。

gdbserver 不是调试桩的完全替代品，因为它本质上需要与 GDB 本身相同的操作系统设施。 实际上，一个能够运行 gdbserver 连接到远程 GDB 也可以运行 GDB 本地！ gdbserver 尽管如此，有时还是有用的，因为它比 GDB 本身小得多。 它也比 GDB, 所以你可以通过使用 gdbserver. 最后，如果你为实时系统开发代码，你可能会发现实时操作所涉及的权衡使得在另一个系统上尽可能多地进行开发工作更加方便，例如通过交叉编译。 你可以使用 gdbserver 做出类似的调试选择。

GDB 和 gdbserver GDB 远程串行协议中不支持。

警告： gdbserver 没有任何内置的安全性。 不要运行 gdbserver 连接到任何公共网络；一个 GDB 连接到 gdbserver 提供对目标系统的访问，其权限与运行 gdbserver.

20.3.1 运行 gdbserver ¶

在目标系统上运行 gdbserver 。 你需要一份你要调试的程序的副本，包括它所需的任何库。 gdbserver 不需要你的程序的符号表，所以如果需要节省空间，你可以剥离程序。 GDB 在主机系统上处理所有的符号处理。

要使用服务器，你必须告诉它如何与 GDB通信；你的程序的名称；以及你的程序的参数。 通常的语法是： 

目标> gdbserver comm 程序 [ 参数 ... ]

comm 可以是设备名称（使用串行线），或者是TCP主机名和端口号，或者 - 或 stdio 使用 gdbserver. 例如，要调试带有参数foo.txt的Emacs，并通过串行端口 GDB 与 /dev/com1通信： 

目标> gdbserver /dev/com1 emacs foo.txt

gdbserver 被动等待主机 GDB 与它通信。

要使用TCP连接而不是串行线： 

目标> gdbserver host:2345 emacs foo.txt

与上一个例子的唯一区别是第一个参数，指定你通过 GDB TCP与主机通信。 ‘host:2345’参数意味着 gdbserver 期望从机器host到本地TCP端口2345的TCP连接。 （目前，‘host’部分被忽略。） 只要端口号不与目标系统上任何已使用的TCP端口冲突，你可以选择任何你想要的端口号（例如， 23 是为 telnet保留的。16 你必须在主机 GDB target remote 命令中使用相同的端口号。

stdio 连接在使用ssh启动 gdbserver 时很有用： 

(gdb) target remote | ssh -T hostname gdbserver - hello

‘-T’选项提供给ssh是因为我们不需要远程pty，并且我们不希望进行转义字符处理。 当提供命令时，ssh默认会这样做，提供该标志是为了明确说明。 如果你愿意，你可以省略它。

使用stdio连接的gdbserver启动的程序有 /dev/null 用于 stdin, 并且 stdout,stderr 通过连接到gdbserver的管道发送回gdb进行显示。 stdout 和 stderr 使用同一个管道。

20.3.1.1 附加到正在运行的程序 ¶

在某些目标系统上， gdbserver 也可以附加到正在运行的程序。 这是通过 --attach 参数实现的。 语法如下： 

目标> gdbserver --attach comm pid

pid 是当前正在运行的进程的进程ID。 没有必要指定 gdbserver 正在运行的进程的二进制文件。

在 target extended-remote 模式下，你也可以使用 GDB attach命令 （参见 远程连接类型中的附加操作）。

如果你的目标系统支持，你可以按名称而不是进程ID来调试进程，使用 pidof 实用工具： 

目标> gdbserver --attach comm `pidof 程序`

如果有多个 程序 正在运行，或者 程序 有多个线程，大多数版本的 pidof 支持 -s 选项，只返回第一个进程ID。

20.3.1.2 的TCP端口分配生命周期 gdbserver ¶

本节仅适用于 gdbserver 在TCP端口上监听的情况。

gdbserver 通常在所有被调试的进程在 target remote 模式下终止后终止。 另一方面，对于 target extended-remote, gdbserver 即使没有进程剩余也会继续运行。 GDB 通常在退出时终止派生的被调试进程，这通常也会终止 gdbserver 在 target remote 模式下。 因此，当连接意外断开，并且 GDB 无法要求 gdbserver 终止其被调试的进程时， gdbserver 即使在 target remote 模式下也会继续运行。

当 gdbserver 继续运行时， GDB 之后可以再次连接到它。 这种重新连接对于像 断开连接跟踪这样的功能很有用。 为了完整性，一次最多只能有一个 GDB 连接。

默认情况下， gdbserver 保持监听的TCP端口开放，以便后续连接成为可能。 但是，如果你使用 gdbserver 并使用 --once 选项，它将在连接到第一个 GDB 会话后停止监听任何进一步的连接尝试。 这意味着在第一次连接之后，将无法再连接到 gdbserver 。 这也意味着 gdbserver 在与远程 GDB 的第一次连接关闭后将终止，即使是意外关闭的连接，即使在 target extended-remote 模式下也是如此。 --once 选项允许将相同的端口号用于连接同一主机上运行的多个 gdbserver 实例，因为每个实例在第一次连接后会关闭其端口。

20.3.1.3 的其他命令行参数 gdbserver ¶

你可以使用 --multi 选项来启动 gdbserver 而不指定要调试的程序或要附加的进程。 然后你可以在 target extended-remote 模式下附加并运行或附加到一个程序。 更多信息，请参见 远程连接类型中的--multi选项.

--debug[=option1,option2,…] 选项告诉 gdbserver 显示有关调试过程的额外诊断信息。 选项（option1, option2, 等等）控制将显示哪些方面的额外信息，可能的值如下： gdbserver

all

这将启用所有可用的诊断输出。

threads

这将启用与线程处理相关的诊断输出。 目前，其他一般诊断输出也包含在这个类别中，但在未来的版本中可能会改变 gdbserver。

event-loop

这将启用与事件循环特定的诊断输出。

remote

这将启用与调试器之间的远程协议数据包传输相关的诊断输出。

如果没有向 --debug 传递选项，那么这将被视为等同于 --debug=threads. 这在未来的版本中可能会改变 gdbserver. 传递给 --debug 的选项从左到右处理，单个选项可以前缀 - （减号）字符以禁用该方面的诊断输出，因此可以使用： 

  目标> gdbserver --debug=all,-event-loop

以便启用除事件循环之外的所有诊断输出。

--debug-file=文件名 选项告诉 gdbserver 将任何调试输出写入给定的 文件名. 这些选项旨在用于 gdbserver 开发和向开发人员提交错误报告。

--debug-format=option1[,option2,...] 选项告诉 gdbserver 在每个输出中包含额外信息。可能的选项如下：

none

关闭调试输出中的所有额外信息。

all

打开调试输出中的所有额外信息。

timestamps

在调试输出的每一行中包含时间戳。

选项按顺序处理。 因此，例如，如果 none 最后出现，那么调试输出中不会添加任何额外信息。

--wrapper 选项指定一个包装器来启动要调试的程序。 该选项后面应跟包装器的名称，然后是要传递给包装器的任何命令行参数，然后 -- 表示包装器参数的结束。

gdbserver 运行指定的包装器程序，其组合命令行包括包装器参数，然后是要调试的程序的名称，然后是程序的任何参数。 包装器运行直到它执行你的程序，然后 GDB 获得控制权。

你可以使用任何最终调用 execve 及其参数的程序作为包装器。 几个标准的Unix实用工具可以做到这一点，例如 env 和 nohup. 任何以 exec "$@" 结尾的Unix shell脚本也可以工作。

例如，你可以使用 env 将环境变量传递给被调试的程序，而无需在 gdbserver的环境中设置该变量： 

$ gdbserver --wrapper env LD_PRELOAD=libtest.so -- :2222 ./testprog

--selftest 选项在 gdbserver中运行自测试： 

$ gdbserver --selftest
运行了2个单元测试，0个失败

这些测试在发布版本中被禁用。

20.3.2 连接到 gdbserver ¶

连接到远程目标的基本过程是：

  • 在主机系统上运行 GDB 。
  • 确保你有必要的符号文件 （参见 主机和目标文件）。 在连接之前，使用 file 命令为你的应用程序加载符号。 使用 set sysroot 来定位目标库（除非你的 GDB 是使用正确的sysroot编译的 --with-sysroot）。
  • 连接到你的目标（参见 连接到远程目标）。 对于TCP连接，你必须在使用 gdbserver 命令之前启动 target 。 否则，你可能会得到一个错误，其文本取决于主机系统，但通常看起来像连接被拒绝。 当使用 load 命令在 GDB 中使用 target remote 模式时，不要使用该命令，因为程序已经在目标系统上。

20.3.3 的监控命令 gdbserver ¶

在使用 GDB 会话期间，你可以使用 gdbserver, 你可以使用 monitor 命令向 gdbserver发送特殊请求。 以下是可用的命令。

monitor help

列出可用的监控命令。

monitor set debug off

禁用gdbserver的所有内部日志记录。

monitor set debug on

启用gdbserver内部的一些常规日志记录。 目前这等同于 monitor set debug threads on, 但在gdbserver的未来版本中可能会改变。

monitor set debug threads off
monitor set debug threads on

禁用或启用与gdbserver中线程处理相关的特定日志消息。 目前这个类别还包括与线程处理没有特别关系的额外输出，这在gdbserver的未来版本中可能会改变。

monitor set debug remote off
monitor set debug remote on

禁用或启用与远程协议相关的特定日志消息（参见 GDB 远程串行协议）。

monitor set debug event-loop off
monitor set debug event-loop on

禁用或启用与gdbserver的事件循环相关的特定日志消息。

monitor set debug-file filename
monitor set debug-file

将任何调试输出发送到给定的文件，或发送到stderr。

monitor set debug-format option1[,option2,...]

指定要添加到调试消息中的额外文本。可能的选项如下：

none

关闭调试输出中的所有额外信息。

all

打开调试输出中的所有额外信息。

timestamps

在调试输出的每一行中包含时间戳。

选项按顺序处理。 因此，例如，如果 none 最后出现，那么调试输出中不会添加任何额外信息。

monitor set libthread-db-search-path [PATH] ¶

当发出此命令时， 路径 是一个以冒号分隔的目录列表，用于搜索 libthread_db （参见 set libthread-db-search-path）。 如果你省略 路径, ‘libthread-db-search-path’将被重置为其默认值。

‘$pdir’对于‘libthread-db-search-path’的特殊条目在 gdbserver.

monitor exit

告诉gdbserver立即退出。 此命令后面应跟 disconnect 以关闭调试会话。 gdbserver 将从任何附加的进程中分离，并杀死它创建的任何进程。 使用 monitor exit 在多进程模式调试会话结束时终止 gdbserver 。

20.3.4 中的跟踪点支持 gdbserver ¶

在某些目标系统上， gdbserver 支持跟踪点和快速跟踪点。

为了使快速跟踪点正常工作，必须在被调试进程中加载一个名为 进程内代理 （IPA）的特殊库。 这个库是作为 gdbserver的一个组成部分构建和分发的。

有几种方法可以在你的程序中加载进程内代理：

Specifying it as dependency at link time

你可以将你的程序与进程内代理库动态链接。 在大多数系统上，这可以通过在链接命令中添加 -linproctrace 来实现。

Using the system's preloading mechanisms

你可以通过使用系统对预加载共享库的支持，在启动时强制加载进程内代理。 许多Unix系统支持预加载用户定义库的概念。 在大多数情况下，你可以通过在环境中指定 LD_PRELOAD=libinproctrace.so 来实现。 另请参阅 gdbserver的 --wrapper 命令行选项的描述。

Using GDB to force loading the agent at run time

在某些系统上，你可以通过在被调试进程中调用一个动态加载器函数来强制被调试进程加载一个共享库，该函数负责动态查找和加载共享库。 在大多数Unix系统上，该函数是 dlopen. 你可以使用 call 命令来实现。 例如： 

(gdb) call dlopen ("libinproctrace.so", ...)

请注意，在大多数Unix系统上，为了使 dlopen 函数可用，程序需要与 -ldl链接。

在具有用户空间动态加载器的系统上，如大多数Unix系统，当你连接到 gdbserver 使用 target remote, 你会发现程序在动态加载器的入口点停止，并且程序的地址空间中还没有加载任何共享库，包括进程内代理。 在这种情况下，在能够使用任何快速或静态跟踪点功能之前，你需要让加载器运行并加载共享库。 最简单的方法是将程序运行到主程序。 例如，如果调试一个C或C++ 程序，像这样启动 gdbserver ： 

$ gdbserver :9999 myprogram

启动GDB并像这样连接到 gdbserver ，并运行到main： 

$ gdb myprogram
(gdb) target remote myhost:9999
0x00007f215893ba60 in ?? () from /lib64/ld-linux-x86-64.so.2
(gdb) b main
(gdb) continue

进程内跟踪代理库现在应该已经加载到进程中；你可以使用 info sharedlibrary 命令来确认，该命令将列出 libinproctrace.so 已加载到进程中。 你现在可以安装快速跟踪点、列出静态跟踪点标记、探测静态跟踪点标记并开始跟踪。
