附录D 维护命令 ¶

除了为 GDB 用户设计的命令外， GDB 还包括一些为 GDB 开发者设计的命令，这些命令在本手册的其他地方未作记录。此处提供这些命令以供参考。（有关开启调试消息的命令，请参阅 关于内部事件的可选消息.)

maint agent [-at linespec,] expression ¶
maint agent-eval [-at linespec,] expression

将给定的 表达式 转换为远程代理字节码。 此命令对于调试代理表达式机制很有用（请参阅 GDB代理表达式机制)。‘代理’版本生成一个对数据收集有用的表达式，例如通过跟踪点，而‘’生成一个直接计算结果的表达式。例如，对于 globa + globb 将包含字节码，以记录 globa 和 globb, 同时丢弃加法的结果，而求值表达式将执行加法并返回和。 如果 -at 被指定，则为 行规范 解析（请参阅 行规范位置). 如果未指定，则为当前帧的PC地址生成远程代理字节码。

set displaced-stepping ¶
show displaced-stepping

控制是否 GDB 将执行 位移步进 如果目标支持的话。位移步进是一种单步跳过断点而不将其从目标进程中移除的方法，通过执行原本位于断点位置的指令的一个离线副本。它也被称为离线单步执行。

set displaced-stepping on

如果目标架构支持， GDB 将使用位移步进跳过断点。

set displaced-stepping off

GDB 即使目标架构支持，也不会使用位移步进跳过断点。

set displaced-stepping auto ¶

这是默认模式。 GDB 仅在非停止模式激活时（请参阅 非停止模式）且目标架构支持位移步进时，才会使用位移步进。

maint info inline-frames ¶
maint info inline-frames address

打印从当前地址开始的内联帧的信息，或者如果指定了 地址 ，则打印从该地址开始的内联帧的信息。

为了允许用户正确地单步进入内联函数， GDB 需要识别哪些内联函数从特定地址开始，并且 GDB 还需要跟踪这些函数中哪个最后被显示给用户作为当前帧。

想象一个情况，函数 main 调用 foo, 然后它调用 bar, 类似这样： 

int
main ()
{
  /* 这里有一些有趣的代码...  */

  foo ();

  /* 这里还有更多有趣的代码... */
}

void
foo ()
{
  bar ();
}

void
bar ()
{
  /* 这里有一些有趣的代码...  */
}

由于 foo 和 bar 都在内联到 main 中，那么 main 内可能有一个地址也是 foo 的起始地址，也是 bar的起始地址。当用户在这个地址处停止时，最初会被告知目标进程处于 main, 如果用户执行一个 单步 那么 GDB 实际上不会单步执行目标进程，而是会告诉用户目标进程进入了 foo. 在下一次 单步 之后，会告诉用户目标进程进入了 bar. 可以使用 maint info inline-frames 命令来查看这个内部 GDB 状态，如下所示： 

(gdb) step
24	  foo ();
(gdb) maintenance info inline-frames
线程 1 的缓存内联状态信息。
程序计数器 = 0x401137
跳过的帧 = 2
  bar
  foo
> main

这里用户在 main 调用 foo处停止。内联帧信息显示，在这个地址处 GDB 找到了内联函数 bar 和 foo的起始地址，但当前 GDB 跳过了2个帧，并认为 main 是当前帧，这用‘>’表示。

如果用户执行一个 单步 进入 foo ，那么情况会更新： 

(gdb) step
foo () 在 inline.c:14
14	  bar ();
(gdb) maintenance info inline-frames
线程 1 的缓存内联状态信息。
程序计数器 = 0x401137
跳过的帧 = 1
  bar
> foo
  main

注意，程序计数器的值 0x401137 没有改变，但现在 GDB 认为 foo 是当前帧，并使用‘>’标记。

最后，用户再执行一次 单步 进入 bar: 

(gdb) step
bar () 在 inline.c:6
6	  ++global_counter;
(gdb) maintenance info inline-frames
线程 1 的缓存内联状态信息。
程序计数器 = 0x401137
跳过的帧 = 0
> bar
  foo
  main

maint flush register-cache ¶
flushregs

刷新寄存器缓存的内容，从而刷新帧缓存。此命令在调试与寄存器获取或帧展开相关的问题时很有用。命令 flushregs 已被弃用，建议使用 maint flush register-cache。

maint flush source-cache ¶

刷新 GDB的源代码文件内容缓存。在 GDB 读取源代码文件并可选地应用样式（请参阅 输出样式）后，文件内容会被缓存。此命令清除该缓存。下次 GDB 想要显示源代码文件的行时，内容将被重新读取。

此命令在调试与源代码样式相关的问题时很有用。刷新缓存后， GDB 显示的任何源代码都将被重新读取和重新样式化。

maint print statistics ¶

此命令为程序中的每个目标文件打印有关该目标文件的各种数据，后跟该目标文件的字节缓存（bcache) 统计信息。目标文件数据包括最小、部分、完整和stabs符号的数量、目标文件定义的类型数量、尚未展开的psym表的数量、行表和字符串表的数量，以及各种表使用的内存量。字节缓存统计信息包括所有和唯一对象的计数、大小和重复计数、最大、平均和中值条目大小、使用的总内存及其开销和节省，以及哈希表大小和链长度的各种度量。

maint print target-stack ¶

一个 目标 是调试器与特定类型的文件或进程之间的接口。目标可以堆叠在 层, 以便多个目标可能对一个请求做出响应。特别是，内存访问将遍历目标栈，直到找到一个对处理该特定地址感兴趣的目标。

此命令打印推送到 目标栈上的每个层的简短描述，从顶层到底层。

maint print type expr ¶

打印由 表达式指定的类型的类型链。参数可以是类型名称或符号。如果是符号，则描述该符号的类型。此命令生成的类型链是存储在 GDB的数据结构中的数据类型的递归定义，包括其标志和包含的类型。

maint print record-instruction ¶
maint print record-instruction N

打印GDB如何记录给定的指令。如果 n 不是正数，则打印在执行第 n条前一条指令之前目标进程存储的值。如果 n 是正数，则打印执行第 n条后一条指令之后的值。如果未指定 n ，则假定为0。

maint selftest [-verbose] [filter] ¶

运行编译到 GDB中的任何自测。这将打印一条消息，显示运行了多少个测试以及有多少个失败。如果传递了 过滤器 ，则仅运行名称中包含 过滤器 的测试。如果传递了 -verbose ，自测可以更详细。

maint set dwarf always-disassemble ¶
maint show dwarf always-disassemble

控制使用DWARF调试信息时 info address 的行为。

默认是 off, 这意味着 GDB 应该尝试以易于阅读的格式描述变量的位置。当 on, GDB 将改为以类似汇编的格式显示DWARF位置表达式。请注意，某些位置对于 GDB 来说太复杂，无法简单描述；在这种情况下，你将始终看到反汇编形式。

以下是生成的反汇编示例： 

(gdb) info addr argc
符号 "argc" 是一个复杂的DWARF表达式：
     1: DW_OP_fbreg 0

有关这些表达式的更多信息，请参阅 DWARF标准.

maint set dwarf max-cache-age ¶
maint show dwarf max-cache-age

控制DWARF编译单元缓存。

在具有编译单元间引用的目标文件中，例如由GCC选项‘-feliminate-dwarf2-dups’生成的文件，DWARF读取器需要频繁引用先前读取的编译单元。此设置控制如果一个编译单元未被引用，它将在缓存中保留多长时间。较高的限制意味着缓存的编译单元将在内存中存储更长时间，并且将使用更多的总内存。将其设置为零将禁用缓存，这将减慢 GDB 的启动速度，但减少内存消耗。

maint set dwarf synchronous ¶
maint show dwarf synchronous

控制是否异步读取DWARF。

在支持线程的主机上，DWARF读取器相对于 GDB的其余部分大多是异步的。也就是说，大部分读取操作在后台进行，并且 GDB 仅在绝对必要时才会暂停以等待此任务完成。

当此设置启用时， GDB 将改为在继续之前等待DWARF处理完成。

在不支持线程的主机上，或者在运行时禁用了工作线程的情况下，此设置无效，因为DWARF读取始终在主线程上进行，因此始终是同步的。

maint set worker-threads ¶

控制 GDB可以使用的工作线程数量。在支持的主机上， GDB 可以使用多个线程来加速某些CPU密集型操作，例如符号名称修饰。虽然 GDB 使用的线程数量可能会有所不同，但此命令可用于设置该数量的上限。默认值是 unlimited, 这让 GDB 选择一个合理的数量。请注意，这仅控制 GDB 自己启动的工作线程； GDB 使用的库可能会自己启动线程。

maint set profile ¶

控制 GDB的性能分析。

在使用‘maint set profile’命令启用之前，性能分析将被禁用。启用性能分析后，系统将开始收集计时和执行计数数据；禁用性能分析或退出 GDB时，结果将写入日志文件。请记住，如果使用性能分析， GDB 将覆盖性能分析日志文件（通常称为 gmon.out)。如果你在 gmon.out 文件中有重要的性能分析数据记录，请务必将其移动到安全的位置。

使用‘--enable-profiling’进行配置会使 GDB 使用‘-pg’编译器选项进行编译。
